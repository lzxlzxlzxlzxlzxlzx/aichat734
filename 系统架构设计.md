## 1. 文档目的

本文件用于把已经完成的模块级详细设计，统一收束为一套可实际落地的系统架构。

目标不是做空泛的“前后端分层图”，而是明确：

- Izumi Studio 作为一个本地优先单用户系统，整体如何运行
- 哪些能力是三种模式共享的底座
- 哪些能力属于模式专属模块
- 一轮对话、一轮创作、一轮工具调用和一轮图片生成分别如何穿过系统
- 哪些设计直接参考了 SillyTavern 的现有主链路，哪些是主动重构

本架构设计应能直接指导后续：

- 代码目录设计
- 服务层拆分
- API 设计
- 核心数据模型设计
- MVP 范围切片

## 2. 架构设计结论

## 2.1 总体结论

Izumi Studio 采用：

- `Vue 3 + TypeScript` 前端
- `FastAPI + Python` 后端
- `SQLite` 作为第一阶段主数据库
- 文件系统作为媒体与配置存储层
- 本地浏览器访问的单用户部署形态

整体采用“前端应用 + 后端应用服务 + 本地存储”的三层结构。

## 2.2 与 SillyTavern 的关系

Izumi Studio 的架构思路明显借鉴 SillyTavern，但不直接沿用其页面和组织方式。

参考点主要有：

- 消息从输入到模型请求的主链路
- 世界书 / 角色卡 / Prompt 注入对对话链路的深度参与
- Prompt 调试与“实际发送消息”查看的必要性
- 多模型适配和 provider 抽象

但 Izumi Studio 做了以下主动重构：

- 从单聊天页产品改为三模式产品
- 从前端重拼逻辑改为以后端管线为中心
- 从“功能叠加”改为“共享底座 + 模式模块”结构
- 从“页面驱动”改为“服务驱动”

## 2.3 当前架构原则

当前架构必须同时满足：

- 单用户本地优先
- 模块边界清晰
- Prompt 可观察性优先
- 文件与图片为一等能力
- 后续可迁移到 PostgreSQL 和云端

## 3.1 来自 SRS 的关键约束

来自 [产品需求文档_V2.md](/d:/Users/1010709/Desktop/test/ST-1/产品需求文档_V2.md) 的关键约束包括：

- 三层记忆系统
- 联网搜索工具调用

- 状态栏 / 悬浮球

## 3.2 来自当前模块设计的关键约束

来自已完成详细设计的关键约束包括：

- 游玩模式已改为“线性会话 + 对话复制 + 任意长度回溯”
- 泉此方预设是全局常驻创作者框架
- Prompt 管线必须以后端可追踪快照为中心
- 记忆系统必须三层并且原始消息永久保留
- 状态系统必须按会话维度维护状态快照
- 图片和文件都应作为消息附件进入会话流

## 3.3 来自 SillyTavern 代码分析的关键启发

来自 [SillyTavern代码分析_V1.md](/d:/Users/1010709/Desktop/test/ST-1/SillyTavern代码分析_V1.md) 的关键启发包括：

- 输入入口和消息组装应有明确统一主链路
- Prompt Inspector 最好同时看到“组装前”和“最终发送前”的关键状态
- 世界书、角色卡、场景、历史消息是天然的上下文组装输入
- provider 最终请求往往会再被改写，因此真正的可观察性不能只停留在前端

这直接推动 Izumi Studio 采用：

- 以后端 Prompt 管线为中心
- 统一保存 PromptTrace 快照
- 统一在后端完成工具调用、Regex 清洗、状态提取、摘要生成等关键流程

## 4. 总体架构

## 4.1 总体拓扑

```text
┌──────────────────────────────────────────────┐
│                Vue 前端应用                   │
│ 模式路由 / 页面状态 / SSE / 文件上传 / UI 组件 │
└──────────────────────┬───────────────────────┘
                       │ HTTP / SSE / WS
┌──────────────────────▼───────────────────────┐
│               FastAPI 后端应用                │
│ API 层 / 应用服务层 / Prompt 管线 / 调度器     │
└───────────────┬───────────────┬──────────────┘
                │               │
       ┌────────▼───────┐ ┌────▼────────────────┐
       │ SQLite 数据库   │ │ 本地文件存储         │
       │ 索引 / 快照 /   │ │ cards/worldbooks/    │
       │ 元数据 / 记录   │ │ sessions/images/files │
       └────────────────┘ └─────────────────────┘
                │
       ┌────────▼──────────────────────────────┐
       │ 外部模型与工具服务                     │
       │ LLM / 搜索 API / 图像生成 API / 视觉模型 │
       └───────────────────────────────────────┘
```

## 4.2 三层划分

系统分为三层：

### A. 表现层

负责：

- 三种模式页面
- 对话界面
- Prompt Inspector 展示
- 状态栏 / 悬浮球
- 文件和图片上传入口

### B. 应用层

负责：

- 模式服务
- Prompt 管线
- 工具调用
- 记忆调度
- 状态快照
- 媒体处理

### C. 存储与外部集成层

负责：

- SQLite
- 本地文件存储
- LLM Provider
- 搜索服务
- 图像服务

## 5. 模块化架构

## 5.1 模式模块

建议把系统拆成三类模块：

### 模式专属模块

- `creation`
- `play`
- `chat`

### 共享业务底座

- `prompt_pipeline`
- `preset_system`
- `memory`
- `state`
- `tooling`
- `media`
- `model_router`
- `trace`

### 通用基础设施

- `storage`
- `repositories`
- `jobs`
- `events`
- `config`

## 5.2 共享底座是架构核心

Izumi Studio 的复杂度不在页面，而在共享底座的协作，因此架构核心不是模式页面，而是：

- Prompt 管线
- 记忆系统
- 状态系统
- 工具调用
- 多模态处理
- PromptTrace

模式模块只负责组织这些底座，不重复实现底层逻辑。

## 6. 前端架构

## 6.1 技术栈

前端建议采用：

- Vue 3
- TypeScript
- Vite
- Pinia
- Axios
- SSE 优先，必要时 WebSocket

## 6.2 前端职责

前端应聚焦：

- 页面路由与用户交互
- 组件级状态
- 流式响应展示
- 附件上传体验
- Prompt Inspector 展示

前端不应承担：

- 核心 Prompt 组装
- 记忆生成
- 工具调用逻辑
- 状态解析

这些应放在后端。

## 6.3 前端页面模块

建议页面层包括：

- `首页 / 模式入口`
- `创作模式页面`
- `游玩模式页面`
- `聊天模式页面`
- `设置页面`

### 游玩模式前端组件

- 角色卡选择页
- 开场白选择弹窗
- 消息流
- Swipe 切换器
- 状态栏 / 悬浮球 / 侧边面板
- Prompt Inspector 抽屉

### 聊天模式前端组件

- 历史会话列表
- 消息流
- 跨模式引用选择器
- Quick Reply 栏
- Prompt Inspector 抽屉

### 创作模式前端组件

- 项目列表
- 世界书编辑器
- 角色卡配置器
- 结构预览区
- 生成对话区

## 6.4 前端状态分层

建议区分：

- `页面状态`
- `会话状态`
- `流式状态`
- `Inspector 展示状态`
- `上传任务状态`

不建议把所有业务逻辑压在前端 store 中。

## 7. 后端架构

## 7.1 技术栈

后端建议采用：

- FastAPI
- Pydantic
- SQLAlchemy 或等价 ORM / Repository 抽象
- BackgroundTasks 或独立任务执行器

## 7.2 后端职责

后端是系统核心，负责：

- API
- 会话编排
- Prompt 管线执行
- 模型调用与 provider 路由
- 工具调用
- 记忆生成与注入
- 状态解析与持久化
- 媒体元数据管理
- PromptTrace 保存

## 7.3 后端应用分层

建议再细分为：

### API 层

负责：

- 路由定义
- 请求验证
- 响应格式
- 鉴于单用户本地，无需复杂权限逻辑

### Application Service 层

负责：

- PlayService
- ChatService
- CreationService
- PromptPipelineService
- MemoryService
- StateService
- MediaService
- ToolService

### Repository 层

负责：

- SQLite 访问
- 文件索引与元数据
- 快照查询

### Integration 层

负责：

- LLM provider clients
- Search client
- Image generation client
- 文档解析器

## 8. 模型与 Provider 架构

## 8.1 设计目标

模型调用必须统一收口，不能让不同模式各自直接调模型。

## 8.2 Model Router

建议设置统一 `ModelRouter`，负责：

- 根据模式和任务选择模型
- 适配不同 provider 的请求格式
- 统一流式 / 非流式响应
- 统一记录 provider 元数据

## 8.3 Provider Adapter

每个 provider 应有单独适配器：

- ClaudeAdapter
- QwenAdapter
- DeepSeekAdapter
- KimiAdapter
- AzureOpenAIAdapter

这些适配器应只负责：

- 请求格式转换
- 返回格式归一化

而不是承担业务逻辑。

## 8.4 特殊任务模型

架构层应支持不同任务使用不同模型：

- 主对话模型
- 摘要模型
- 图片理解模型
- 图片生成模型

这在后续做成本优化时非常重要。

## 9. Prompt 管线架构

## 9.1 后端中心化 Prompt 管线

系统采用“后端中心化 Prompt 管线”。

原因：

- 需要统一可追踪性
- 需要统一工具调用
- 需要统一状态提取
- 需要统一记忆注入
- 需要统一多模态附件处理

## 9.2 Prompt 管线步骤

建议统一为：

1. 输入预处理
2. 预设层收集
3. 动态注入项收集
4. 结构化注入项列表生成
5. `messages` 数组组装
6. 首次模型调用
7. 工具调用链处理
8. 原始输出接收
9. Regex / `state_update` / 清洗
10. PromptTrace 保存

## 9.3 与 SillyTavern 的架构差异

SillyTavern 的启发是“消息组装层必须是核心”，但 Izumi Studio 不直接照搬其前端中心式实现。

Izumi Studio 选择：

- 前端负责交互
- 后端负责实际 Prompt 生命周期

这样更适合：

- Prompt Inspector
- 工具调用
- 多模态
- 背景摘要

## 10. 会话与消息架构

## 10.1 统一 Session 抽象

建议统一定义 `Session` 抽象，模式只是其属性之一：

- `mode = play`
- `mode = chat`
- `mode = creation`

这样有利于：

- 统一消息存储
- 统一 PromptTrace 关联
- 统一记忆和附件挂载

## 10.2 模式差异由策略层决定

虽然共享 Session 抽象，但模式差异由策略控制：

- play session 绑定角色卡与状态系统
- chat session 绑定泉此方常驻人格
- creation session 绑定项目与草稿

## 10.3 当前游玩模式特殊约束

尽管 SRS 仍写有 Branch / Checkpoint，但按当前真实设计，系统架构应以最新模块文档为准：

- 游玩模式采用线性会话
- 支持复制当前对话
- 支持任意长度回溯

因此架构中不再把消息树作为第一阶段核心结构。

---

## 11. 记忆架构

## 11.1 三层记忆归属于共享底座

短期 / 中期 / 长期记忆应为共享底座，不属于任一模式单独实现。

## 11.2 关键架构决策

- 原始消息永久保留
- 摘要与原消息分离存储
- 长期记忆按作用域绑定
- 记忆注入项结构化输出给 Prompt 管线

## 11.3 后台任务

中期摘要建议作为后台任务异步执行。

这意味着系统架构必须预留轻量任务调度层。

---

## 12. 状态系统架构

## 12.1 会话级状态存储

状态系统应按 `session_id` 挂载。

## 12.2 状态快照

状态必须支持：

- 当前值
- 历史增量
- 完整快照

原因是游玩模式的：

- 回溯
- 对话复制
- 摘要状态快照引用

都依赖它。

## 12.3 状态推送

前端状态 UI 更新建议走：

- 请求后返回最新状态
- 或 SSE / WebSocket 推送

第一阶段优先简单可用，可先使用“请求后返回最新状态 + 局部刷新”。

---

## 13. 工具调用架构

## 13.1 工具调用是 Prompt 生命周期的一部分

工具调用不单独做 Agent 层，而是作为 Prompt 生命周期中的一步。

## 13.2 第一阶段范围

第一阶段只需要：

- `web_search`

通过标准化 `tool_call` / `tool_result` 接入即可。

## 13.3 调用记录

工具调用结果和元数据必须进入：

- PromptTrace
- Prompt Inspector

## 14. 多模态架构

## 14.1 媒体作为消息附件

图片和文件不应作为“外部临时对象”，而应作为消息附件进入会话流。

## 14.2 媒体分层

建议拆为：

- `media binary storage`
- `media metadata storage`
- `message attachment relation`

## 14.3 第一阶段支持

第一阶段架构要直接支持：

- 图片上传
- 图片理解输入
- TXT / DOCX / DOC 文件输入
- 手动图片生成
- 参考图挂载

## 14.4 后续扩展

后续可以自然扩展：

- PDF
- 图片编辑
- 自动插图
- 视频能力

## 15. 存储架构

## 15.1 主结论

第一阶段采用：

- SQLite 存结构化数据
- 文件系统存媒体与大对象

## 15.2 为什么不把所有内容都塞进数据库

原因：

- 图片、文件和导出物本来就更适合文件系统
- 本地单用户场景优先追求简单稳定
- 后续云迁移时也更容易拆分对象存储

## 15.3 数据分类

### SQLite 中保存

- session 索引
- message 元数据
- summary
- long-term memory
- state snapshots
- prompt traces
- tool call logs
- media metadata
- config

### 文件系统中保存

- 角色卡 JSON
- 世界书 JSON
- session 原始消息流水
- 图片文件
- 参考图
- 上传文件
- 生成图片

## 15.4 迁移原则

所有存储访问都应通过 repository / service 抽象层，避免业务直接绑定 SQLite。

这样后续迁移 PostgreSQL 时：

- 结构化数据可迁移
- 文件仍保持本地或转对象存储

## 16. 后台任务与异步架构

## 16.1 需要异步化的任务

建议异步化的任务包括：

- 中期摘要生成
- 图片生成
- 文件文本提取
- 搜索缓存预热（可选）

## 16.2 第一阶段建议实现

第一阶段不必上复杂消息队列。

建议采用：

- FastAPI 后台任务
- 或轻量本地任务执行器

前提是：

- 每个任务可追踪状态
- 失败可重试
- 不阻塞主对话响应

## 17. 事件与可观察性架构

## 17.1 PromptTrace 是系统可观察性核心

每一轮都应保存：

- 输入
- 注入项
- 最终 `messages`
- 工具调用
- 原始输出
- 清洗结果
- 状态更新结果

## 17.2 除 PromptTrace 外的运行日志

系统还应记录：

- 任务执行日志
- 媒体处理日志
- Provider 调用日志
- 错误日志

第一阶段可以先保持文件日志 + 数据库关键记录。

## 19. 第一阶段实现架构建议

## 19.1 MVP-1 主闭环

第一阶段真正要打通的闭环应是：

```text
角色卡选择
  → 新建游玩会话
  → 发送消息
  → Prompt 管线组装
  → 单模型调用
  → 返回回复
  → Regex / state_update
  → 保存 PromptTrace
  → 前端展示状态与 Inspector
```

## 19.2 第一阶段必须直接支持的共享底座

- Prompt 管线
- PromptTrace
- StateStore
- 基础三层记忆
- 基础 media attachments
- 基础 web_search

## 19.3 第一阶段可以弱实现但不能缺席的能力

- 聊天模式基础入口
- 文件输入
- 图片上传
- 记忆中心基础查看

## 20. 风险与架构防线

## 20.1 最大风险

当前系统的最大风险不是“技术不会”，而是“功能很多导致底座失控”。

## 20.2 架构防线

因此必须守住以下边界：

- Prompt 生命周期统一归后端
- 三种模式共享底座，不复制实现
- Session 抽象统一
- PromptTrace 必须存在
- 媒体和文件必须是一等消息附件
- 状态与记忆必须作为独立子系统存在

---

## 21. 与 SillyTavern 的最终架构关系判断

## 21.1 可借鉴

- 角色卡 / 世界书 / 历史消息深度参与 Prompt 组装
- 多 provider 适配思路
- Prompt 调试能力的必要性

## 21.2 不继续沿用

- 单页面功能堆叠式架构
- 以 jQuery 风格 UI 组织业务逻辑
- 把复杂业务逻辑留在前端

## 21.3 最终判断

Izumi Studio 不是对 SillyTavern 的小修小补，而是：

- 业务目标继承 SillyTavern 的能力边界
- 运行架构改为“本地服务化平台”

## 22. 结论

Izumi Studio 的最合理系统架构不是“聊天网页 + 一堆附加功能”，而是：

- 以前后端分离为基础
- 以 Prompt 管线为中心
- 以记忆、状态、工具、多模态为共享底座
- 以创作 / 游玩 / 聊天为三种模式化业务入口

这套架构既能承接 SillyTavern 的能力边界，又能支持你当前已经确认的产品方向：

- 全局泉此方创作者框架
- 游玩模式线性会话
- Prompt Inspector 强可观察性
- 图片与文件输入输出
- 本地优先、后续可迁移
