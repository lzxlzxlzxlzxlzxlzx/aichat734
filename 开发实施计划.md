## 1. 计划目标

这份计划用于解决两个现实问题：

- 项目已经进入代码实现阶段，后续任务会越来越多，不能只靠即时上下文推进
- 你的设计文档已经很完整，后续每一步都应该“按文档落代码”，而不是凭印象实现

因此本计划的目标是：

- 明确接下来要做的所有主要阶段
- 明确每一步的输入文档
- 明确每一步的输出成果
- 明确阶段间依赖关系

## 7. 当前建议

基于当前代码状态，下一步最值得立刻做的是：

- `A-1 真实模型路由层`
- `B-1 Prompt 组装器`

原因是：

- 现在消息主链路已经有了
- 但回复仍然是 mock
- 如果这两步先完成，后面状态、记忆、多模态、Inspector 都能挂到真实主链路上

因此，接下来的实际开发优先级建议定为：

1. 真实模型调用
2. Prompt 管线
3. PromptTrace 完整化

这三步完成后，项目就会从“能跑通的后端原型”升级成“真正具备产品骨架的系统”。

- 统一空状态、骨架屏、错误态

2. 页面体验任务
- 调整页面留白
- 调整字号层级
- 调整侧栏宽度
- 调整消息列阅读宽度
- 调整滚动与吸底体验
- 调整窄屏与移动端表现

3. 质量检查任务
- 检查三个模式底部导航是否稳定
- 检查所有主流程是否无阻塞
- 检查 API 错误处理是否一致
- 检查页面刷新后的状态恢复
- 检查空列表、慢请求、失败请求表现

4. 自测清单
- 从聊天模式进入并完成一轮对话
- 从游玩模式选择角色卡并完成一轮游玩
- 从创作模式进入角色卡编辑并保存
- 从聊天 / 游玩 / 创作进行跨模式跳转
- 在游玩模式查看 Inspector
- 在聊天或游玩中上传图片和文件

5. 完成定义
- 三模式主流程全部可跑通
- 页面视觉达到统一、清晰、可演示水平
- 可以进入 MVP 收敛阶段

### 0.9 前端实现顺序建议（实际编码顺序）

建议实际编码顺序固定为：

1. `G-1-Task`
2. `G-2-Task`
3. `G-3-Task`
4. `G-6-Task` 中的基础附件上传与基础 Inspector 入口
5. `G-4-Task`
6. `G-5-Task`
7. `G-6-Task` 剩余部分
8. `G-7-Task`

这样安排的原因：

- 先把统一底座和游玩主链路打通，最快形成可见成果
- 然后把 Inspector 和附件的“基础能力”尽早挂上，避免后续大改结构
- 再补聊天和创作两个模式
- 最后统一 polish，而不是每个页面各自零散修样式
