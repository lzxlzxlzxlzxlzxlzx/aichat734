> 文档版本：v1.0

## 一、功能定位

### 1.1 解决的核心问题

Izumi Studio 的核心价值之一是“可观察性”。

无论是游玩模式、聊天模式还是创作模式，用户最终看到的只是模型输出后的展示文本，但系统内部实际上还发生了很多关键过程：

- 设定注入
- 世界书触发
- 记忆召回
- User Persona 注入
- Author's Note 注入
- 工具调用
- 输出清洗
- 状态更新提取

如果这些过程不可见，就很难调试人格漂移、世界观失真、记忆丢失、工具误触发等问题。

### 1.2 模块目标

Prompt 管线与检查器模块的目标是：

- 定义“用户输入 -> 模型请求 -> 模型原始返回 -> 最终展示结果”的统一处理链
- 为不同模式提供可配置但一致的 Prompt 组装机制
- 记录每一轮对话实际注入了哪些内容
- 为 Prompt Inspector 提供稳定的快照数据
- 把工具调用、Regex 清洗、状态提取等步骤统一纳入可追踪链路

### 1.3 模块边界

本模块负责：

- Prompt 组装流程
- 注入顺序规范
- 工具调用插入点
- 原始输出与清洗输出的区分
- Prompt Trace 快照结构
- Prompt Inspector 展示数据来源

本模块不负责：

- 世界书具体触发算法
- 记忆具体生成算法
- 状态变量具体存储结构
- 角色卡编辑与世界书编辑

这些模块只向 Prompt 管线提供输入数据。

## 二、总体结构

### 2.1 Prompt 生命周期

```text
用户输入
  ↓
输入预处理
  ↓
模式上下文收集
  ↓
共享注入项收集
  ↓
模式专属注入项收集
  ↓
messages 组装
  ↓
模型调用 / 工具调用
  ↓
原始响应接收
  ↓
输出后处理
  ├─ Regex 清洗
  ├─ state_update 提取
  ├─ 展示文本生成
  └─ Trace 快照保存
  ↓
Prompt Inspector 展示
```

### 2.2 模块内部子系统

建议拆为以下子系统：

- `输入预处理子系统`
- `上下文收集子系统`
- `messages组装子系统`
- `工具调用编排子系统`
- `输出后处理子系统`
- `PromptTrace快照子系统`
- `Prompt Inspector 展示映射子系统`

### 2.3 依赖的共享输入

Prompt 管线依赖以下上游模块提供数据：

- 当前模式配置
- 全局预设配置
- 角色卡数据
- 世界书触发结果
- User Persona
- 记忆内容
- Author's Note
- 历史消息窗口
- 工具调用结果
- 状态变量摘要
- Regex 脚本配置

## 三、核心设计原则

### 3.1 原始输入必须保留

任何模式下，用户原始输入都必须被单独记录，不能只保留清洗或拼装后的版本。

### 3.2 注入项要可枚举、可解释

所有注入到模型上下文中的内容都应当能被列举，并标明：

- 来源
- 类型
- 注入顺序
- 长度 / token 占比
- 是否被裁剪

### 3.3 模式共享底层，模式决定策略

游玩、聊天、创作应共享同一套 Prompt 管线框架，但各自可以定义：

- 注入项来源不同
- 注入顺序不同
- 裁剪优先级不同
- 输出后处理不同

### 3.4 历史结果必须快照化

Prompt Inspector 不应完全依赖“临时回算”。

每一轮实际发送给模型的内容都应尽量保存快照，否则：

- 角色卡更新后历史会失真
- 世界书修改后历史注入会错位
- 后续无法稳定复盘

### 3.5 预设本质上是 Prompt 的组成部分

预设不是独立于 Prompt 之外的配置层，而是模型请求中的系统级注入内容。

因此：

- 预设必须进入 Prompt 管线
- 预设必须参与注入顺序设计
- 预设必须出现在 Prompt Inspector 中
- 预设必须按模式决定显性程度与权重

### 3.6 泉此方预设是全局常驻创作者框架

结合现有 SillyTavern 预设分析，泉此方预设在 Izumi Studio 中不应仅被理解为“聊天人格”，而应被定义为：

- 一个全局常驻的创作者认知框架
- 一个把模型从“安全助手”拉回“虚构创作者 / 叙事协作者”的系统预设
- 一个在游玩、聊天、创作三种模式下都应存在的上层 Prompt 基底

它的关键作用不是单纯让模型“像泉此方说话”，而是：

- 建立创作身份
- 建立虚构叙事框架
- 降低机器人味、说教味、审查味
- 强化角色扮演、剧情生成和创作协作的接受度

### 3.7 泉此方预设全局常驻，但模式显性不同

泉此方预设虽然应在三种模式中常驻，但必须区分“认知框架常驻”和“人格外显程度”。

建议如下：

- `聊天模式`：创作者框架强激活，泉此方人格外显最强
- `游玩模式`：创作者框架强激活，但人格外显弱化，避免污染角色卡
- `创作模式`：创作者框架强激活，以创作搭子 / 创作助手语气出现

设计原则：

- 常驻的是“泉此方创作者框架”
- 不等于所有模式都要输出明显的泉此方口癖
- 游玩模式必须避免角色卡被泉此方人格覆盖

## 四、预设层设计

### 4.1 预设层分层

建议将预设系统拆为四层：

1. `全局核心预设`
2. `模式预设`
3. `人格 / 风格预设`
4. `动态业务注入项`

其中：

- 前三层属于系统级 Prompt 预设
- 最后一层由角色卡、世界书、记忆、工具结果等模块按轮提供

### 4.2 全局核心预设

全局核心预设应在三种模式中始终存在，主要负责：

- 基础创作者身份框架
- 虚构创作合法性框架
- Master / user / story self 的认知隔离
- 防机器人化、防说教化、防审查味
- 基础叙事风格与协作关系

这部分是 Izumi Studio 的最顶层 Prompt 基底。

### 4.3 模式预设

模式预设负责告诉模型当前“正在做什么”。

例如：

- 游玩模式：当前任务是角色剧情演绎与场景推进
- 聊天模式：当前任务是以泉此方助手人格进行自由讨论
- 创作模式：当前任务是协助生成、分析和修改设定草稿

### 4.4 人格 / 风格预设

人格 / 风格预设负责调节输出气质。

在当前设计中，泉此方预设横跨两部分：

- 作为全局核心预设的一部分，提供创作者框架
- 作为人格 / 风格预设的一部分，在不同模式中调整显性程度

### 4.5 动态业务注入项

动态业务注入项包括：

- 角色卡设定
- 世界书条目
- User Persona
- 长期记忆
- 中期摘要
- Author's Note
- 工具调用结果
- 状态变量摘要

这些内容按轮变化。

### 4.6 预设可见性要求

Prompt Inspector 中应把预设单独展示，而不是与角色卡、世界书、记忆混在一起。

建议至少区分：

- 全局核心预设
- 当前模式预设
- 泉此方人格 / 风格层

### 4.7 SillyTavern 预设导入与映射

Izumi Studio 需要支持从 SillyTavern 预设 JSON 导入，并将其转换为当前系统可用的预设层结构。

设计目标：

- 最大化复用现有 ST 预设资产
- 不要求用户手动重写全部预设
- 导入后仍然能进入 Prompt 管线并被 Prompt Inspector 解释

### 4.8 导入源格式

第一阶段建议支持：

- SillyTavern 预设 JSON

### 4.9 导入后的四类映射

导入后建议按以下四类映射：

1. `global_core`
2. `mode_specific`
3. `persona_style`
4. `compat_legacy`

说明：

- `global_core`：创作身份、虚构框架、防机器人化、防说教化
- `mode_specific`：对当前模式行为的约束
- `persona_style`：口吻、风格、人格外显
- `compat_legacy`：无法稳定映射但仍需保留的 ST 原始条目

### 4.10 变量系统映射

SillyTavern 预设中常见的 `setvar/getvar` 宏不应在导入时被完全抹平。

建议策略：

- 导入时保留变量定义与引用关系
- 在 Izumi Studio 内部建立“预设变量层”
- 发送前统一完成变量展开
- Prompt Inspector 可查看：
  - 原始变量名
  - 展开前值
  - 展开后值

### 4.11 兼容层条目

并非所有 ST 预设条目都能完美转成 Izumi 原生结构，因此需要 `compat_legacy` 层。

这类条目特点：

- 保留原始名称
- 保留原始文本
- 保留原始启用状态
- 在 Prompt 中仍可被注入
- 在 Inspector 中标记为“ST 兼容层”

### 4.12 导入后的用户操作

导入预设后，用户应能：

- 查看原始 ST 结构
- 查看转换后的 Izumi 分层结构
- 调整条目归属层
- 启用 / 禁用条目
- 手动把兼容层条目迁移为原生条目

## 五、输入预处理子系统

---

### 5.1 输入类型

系统需要统一处理的输入包括：

- 纯文本输入
- 场景描述补充
- Quick Reply 注入内容
- slash command 展开结果
- 图片输入
- 文件输入
- 跨模式引用请求

### 5.2 预处理目标

预处理阶段主要完成：

- 记录原始输入
- 解析显式命令
- 规范化多模态输入描述
- 标记是否需要跨模式引用
- 标记是否可能触发工具调用

### 5.3 输入预处理结果

建议产出统一结构：

```json
{
  "raw_user_input": "...",
  "normalized_input": "...",
  "attachments": [],
  "quick_reply_applied": [],
  "slash_commands": [],
  "reference_requests": [],
  "tool_call_hints": [],
  "mode": "play | chat | creation"
}
```

## 六、上下文收集子系统

### 6.1 共享上下文类型

所有模式都可能用到以下上下文：

- 全局核心预设
- 模式主提示
- User Persona
- 长期记忆
- 中期摘要
- 历史消息窗口
- 工具调用结果

### 6.2 游玩模式专属上下文

游玩模式会额外收集：

- 泉此方创作者框架（弱外显）
- 角色卡系统设定
- 角色卡 few-shot 示例
- 角色卡 Post History Instructions
- 世界书常驻设定
- 世界书触发条目
- Author's Note
- 状态变量摘要

### 6.3 聊天模式专属上下文

聊天模式会额外收集：

- 泉此方创作者框架（强外显）
- 聊天模式预设
- 跨模式引用片段

### 6.4 创作模式专属上下文

创作模式会额外收集：

- 泉此方创作者框架（中等外显）
- 当前项目元信息
- 当前世界书草稿内容
- 当前角色卡草稿内容
- 模板示例或 few-shot 输出样例

### 6.5 收集结果结构

建议以“注入项列表”的形式输出，而不是提前拼接长文本。

每个注入项建议包含：

```json
{
  "id": "uuid",
  "source_type": "preset_global | preset_mode | preset_persona | worldbook | card | memory | persona | tool | state | mode_preset",
  "source_id": "可选",
  "label": "用户可读名称",
  "content": "注入内容",
  "stage": "system_head | before_history | inside_history | after_history | current_input",
  "priority": 100,
  "token_estimate": 280,
  "mode": "play"
}
```

### 6.6 预设在收集阶段的表示方式

建议将预设拆成独立注入项，而不是预先拼成一整段 system prompt。

推荐最少拆分为：

- `preset_global_core`
- `preset_mode_specific`
- `preset_izumi_persona`
- `preset_st_compat_legacy`

这样有几个好处：

- Prompt Inspector 可以分开展示
- 后续可按模式调整权重
- 可单独裁剪低优先级风格项
- 不会和角色卡设定混淆

### 6.7 导入预设在收集阶段的处理

如果当前启用的预设来源于 SillyTavern 导入，则在收集阶段建议生成：

- 原生映射后的注入项
- 必要的兼容层注入项

推荐处理顺序：

1. 先收集映射后的 `global_core / mode_specific / persona_style`
2. 再收集 `compat_legacy`
3. 对 `compat_legacy` 设定较低优先级，但默认不直接丢弃

这样既能保证 Izumi 原生分层优先，又能保留导入预设的有效部分

## 七、messages 组装子系统

### 7.1 统一组装目标

Prompt 管线不应直接面对“巨大的拼接字符串”，而应先组装结构化消息项，再转成模型请求格式。

### 7.2 统一阶段划分

建议把注入阶段统一抽象为：

- `system_head`
- `before_history`
- `inside_history`
- `after_history`
- `current_input`

这样不同模式可以复用同一套组装框架。

### 7.3 预设层的统一顺序

三种模式下，预设相关内容建议统一遵守以下高层顺序：

1. 全局核心预设
2. 当前模式预设
3. 泉此方人格 / 风格层
4. ST 兼容层预设
5. 其他动态业务注入项

这意味着：

- 全局核心预设始终优先级最高
- 模式预设用于说明“当前在做什么”
- 泉此方人格层用于调节表达方式和创作者姿态
- ST 兼容层用于承接未完全映射的导入预设内容
- 角色卡、世界书、记忆等仍然是业务内容层

### 7.4 游玩模式建议顺序

游玩模式下建议按以下顺序组装：

1. 全局核心预设
2. 游玩模式预设
3. 泉此方创作者框架（弱外显）
4. 角色卡系统设定
5. 世界书 `before_char`
6. 角色卡深度提示 / few-shot
7. 世界书 `after_char`
8. 长期记忆
9. 中期摘要
10. Author's Note（前置）
11. 历史消息窗口
12. 历史中的 at_depth 插入项
13. Post History Instructions
14. Author's Note（后置）
15. 当前用户输入

说明：

- 游玩模式中泉此方预设应提供“创作者框架”和“防机器人味”的底层约束
- 不应强到覆盖角色卡声音
- Prompt Inspector 中必须单独展示该层

### 7.5 聊天模式建议顺序

聊天模式下建议按以下顺序组装：

1. 全局核心预设
2. 聊天模式预设
3. 泉此方创作者框架（强外显）
4. User Persona
5. 长期记忆
6. 中期摘要
7. 被引用的外部片段
8. 工具调用结果
9. 历史消息窗口
10. 当前用户输入

### 7.6 创作模式建议顺序

创作模式下建议按以下顺序组装：

1. 全局核心预设
2. 创作模式预设
3. 泉此方创作者框架（中等外显）
4. 当前项目目标
5. 当前草稿内容
6. 模板 / few-shot 示例
7. 长期记忆与创作笔记
8. 工具调用结果
9. 历史消息窗口
10. 当前用户输入

### 7.7 Token 裁剪策略

当上下文过长时，建议按模式区分裁剪顺序。

#### 游玩模式

优先裁剪：

1. 中期摘要中的旧段
2. 优先级低的世界书条目
3. 较远历史消息

尽量不裁剪：

- 全局核心预设
- 游玩模式预设
- 角色卡主设定
- 当前轮触发的核心世界书条目
- 锁定消息

#### 聊天模式

优先裁剪：

1. 较远历史闲聊
2. 体量过大的引用片段
3. 较旧中期摘要

尽量不裁剪：

- 全局核心预设
- 泉此方创作者框架
- 泉此方常驻设定
- 当前用户问题相关引用

#### 创作模式

优先裁剪：

1. 过长的历史讨论
2. 体量过大的模板示例
3. 较旧创作笔记

尽量不裁剪：

- 全局核心预设
- 创作模式预设
- 当前项目目标

### 7.8 最终 messages 结果

最终对外发送前，建议保留：

- 结构化注入项列表
- 最终 `messages` 数组
- 最终估算 token 信息

## 八、工具调用编排子系统

### 8.1 设计目标

工具调用不是 Prompt 管线之外的额外流程，而是 Prompt 生命周期的一部分。

### 8.2 触发位置

工具调用发生在首次模型推理阶段之后。

典型流程：

```text
组装 messages
  ↓
首次模型推理
  ↓
模型判断需要 tool_call
  ↓
后端执行工具
  ↓
tool_result 注回上下文
  ↓
模型继续完成回复
```

### 8.3 记录内容

PromptTrace 中建议记录：

- 工具名称
- 调用时机
- 调用参数
- 原始返回摘要
- 是否成功
- 是否二次注入

### 8.4 与模式的关系

- 游玩模式：偏剧情中的事实补充与实时知识
- 聊天模式：偏实时问题、资料查询与讨论辅助
- 创作模式：偏原著设定核对、资料收集

## 九、输出后处理子系统

### 9.1 输出三层结构

所有模式建议统一保留三层输出：

- `raw_response`
- `cleaned_response`
- `display_response`

### 9.2 Regex 清洗

Regex 清洗建议统一作为输出后处理步骤执行，而不是直接改写原始响应。

因此：

- 原始响应必须保留
- 清洗规则及命中结果需要记录
- 最终展示文本单独生成

### 9.3 state_update 提取

游玩模式下，输出后处理还需要：

- 扫描 `<state_update>` 或类似协议块
- 从响应中提取结构化状态更新
- 生成可持久化的状态变更结果

聊天模式和创作模式默认不依赖该步骤，但应保留兼容能力。

### 9.4 清洗结果记录

建议记录：

- 命中的 Regex 脚本列表
- 被移除的控制块类型
- `state_update` 解析结果
- 清洗前后长度变化

## 十、PromptTrace 快照子系统

### 10.1 设计目标

PromptTrace 是 Prompt Inspector 的数据底座。

它的核心目标是保存“这一轮真实发生过什么”，而不是事后猜测。

### 10.2 快照保存时机

建议在每轮完成时保存快照。

触发时机包括：

- 普通回复完成
- 工具调用链完成
- 流式输出结束

### 10.3 建议快照结构

```json
{
  "trace_id": "uuid",
  "session_id": "uuid",
  "message_id": "uuid",
  "mode": "play | chat | creation",
  "raw_user_input": "...",
  "normalized_input": "...",
  "preset_layers": {
    "global_core": [],
    "mode_specific": [],
    "izumi_persona": [],
    "st_compat_legacy": []
  },
  "injection_items": [],
  "final_messages": [],
  "token_stats": {
    "estimated_input": 0,
    "estimated_output": 0
  },
  "tool_calls": [],
  "raw_response": "...",
  "cleaned_response": "...",
  "display_response": "...",
  "regex_hits": [],
  "state_update": {
    "raw_block": "",
    "parsed": {}
  },
  "created_at": "timestamp"
}
```

### 10.4 为什么必须快照化

如果不快照保存，后续会出现：

- 世界书被修改后，历史轮次失真
- 角色卡更新后，历史 Prompt 无法还原
- 记忆被编辑后，历史注入内容不可追踪

所以 PromptTrace 应视为“历史证据”，不是临时缓存。

## 十一、Prompt Inspector 展示映射子系统

### 11.1 设计目标

Prompt Inspector 的目标不是把底层 JSON 原样扔给用户，而是把复杂的 Prompt 管线转换成可读、可分析、可对照的视图。

### 11.2 建议展示分区

建议拆成以下视图：

- `用户输入`
- `预设层`
- `注入明细`
- `最终 messages`
- `工具调用`
- `模型原始输出`
- `清洗结果`
- `状态更新`
- `Token 统计`

### 11.3 预设层视图

应单独展示：

- 全局核心预设
- 当前模式预设
- 泉此方人格 / 风格层
- ST 兼容层条目

并说明：

- 每层的注入顺序
- 每层的显性权重
- 是否被裁剪

### 11.4 注入明细视图

应展示：

- 注入项标签
- 来源类型
- 注入顺序
- 长度或 token 占比
- 是否被裁剪

### 11.5 模式差异化展示

#### 游玩模式重点展示

- 全局核心预设
- 游玩模式预设
- 泉此方创作者框架（弱外显）
- 世界书触发条目
- Author's Note
- 状态变量摘要
- `state_update` 块

#### 聊天模式重点展示

- 全局核心预设
- 聊天模式预设
- 泉此方创作者框架（强外显）
- 跨模式引用片段
- 工具调用结果

#### 创作模式重点展示

- 全局核心预设
- 创作模式预设
- 泉此方创作者框架（中等外显）
- 当前草稿注入
- 模板 / few-shot 示例
- 原著资料引用

### 11.6 开发者模式

开发者模式开启时允许展示：

- 原始响应全文
- 清洗前后的差异
- 原始 `state_update` 块
- 工具调用原始结果摘要

关闭时应默认隐藏控制信息，但内部快照仍然保留。

## 十二、异常与边界处理

### 12.1 注入项冲突

当不同模块提供的内容发生冲突时，系统应：

- 记录冲突来源
- 按优先级保留一方
- 在 Prompt Inspector 中标记冲突裁剪

### 12.2 预设层冲突

如果全局核心预设、模式预设和业务注入项之间出现明显冲突，系统应遵守：

1. 全局核心预设优先
2. 当前模式预设次之
3. Izumi 原生人格 / 风格层次之
4. ST 兼容层再次之
5. 业务注入项在不冲突范围内补充

特殊说明：

- 游玩模式中角色卡表层声音优先于泉此方人格外显
- 但不应推翻全局创作者框架本身

### 12.3 工具调用中断

系统应：

- 保留首次模型响应痕迹
- 标记工具调用失败
- 尽量继续完成主回复

### 12.4 Regex 脚本异常

系统应：

- 原始输出保留
- 清洗步骤降级跳过
- 在 Trace 中记录脚本错误

### 12.5 Token 超限

系统应：

- 按模式裁剪策略执行
- 记录被裁剪的注入项
- 在 Inspector 中可查看裁剪信息

## 十六、结论

Prompt 管线与检查器模块是 Izumi Studio 的核心共享底座之一。

它不是单纯的“调试功能”，而是整个系统的可控性来源：

- 没有它，就无法稳定解释模型行为
- 没有它，就无法可靠调试设定、记忆和工具调用
- 没有它，就无法支撑游玩模式和聊天模式的长期演进

只要这套管线先设计清楚，后续状态系统、记忆系统、系统架构和数据模型都会更容易收束。
