> 所属模块：共享能力
> 文档版本：v1.0

> 依赖文档：产品需求文档_V2.md、Prompt管线与检查器设计.md、游玩模块详细设计.md、聊天模块详细设计.md

## 一、功能定位

### 1.1 解决的核心问题

Izumi Studio 不应只是一套文本聊天系统。

你当前最看重的多模态能力主要集中在图片：

- 用户上传图片给模型看
- 模型基于图片理解当前场景或参考素材
- 系统基于剧情或描述生成图片
- 生成结果自然嵌入当前会话流

同时，多模态模块也应支持“文件作为输入上下文”，尤其是：

- `txt`
- `docx`
- `doc`

### 1.2 模块目标

多模态支持模块的目标是：

- 提供稳定的图片输入能力
- 提供稳定的文件输入能力
- 提供可配置的图片生成能力
- 让图片和消息会话天然绑定
- 让文件和消息会话天然绑定
- 支持角色一致性与风格参考图
- 让 Prompt Inspector 能看到图片输入、文件输入和图片生成提示词

### 1.3 模块边界

本模块负责：

- 图片上传
- 文件上传
- 图片引用与存储
- 文件解析与引用
- 图片理解输入格式
- 图片生成流程
- 参考图管理
- 媒体结果与会话绑定

本模块不负责：

- 具体视觉模型的最终供应商绑定
- 复杂图像编辑工作流
- 视频生成

## 二、能力范围

### 2.1 第一阶段重点

第一阶段建议优先做你最在意也最值回票价的两条能力：

- `图片理解`
- `图片生成`

同时建议一起纳入一个轻量但很实用的能力：

- `文件输入理解`

### 2.2 适用模式

#### 图片理解

适用于：

- 游玩模式
- 聊天模式
- 创作模式

#### 图片生成

优先适用于：

- 游玩模式
- 聊天模式

创作模式后续也可扩展为“为角色卡 / 世界书生成视觉素材”。

#### 文件输入理解

适用于：

- 游玩模式
- 聊天模式
- 创作模式

## 三、文件输入子系统

### 3.1 设计目标

文件输入的目标不是做完整文档编辑器，而是让用户能把已有资料作为当前对话的输入上下文。

### 3.2 第一阶段重点格式

建议优先支持：

- TXT
- DOCX
- DOC

后续可扩展：

- PDF
- Markdown
- HTML

### 3.3 支持方式

建议支持：

- 点击上传
- 拖拽上传

第一阶段不要求像图片那样支持粘贴。

### 3.4 存储策略

上传文件建议保存到：

- `data/files/uploads/`

并为每个文件生成元数据记录。

### 3.5 解析策略

#### TXT

- 直接按 UTF-8 为主读取
- 若编码不明则尝试常见编码回退

#### DOCX

- 提取正文文本
- 尽量保留段落结构
- 第一阶段不要求完整保留复杂样式、批注、页眉页脚

#### DOC

- 允许支持，但第一阶段可以采取“尽力提取文本”的策略
- 如解析失败，应明确提示用户转换为 `docx` 或 `txt`

### 3.6 文件解析结果

解析后建议生成：

- 原文件元数据
- 抽取出的纯文本内容
- 可选摘要

### 3.7 与消息的关系

文件和图片一样，应作为消息附件进入会话流。

建议每条消息可附带：

- 0 到多张图片
- 0 到多个文件

### 3.8 文件在 Prompt 中的表示

Prompt 管线中建议将文件输入表示为结构化附件：

```json
{
  "type": "document",
  "file_id": "uuid",
  "file_name": "设定草稿.docx",
  "mime_type": "application/vnd.openxmlformats-officedocument.wordprocessingml.document",
  "extracted_text": "文档提取文本...",
  "summary": "可选摘要"
}
```

### 3.9 注入策略

文件通常体量比图片大，因此不建议把原文永远全文注入。

建议策略：

- 小文件：可直接全文进入当前轮 Prompt
- 大文件：优先生成摘要，再按需附加片段
- 用户可选择“全文理解”或“摘要理解”

### 3.10 Prompt Inspector 可见性

Prompt Inspector 应展示：

- 本轮上传了哪些文件
- 文件类型和大小
- 是否成功解析
- 实际进入 Prompt 的内容是全文、摘要还是节选

## 四、图片输入子系统

---

### 4.1 设计目标

图片输入要做到足够自然，不应让用户感觉是在走单独的附件上传流程。

### 4.2 支持方式

建议支持：

- 粘贴图片
- 点击上传
- 拖拽上传

### 4.3 支持格式

第一阶段建议支持：

- PNG
- JPG / JPEG
- WEBP
- GIF（静态帧）

### 4.4 输入限制

建议单张最大：

- 20MB

### 4.5 存储策略

上传图片保存到：

- `data/images/uploads/`

消息中只引用图片 ID 或文件路径，不重复保存二进制内容。

### 4.6 与消息的关系

图片不应被设计成“独立于消息存在”的对象，而应作为消息附件的一部分进入会话流。

建议每条消息可附带：

- 0 到多张图片
- 每张图片可有可选说明文字

### 4.7 图片理解在 Prompt 中的表示

Prompt 管线中需要为图片输入生成结构化附件描述，例如：

```json
{
  "type": "image",
  "image_id": "uuid",
  "file_path": "data/images/uploads/...",
  "mime_type": "image/png",
  "caption": "可选说明"
}
```

### 4.8 Prompt Inspector 可见性

Prompt Inspector 应展示：

- 本轮上传了哪些图片
- 图片文件信息
- 图片是否进入模型请求
- 附带说明文字是什么

## 五、图片理解子系统

### 5.1 设计目标

图片理解应服务于对话，而不是成为孤立的识图功能。

### 5.2 适用场景

#### 游玩模式

- 场景参考
- 人物服装参考
- 地图 / 界面 / 图片线索识别

#### 聊天模式

- 日常看图讨论
- 设定参考图讨论
- 发图让模型吐槽或分析

#### 创作模式

- 角色官图参考
- 风格图参考
- 视觉素材辅助世界书条目生成

### 5.3 模型适配

需要兼容“支持视觉输入的模型”和“纯文本模型”两类情况。

如果当前模型不支持视觉：

- 前端应提示
- 或自动禁用图片理解入口

### 5.4 输入到模型的方式

对于支持视觉输入的模型，图片应随当前消息一起发送。

如果同轮包含多张图片：

- 按用户上传顺序发送
- 保留每张图片的说明文字

### 5.5 与上下文组装的关系

图片理解是“当前用户输入”的一部分，不应作为长期常驻上下文。

但其分析结论后续可被：

- 摘要系统压缩
- 手动标记为长期记忆

## 六、文件理解子系统

### 6.1 设计目标

文件理解应服务于“带资料的对话”，而不是变成独立阅读器。

### 6.2 适用场景

#### 游玩模式

- 上传规则文档或线索文档，让模型结合剧情理解
- 上传设定文档辅助当前角色扮演

#### 聊天模式

- 上传 `txt` 或 `word` 文档，请模型总结、分析、吐槽、提炼重点

#### 创作模式

- 上传已有设定稿、人物小传、旧版文档
- 让模型基于资料生成或修改世界书 / 角色卡

### 6.3 与上下文组装的关系

文件理解通常属于“当前输入增强”，而不是常驻上下文。

如果文件内容很重要，后续可通过：

- 摘要系统压缩
- 手动标记长期记忆

### 6.4 输出形式

文件理解的结果仍以普通文本回复返回，但 Prompt Inspector 中应保留文件注入痕迹。

## 七、图片生成子系统

### 7.1 设计目标

图片生成不是单独的图片工具页，而应成为会话中的自然能力。

### 7.2 触发方式

建议支持两类：

#### 手动触发

- 点击“生成插图”
- 右键某条消息“为此场景生成图片”
- 单独进入图片生成面板

#### 自动触发

- 游玩模式可选开启
- 在模型回复后，根据当前场景描述自动生成插图

第一阶段建议优先手动触发，自动触发先做预留。

### 7.3 提示词生成流程

建议流程：

```text
当前场景描述
  + 角色外貌设定
  + 世界书视觉设定
  + 固定风格词
  ↓
生成图片提示词
  ↓
用户预览 / 编辑
  ↓
调用图片生成服务
```

### 7.4 图片生成服务

根据 PRD，候选服务包括：

- DALL-E 3
- Qwen Image 2.0 Pro
- Stable Diffusion（本地）
- Flux

### 7.5 生成结果管理

生成图片保存到：

- `data/images/generated/{session_id}/`

每张图建议记录：

- 图片 ID
- 会话 ID
- 关联消息 ID
- 使用服务
- 提示词
- 使用的参考图
- 生成时间

### 7.6 结果在会话中的呈现

图片生成完成后，应作为一条“媒体消息”或“消息附件”嵌入当前会话流。

用户在聊天 / 游玩过程中应能直接看到：

- 生成图缩略图
- 提示词入口
- 重新生成入口

## 八、参考图与一致性子系统

### 8.1 设计目标

如果没有参考图机制，角色形象很容易每次都变。

### 8.2 参考图存储

参考图保存到：

- `data/images/references/{card_id}/`

### 8.3 参考图用途

参考图主要用于：

- 角色外貌一致性
- 场景风格一致性
- 某个角色卡的长期视觉锚点

### 8.4 参考图标记

建议用户可标记参考图类型：

- 主角外貌参考
- 场景风格参考
- 服装参考

### 8.5 第一阶段建议能力

第一阶段先支持：

- 上传参考图
- 生成时选择参考图
- 记录本次生成使用了哪张参考图

更复杂的自动检索最相关参考图，可先延后。

## 九、消息流与会话绑定

### 9.1 设计目标

图片不是孤立素材仓库中的文件，而应与消息历史严格绑定。

### 9.2 绑定关系

建议：

- 上传图绑定到某条用户消息
- 上传文件绑定到某条用户消息
- 生成图绑定到某条模型消息或用户触发动作

### 9.3 会话导出

后续导出会话时，应允许：

- 只导出文字
- 文字 + 图片引用
- 文字 + 文件引用
- 导出该会话关联的全部生成图

## 十、Prompt Inspector 与可追踪性

### 10.1 图片理解场景

Prompt Inspector 应展示：

- 当前消息附带了哪些图片
- 图片的说明文字
- 图片是否成功进入模型输入

### 10.2 文件理解场景

Prompt Inspector 应展示：

- 当前消息附带了哪些文件
- 文件解析是否成功
- 实际注入的是全文、摘要还是节选
- 文件内容是否被裁剪

### 10.3 图片生成场景

Prompt Inspector 应展示：

- 生成提示词
- 是否使用参考图
- 使用了哪个服务
- 生成参数摘要

### 10.4 设计意义

你非常重视图片输入输出，所以这块必须可追踪，不然以后很难调：

- 为什么这张图生成成这样
- 为什么这次角色不像上次
- 为什么模型对图的理解和用户预期不一致

文件输入也一样：

- 为什么模型没读到文档重点
- 为什么这次只用了摘要没用全文
- 为什么 doc 文档解析失败

## 十一、异常与边界处理

### 11.1 当前模型不支持视觉

系统应：

- 提示当前模型不支持图片理解
- 允许用户切换模型
- 不直接吞掉图片输入

### 11.2 文件解析失败

系统应：

- 明确提示解析失败
- 保留原文件记录
- 建议用户改传 `txt` 或 `docx`

### 11.3 图片上传失败

系统应：

- 给出明确失败反馈
- 不发送不完整消息

### 11.4 图片生成失败

系统应：

- 记录失败信息
- 保留提示词
- 支持用户重试

### 11.5 参考图缺失

系统应：

- 允许无参考图继续生成
- 明确提示当前未使用角色一致性约束

## 十四、结论

多模态支持，尤其是图片输入输出和文件输入，不是 Izumi Studio 的附属功能，而是非常重要的体验增强器。

它的实现不一定比文本系统复杂很多，但对产品质感提升很大：

- 图片输入让模型真正能“看图”
- 文件输入让模型真正能“读资料”
- 图片生成让剧情和聊天有更强沉浸感
- 参考图机制让角色视觉表现更稳定

这部分非常值得在第一阶段就认真设计并尽早落地。

## 十五、V1.1 存储、展示与 LLM 关系定稿

本节用于明确当前阶段已经确定的实现方案。若与前文较早版本存在表述差异，以本节为准。

### 15.1 总体原则

当前阶段统一采用：

- 文件本体落项目目录，不直接存入 SQLite。
- 媒体元数据进入 `media_assets` 表。
- 媒体与消息的关系进入 `message_attachments` 表。
- 前端一律通过后端提供的媒体接口访问文件，不直接拼接本地路径。
- 图片与文件都先作为“消息附件”存在，再决定是否进一步进入 LLM。

这样设计的原因是：

- SQLite 适合存元数据，不适合长期承载大量二进制内容。
- 项目仍在本地单机阶段，磁盘文件 + SQLite 最简单、最稳定、最好调试。
- 后续如果迁移到对象存储，只需要替换文件存储层，不需要推翻数据库模型。

### 15.2 存储模型

#### 15.2.1 数据库存储

媒体系统当前统一使用两张核心表：

- `media_assets`
  - 记录媒体本体元数据。
  - 包括 `id`、`media_type`、`category`、`file_name`、`file_path`、`mime_type`、`size_bytes`、`meta`、`created_at`。
- `message_attachments`
  - 记录某条消息绑定了哪些媒体。
  - 包括 `message_id`、`media_asset_id`、`attachment_type`、`order_index`、`caption`。

设计约束如下：

- `media_assets` 只描述“文件本身”。
- `message_attachments` 只描述“文件在某条消息中的使用方式”。
- 同一个 `media_asset` 可以被多条消息重复引用。
- 后续角色头像、封面、参考图、生成图，也尽量继续复用 `media_assets`，不要额外分裂出新的媒体表。

#### 15.2.2 文件系统存储

当前阶段文件本体统一存放在项目 `data/` 目录内。

目录约定如下：

- `data/uploads/image/`
  - 用户上传的输入图片。
- `data/uploads/document/`
  - 用户上传的输入文件。
- `data/images/generated/`
  - 模型生成图片。
- `data/images/references/`
  - 角色参考图、风格参考图。
- `data/exports/`
  - 会话或项目导出产物。

当前代码已落地的最小实现是：

- `data/uploads/image/`
- `data/uploads/document/`

后续目录可以继续细化，但不改变核心原则。

#### 15.2.3 文件命名规则

媒体文件在磁盘上的命名规则确定为：

- 使用 `uuid + "_" + 净化后的原始文件名`

示例：

- `a4879c8a-d6f2-45bd-acab-e85518b2cf9e_note.txt`
- `ce7fd3ab-0fbf-4018-8b1c-d6fa4364ba79_reference.png`

这样做的目的是：

- 避免重名覆盖。
- 保留人类可读的原始文件名。
- 方便通过文件名反查数据库记录。
- 便于后续做导出、备份和垃圾清理。

### 15.3 图片与文件的分类

#### 15.3.1 `media_type`

媒体本体当前只分两类：

- `image`
- `document`

#### 15.3.2 `category`

媒体来源和用途使用 `category` 区分：

- `upload`
  - 用户主动上传。
- `generated`
  - 系统或模型生成。
- `reference`
  - 参考图、风格图。
- `cover`
  - 角色卡封面。
- `avatar`
  - 头像。

#### 15.3.3 `attachment_type`

媒体在消息中的使用方式使用 `attachment_type` 区分：

- `input_image`
  - 用户输入图片。
- `input_document`
  - 用户输入文档。
- `generated_image`
  - 生成图结果。

这三个字段不要混用：

- `media_type` 解决“它是什么文件”。
- `category` 解决“它从哪里来、主要做什么”。
- `attachment_type` 解决“它在这条消息里扮演什么角色”。

### 15.4 前端展示规则

#### 15.4.1 通用原则

前端不直接使用 `file_path` 作为展示地址，而是始终通过后端提供的下载接口访问媒体。

当前约定的接口形式：

- `GET /v1/media/{media_asset_id}`
  - 获取媒体元数据。
- `GET /v1/media/{media_asset_id}/download`
  - 获取媒体文件内容。

这样做的原因是：

- 可以隐藏真实磁盘路径。
- 以后迁移对象存储时，前端无需改协议。
- 可以在后端统一处理权限、鉴权、日志和缓存。

#### 15.4.2 图片展示

图片在消息区中的展示规则如下：

- 图片作为消息附件显示在气泡内部或气泡下方。
- 多张图片按 `order_index` 顺序展示。
- 每张图片可以带 `caption`。
- 点击缩略图后进入大图预览。
- 图片加载失败时展示占位提示，不影响整条消息文本展示。

第一阶段不要求：

- 图片编辑器。
- 智能裁切。
- 多规格缩略图服务。

#### 15.4.3 文件展示

文件在消息区中的展示规则如下：

- 文件不直接展示全文，而是展示附件卡片。
- 卡片中至少展示：
  - 原始文件名
  - 文件类型
  - 文件大小
  - 上传时间
  - 可选备注 `caption`
- 文件卡片提供两个基础操作：
  - 下载/打开原文件
  - 查看系统解析状态

如果文件已成功解析，前端还应支持展示：

- 文本预览摘要
- 是否已截断
- 解析失败原因

#### 15.4.4 生成图展示

模型生成图片的展示规则如下：

- 生成图也视为消息附件，而不是独立于会话历史存在。
- 生成完成后，以 `generated_image` 形式绑定到一条消息。
- 用户在会话流中应直接看到：
  - 生成图缩略图
  - 重新生成入口
  - 查看提示词入口

### 15.5 图片与 LLM 的关系

#### 15.5.1 图片本质

图片不是“长期文本记忆”，而是“当前轮输入附件”。

也就是说：

- 图片的主要作用是参与当前轮或最近几轮的视觉理解。
- 图片不会自动永久进入每一轮 Prompt。
- 后续如果某张图片的理解结果很重要，应通过摘要、状态、记忆等机制沉淀，而不是无限保留原图输入。

#### 15.5.2 图片进入 LLM 的条件

图片是否真正进入模型请求，取决于当前模型是否支持视觉输入。

分两种情况：

- 模型支持视觉输入
  - 图片作为多模态输入附件进入模型请求。
  - 顺序按 `order_index` 保留。
  - `caption` 作为辅助说明一并传入。
- 模型不支持视觉输入
  - 前端应提示当前模型不支持图片理解。
  - 后端不应假装图片已经被模型读取。
  - 可以保留附件记录，但不应标记为“已进入模型请求”。

当前阶段的实现重点是先把：

- 图片上传
- 图片绑定消息
- Prompt Inspector 可追踪

打通；真正的多模态模型请求接入属于后续图片理解链路实现。

#### 15.5.3 图片与 Prompt Inspector

Prompt Inspector 中必须能看到：

- 当前轮绑定了哪些图片
- 图片顺序
- 图片 MIME type
- 图片 caption
- 当前模型是否支持视觉输入
- 该图片是否真实进入了模型请求

### 15.6 文件与 LLM 的关系

#### 15.6.1 文件本质

文件不是直接“喂给模型一个二进制附件”，而是：

- 先上传文件
- 再提取文本
- 再把提取后的文本结果作为当前轮 Prompt 的一部分注入

所以文件与 LLM 的关系本质上是：

- 文件本体用于留档、下载、复用
- 解析后的文本用于真正进入 LLM

#### 15.6.2 文件进入 LLM 的流程

当前确定的标准链路为：

1. 用户上传文件。
2. 系统写入 `media_assets`。
3. 用户发送消息时，通过 `message_attachments` 绑定该文件。
4. Prompt 组装阶段识别 `input_document` 附件。
5. 文档解析器从磁盘读取文件并提取文本。
6. 解析结果以 `document_attachment` 注入项进入 Prompt。
7. Prompt Inspector 保留解析状态、截断状态和注入内容痕迹。

#### 15.6.3 当前支持范围

当前阶段已确定支持：

- `txt`
- `md`
- `rtf`
- `docx`
- `doc`
- `pdf`

其中：

- `txt/md/rtf` 走文本读取与编码回退。
- `docx` 提取正文 XML 文本。
- `doc` 采用 best-effort 文本提取。
- `pdf` 只考虑可提取文本的 PDF，不考虑扫描版 OCR。

#### 15.6.4 文件注入策略

文件内容进入 Prompt 时不无限制注入全文。

当前规则确定为：

- 优先提取纯文本。
- 文本进入 Prompt 前做清洗和空白压缩。
- 按字符上限截断后注入。
- 若已截断，必须在注入内容中明确标记“已截断”。

这意味着：

- LLM 看到的是“处理后的文档文本片段”。
- 原文件始终保留，可供用户下载与后续更深处理。
- 如果后续需要“全文理解 / 摘要理解 / 片段检索”，应作为独立策略扩展，而不是改变当前基础链路。

#### 15.6.5 文件与 Prompt Inspector

Prompt Inspector 中必须能看到：

- 当前轮附带了哪些文档
- 每个文档的文件名、类型、大小
- 使用了哪个解析器
- 解析是否成功
- 注入的是全文还是截断片段
- 失败时的原因

### 15.7 图片、文件与消息关系

#### 15.7.1 统一关系模型

当前方案明确规定：

- 图片和文件都不直接挂在“会话”上。
- 它们首先挂在“消息”上。
- 会话只是通过消息历史间接拥有这些附件。

也就是说：

- 图片/文件的最小归属单位是消息。
- 会话复制、回溯、导出时，应按消息附件一起处理。

#### 15.7.2 复制与回溯

当前已经确定的行为：

- 会话复制时，需要复制附件绑定关系。
- 回溯和恢复快照时，附件应和对应消息一起恢复可见性。
- 编辑消息时，附件列表应允许被替换，而不是只改文本不改附件。

#### 15.7.3 导出

后续会话导出时，应支持三类导出级别：

- 仅文本
- 文本 + 附件引用
- 文本 + 附件实体文件

第一阶段不强行实现完整导出，但数据模型必须兼容这一目标。

### 15.8 当前阶段最终结论

当前阶段已经确定的正式方案如下：

- 图片和文件本体都存项目 `data/` 目录，不存入 SQLite 二进制字段。
- 数据库只存媒体元数据和消息绑定关系。
- 前端统一通过后端媒体接口展示与下载。
- 图片首先是消息附件，只有在模型支持视觉输入时才真正进入视觉请求。
- 文件首先是消息附件，但进入 LLM 时依赖“解析后的文本结果”，不是原二进制本体。
- Prompt Inspector 必须同时追踪：
  - 附件元数据
  - 是否真正进入模型
  - 解析成功/失败
  - 注入内容及截断情况

这套方案既能满足当前本地 SQLite 阶段的可用性，也为后续迁移到对象存储、多模态模型路由、摘要/检索增强预留了稳定接口。

### 15.9 前后端接口清单

本节用于补充第一阶段实现时需要直接对齐的接口与字段。

#### 15.9.1 媒体上传接口

- `POST /v1/media/upload`
  - 用途：上传图片或文件。
  - 输入：
    - `file`
    - `category`
  - 输出：
    - `asset.id`
    - `asset.media_type`
    - `asset.category`
    - `asset.file_name`
    - `asset.mime_type`
    - `asset.size_bytes`
    - `asset.download_url`

前端行为约定：

- 上传成功后，不立即默认发送消息。
- 上传成功后，先把返回的 `media_asset_id` 挂到待发送消息草稿上。
- 用户点击发送后，再通过消息接口绑定附件。

#### 15.9.2 媒体查询与下载接口

- `GET /v1/media/{media_asset_id}`
  - 用途：获取媒体元数据。
- `GET /v1/media/{media_asset_id}/download`
  - 用途：下载或打开媒体文件内容。

前端行为约定：

- 图片预览优先使用下载接口。
- 文件卡片中的“打开原文件 / 下载文件”统一使用下载接口。
- 不在前端拼接本地路径。

#### 15.9.3 消息发送接口中的附件字段

消息发送与编辑请求需要支持：

```json
{
  "content": "请阅读附件",
  "attachments": [
    {
      "media_asset_id": "uuid",
      "attachment_type": "input_document",
      "caption": "可选备注",
      "order_index": 0
    }
  ]
}
```

说明：

- `content` 可以为空，但 `attachments` 不能为空。
- `order_index` 用于保证图片和文件在同一条消息中的顺序。
- `caption` 是展示层字段，也可在图片理解时作为辅助说明。

#### 15.9.4 消息返回结构中的附件字段

消息查询、消息详情、发送返回值都应包含：

- 附件绑定信息
- 对应媒体元数据
- 下载地址

返回结构应至少满足：

```json
{
  "attachments": [
    {
      "id": "attachment_id",
      "message_id": "message_id",
      "media_asset_id": "asset_id",
      "attachment_type": "input_document",
      "order_index": 0,
      "caption": "可选备注",
      "asset": {
        "id": "asset_id",
        "media_type": "document",
        "category": "upload",
        "file_name": "setting.docx",
        "mime_type": "application/vnd.openxmlformats-officedocument.wordprocessingml.document",
        "size_bytes": 12345,
        "download_url": "/v1/media/asset_id/download"
      }
    }
  ]
}
```

#### 15.9.5 Prompt Inspector 需暴露的多模态字段

Prompt Inspector 中，多模态相关可见信息至少应包含：

- 当前轮附件列表
- 每个附件的：
  - 文件名
  - 媒体类型
  - MIME type
  - 附件用途
  - 解析器名称
  - 是否解析成功
  - 是否截断
  - 进入 Prompt 的最终文本片段
- 对图片：
  - 是否真正进入视觉请求
- 对文件：
  - 是否进入文本注入链路

#### 15.9.6 当前第一阶段的实现边界

第一阶段明确只要求以下闭环：

- 图片可上传、绑定、展示、下载。
- 文件可上传、绑定、展示、下载。
- 文件可解析成文本并进入 Prompt。
- Prompt Inspector 可追踪附件和文档注入。

第一阶段暂不要求：

- 图片 OCR
- 扫描版 PDF OCR
- 图片向量检索
- 文档向量检索
- 文档分块召回
- 自动摘要缓存策略
- 图片缩略图派生规格服务
