# 多模态设计预案

> 所属模块：共享能力
> 文档版本：v1.0
> 日期：2026-04-15
> 依赖文档：产品需求文档_V2.md、Prompt管线与检查器设计.md、游玩模块详细设计.md、聊天模块详细设计.md

## 一、功能定位

### 1.1 解决的核心问题

Izumi Studio 不应只是一套文本聊天系统。

你当前最看重的多模态能力主要集中在图片：

- 用户上传图片给模型看
- 模型基于图片理解当前场景或参考素材
- 系统基于剧情或描述生成图片
- 生成结果自然嵌入当前会话流

同时，多模态模块也应支持“文件作为输入上下文”，尤其是：

- `txt`
- `docx`
- `doc`

### 1.2 模块目标

多模态支持模块的目标是：

- 提供稳定的图片输入能力
- 提供稳定的文件输入能力
- 提供可配置的图片生成能力
- 让图片和消息会话天然绑定
- 让文件和消息会话天然绑定
- 支持角色一致性与风格参考图
- 让 Prompt Inspector 能看到图片输入、文件输入和图片生成提示词

### 1.3 模块边界

本模块负责：

- 图片上传
- 文件上传
- 图片引用与存储
- 文件解析与引用
- 图片理解输入格式
- 图片生成流程
- 参考图管理
- 媒体结果与会话绑定

本模块不负责：

- 具体视觉模型的最终供应商绑定
- 复杂图像编辑工作流
- 视频生成

## 二、能力范围

### 2.1 第一阶段重点

第一阶段建议优先做你最在意也最值回票价的两条能力：

- `图片理解`
- `图片生成`

同时建议一起纳入一个轻量但很实用的能力：

- `文件输入理解`

### 2.2 适用模式

#### 图片理解

适用于：

- 游玩模式
- 聊天模式
- 创作模式

#### 图片生成

优先适用于：

- 游玩模式
- 聊天模式

创作模式后续也可扩展为“为角色卡 / 世界书生成视觉素材”。

#### 文件输入理解

适用于：

- 游玩模式
- 聊天模式
- 创作模式

## 三、文件输入子系统

### 3.1 设计目标

文件输入的目标不是做完整文档编辑器，而是让用户能把已有资料作为当前对话的输入上下文。

### 3.2 第一阶段重点格式

建议优先支持：

- TXT
- DOCX
- DOC

后续可扩展：

- PDF
- Markdown
- HTML

### 3.3 支持方式

建议支持：

- 点击上传
- 拖拽上传

第一阶段不要求像图片那样支持粘贴。

### 3.4 存储策略

上传文件建议保存到：

- `data/files/uploads/`

并为每个文件生成元数据记录。

### 3.5 解析策略

#### TXT

- 直接按 UTF-8 为主读取
- 若编码不明则尝试常见编码回退

#### DOCX

- 提取正文文本
- 尽量保留段落结构
- 第一阶段不要求完整保留复杂样式、批注、页眉页脚

#### DOC

- 允许支持，但第一阶段可以采取“尽力提取文本”的策略
- 如解析失败，应明确提示用户转换为 `docx` 或 `txt`

### 3.6 文件解析结果

解析后建议生成：

- 原文件元数据
- 抽取出的纯文本内容
- 可选摘要

### 3.7 与消息的关系

文件和图片一样，应作为消息附件进入会话流。

建议每条消息可附带：

- 0 到多张图片
- 0 到多个文件

### 3.8 文件在 Prompt 中的表示

Prompt 管线中建议将文件输入表示为结构化附件：

```json
{
  "type": "document",
  "file_id": "uuid",
  "file_name": "设定草稿.docx",
  "mime_type": "application/vnd.openxmlformats-officedocument.wordprocessingml.document",
  "extracted_text": "文档提取文本...",
  "summary": "可选摘要"
}
```

### 3.9 注入策略

文件通常体量比图片大，因此不建议把原文永远全文注入。

建议策略：

- 小文件：可直接全文进入当前轮 Prompt
- 大文件：优先生成摘要，再按需附加片段
- 用户可选择“全文理解”或“摘要理解”

### 3.10 Prompt Inspector 可见性

Prompt Inspector 应展示：

- 本轮上传了哪些文件
- 文件类型和大小
- 是否成功解析
- 实际进入 Prompt 的内容是全文、摘要还是节选

## 四、图片输入子系统

---

### 4.1 设计目标

图片输入要做到足够自然，不应让用户感觉是在走单独的附件上传流程。

### 4.2 支持方式

建议支持：

- 粘贴图片
- 点击上传
- 拖拽上传

### 4.3 支持格式

第一阶段建议支持：

- PNG
- JPG / JPEG
- WEBP
- GIF（静态帧）

### 4.4 输入限制

建议单张最大：

- 20MB

### 4.5 存储策略

上传图片保存到：

- `data/images/uploads/`

消息中只引用图片 ID 或文件路径，不重复保存二进制内容。

### 4.6 与消息的关系

图片不应被设计成“独立于消息存在”的对象，而应作为消息附件的一部分进入会话流。

建议每条消息可附带：

- 0 到多张图片
- 每张图片可有可选说明文字

### 4.7 图片理解在 Prompt 中的表示

Prompt 管线中需要为图片输入生成结构化附件描述，例如：

```json
{
  "type": "image",
  "image_id": "uuid",
  "file_path": "data/images/uploads/...",
  "mime_type": "image/png",
  "caption": "可选说明"
}
```

### 4.8 Prompt Inspector 可见性

Prompt Inspector 应展示：

- 本轮上传了哪些图片
- 图片文件信息
- 图片是否进入模型请求
- 附带说明文字是什么

## 五、图片理解子系统

### 5.1 设计目标

图片理解应服务于对话，而不是成为孤立的识图功能。

### 5.2 适用场景

#### 游玩模式

- 场景参考
- 人物服装参考
- 地图 / 界面 / 图片线索识别

#### 聊天模式

- 日常看图讨论
- 设定参考图讨论
- 发图让模型吐槽或分析

#### 创作模式

- 角色官图参考
- 风格图参考
- 视觉素材辅助世界书条目生成

### 5.3 模型适配

需要兼容“支持视觉输入的模型”和“纯文本模型”两类情况。

如果当前模型不支持视觉：

- 前端应提示
- 或自动禁用图片理解入口

### 5.4 输入到模型的方式

对于支持视觉输入的模型，图片应随当前消息一起发送。

如果同轮包含多张图片：

- 按用户上传顺序发送
- 保留每张图片的说明文字

### 5.5 与上下文组装的关系

图片理解是“当前用户输入”的一部分，不应作为长期常驻上下文。

但其分析结论后续可被：

- 摘要系统压缩
- 手动标记为长期记忆

## 六、文件理解子系统

### 6.1 设计目标

文件理解应服务于“带资料的对话”，而不是变成独立阅读器。

### 6.2 适用场景

#### 游玩模式

- 上传规则文档或线索文档，让模型结合剧情理解
- 上传设定文档辅助当前角色扮演

#### 聊天模式

- 上传 `txt` 或 `word` 文档，请模型总结、分析、吐槽、提炼重点

#### 创作模式

- 上传已有设定稿、人物小传、旧版文档
- 让模型基于资料生成或修改世界书 / 角色卡

### 6.3 与上下文组装的关系

文件理解通常属于“当前输入增强”，而不是常驻上下文。

如果文件内容很重要，后续可通过：

- 摘要系统压缩
- 手动标记长期记忆

### 6.4 输出形式

文件理解的结果仍以普通文本回复返回，但 Prompt Inspector 中应保留文件注入痕迹。

## 七、图片生成子系统

### 7.1 设计目标

图片生成不是单独的图片工具页，而应成为会话中的自然能力。

### 7.2 触发方式

建议支持两类：

#### 手动触发

- 点击“生成插图”
- 右键某条消息“为此场景生成图片”
- 单独进入图片生成面板

#### 自动触发

- 游玩模式可选开启
- 在模型回复后，根据当前场景描述自动生成插图

第一阶段建议优先手动触发，自动触发先做预留。

### 7.3 提示词生成流程

建议流程：

```text
当前场景描述
  + 角色外貌设定
  + 世界书视觉设定
  + 固定风格词
  ↓
生成图片提示词
  ↓
用户预览 / 编辑
  ↓
调用图片生成服务
```

### 7.4 图片生成服务

根据 PRD，候选服务包括：

- DALL-E 3
- Qwen Image 2.0 Pro
- Stable Diffusion（本地）
- Flux

### 7.5 生成结果管理

生成图片保存到：

- `data/images/generated/{session_id}/`

每张图建议记录：

- 图片 ID
- 会话 ID
- 关联消息 ID
- 使用服务
- 提示词
- 使用的参考图
- 生成时间

### 7.6 结果在会话中的呈现

图片生成完成后，应作为一条“媒体消息”或“消息附件”嵌入当前会话流。

用户在聊天 / 游玩过程中应能直接看到：

- 生成图缩略图
- 提示词入口
- 重新生成入口

## 八、参考图与一致性子系统

### 8.1 设计目标

如果没有参考图机制，角色形象很容易每次都变。

### 8.2 参考图存储

参考图保存到：

- `data/images/references/{card_id}/`

### 8.3 参考图用途

参考图主要用于：

- 角色外貌一致性
- 场景风格一致性
- 某个角色卡的长期视觉锚点

### 8.4 参考图标记

建议用户可标记参考图类型：

- 主角外貌参考
- 场景风格参考
- 服装参考

### 8.5 第一阶段建议能力

第一阶段先支持：

- 上传参考图
- 生成时选择参考图
- 记录本次生成使用了哪张参考图

更复杂的自动检索最相关参考图，可先延后。

## 九、消息流与会话绑定

### 9.1 设计目标

图片不是孤立素材仓库中的文件，而应与消息历史严格绑定。

### 9.2 绑定关系

建议：

- 上传图绑定到某条用户消息
- 上传文件绑定到某条用户消息
- 生成图绑定到某条模型消息或用户触发动作

### 9.3 会话导出

后续导出会话时，应允许：

- 只导出文字
- 文字 + 图片引用
- 文字 + 文件引用
- 导出该会话关联的全部生成图

## 十、Prompt Inspector 与可追踪性

### 10.1 图片理解场景

Prompt Inspector 应展示：

- 当前消息附带了哪些图片
- 图片的说明文字
- 图片是否成功进入模型输入

### 10.2 文件理解场景

Prompt Inspector 应展示：

- 当前消息附带了哪些文件
- 文件解析是否成功
- 实际注入的是全文、摘要还是节选
- 文件内容是否被裁剪

### 10.3 图片生成场景

Prompt Inspector 应展示：

- 生成提示词
- 是否使用参考图
- 使用了哪个服务
- 生成参数摘要

### 10.4 设计意义

你非常重视图片输入输出，所以这块必须可追踪，不然以后很难调：

- 为什么这张图生成成这样
- 为什么这次角色不像上次
- 为什么模型对图的理解和用户预期不一致

文件输入也一样：

- 为什么模型没读到文档重点
- 为什么这次只用了摘要没用全文
- 为什么 doc 文档解析失败

## 十一、异常与边界处理

### 11.1 当前模型不支持视觉

系统应：

- 提示当前模型不支持图片理解
- 允许用户切换模型
- 不直接吞掉图片输入

### 11.2 文件解析失败

系统应：

- 明确提示解析失败
- 保留原文件记录
- 建议用户改传 `txt` 或 `docx`

### 11.3 图片上传失败

系统应：

- 给出明确失败反馈
- 不发送不完整消息

### 11.4 图片生成失败

系统应：

- 记录失败信息
- 保留提示词
- 支持用户重试

### 11.5 参考图缺失

系统应：

- 允许无参考图继续生成
- 明确提示当前未使用角色一致性约束

## 十二、与其他模块的关系

### 12.1 对 Prompt 管线的支撑

多模态模块为 Prompt 管线提供：

- 图片输入附件结构
- 文件输入附件结构
- 图片生成记录
- 图片相关 PromptTrace 数据

### 12.2 对游玩模块的支撑

游玩模式通过多模态模块获得：

- 场景参考图输入
- 文件资料输入
- 剧情插图生成
- 角色形象一致性支撑

### 12.3 对聊天模块的支撑

聊天模式通过多模态模块获得：

- 发图聊天
- 发文件聊天
- 设定图讨论
- 随手生成图像的能力

### 12.4 对创作模式的支撑

创作模式通过多模态模块获得：

- 上传参考图辅助生成设定
- 上传 `word` / `txt` 资料稿辅助生成设定
- 未来扩展角色卡封面与视觉素材生成

### 12.5 对系统架构设计的反向要求

未来系统架构必须支持：

- 文件存储层
- 消息与媒体附件绑定
- 文档解析层
- 图片生成任务调度
- 媒体元数据持久化

## 十三、第一阶段开发建议切片

### 13.1 MVP-1 建议纳入

- 图片上传
- 图片理解输入
- TXT / DOCX / DOC 文件上传
- 文件解析与文件理解输入
- 手动图片生成
- 参考图上传与选择
- 会话中的图片结果展示
- Prompt Inspector 中的图片输入 / 文件输入 / 生成记录

### 13.2 MVP-1 建议延后

- 自动剧情插图

- 自动参考图匹配
- 更复杂文档格式支持
- 视频能力

## 十四、结论

多模态支持，尤其是图片输入输出和文件输入，不是 Izumi Studio 的附属功能，而是非常重要的体验增强器。

它的实现不一定比文本系统复杂很多，但对产品质感提升很大：

- 图片输入让模型真正能“看图”
- 文件输入让模型真正能“读资料”
- 图片生成让剧情和聊天有更强沉浸感
- 参考图机制让角色视觉表现更稳定

这部分非常值得在第一阶段就认真设计并尽早落地。
