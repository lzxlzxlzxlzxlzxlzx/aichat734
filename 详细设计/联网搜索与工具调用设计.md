## 一、功能定位

### 1.1 解决的核心问题

模型本身的知识存在两个限制：

- 不是实时的
- 对某些原著设定、现实信息、专业知识细节不够稳定

如果完全不接搜索，Izumi Studio 在以下场景会明显受限：

- 创作模式中需要核对原著 IP 设定
- 聊天模式中用户询问实时信息
- 游玩模式中剧情涉及模型不确定的现实知识

### 1.2 模块目标

联网搜索与工具调用模块的目标是：

- 让模型在需要时自主调用搜索工具
- 把搜索结果安全地回注入 Prompt 管线
- 不打断主对话体验
- 在 Prompt Inspector 中可追踪本轮调用过程

### 1.3 模块边界

本模块负责：

- 搜索工具定义
- 工具调用流程
- 查询词生成与执行
- 搜索结果摘要回注
- 工具调用记录与缓存

本模块不负责：

- Prompt 最终组装顺序
- 第三方搜索供应商的最终商业选择
- 前端页面细节

## 二、总体结构

### 2.1 工具调用生命周期

```text
模型收到 messages
  ↓
判断是否需要搜索
  ↓
输出 tool_call
  ↓
后端执行搜索
  ↓
返回 tool_result
  ↓
模型继续生成最终回复
  ↓
PromptTrace 保存调用记录
```

### 2.2 当前建议的工具范围

第一阶段建议只引入一个基础工具：

- `web_search`

后续再扩展：

- 原著资料专搜
- 本地资料库检索
- 图片检索

### 2.3 适用模式

工具调用支持：

- 创作模式
- 游玩模式
- 聊天模式

---

## 三、触发策略

### 3.1 设计原则

联网搜索不应默认每轮执行，而应是“按需触发”的系统工具。

### 3.2 典型触发场景

#### 创作模式

- 用户明确提及原著 IP
- 需要核对原著角色、地点、时间线、设定
- 需要对比多个来源的资料

#### 游玩模式

- 剧情涉及现实历史、地理、专业知识
- 当前世界书未覆盖，但剧情需要外部事实补充

#### 聊天模式

- 用户询问实时问题
- 用户要求查资料、查新闻、查人物近况

### 3.3 触发方式

第一阶段建议采用：

- 模型自主判断并发出 `tool_call`

同时允许后续扩展：

- 用户显式指令触发
- 某些模式下的系统强制触发

---

## 四、工具定义

### 4.1 基础工具格式

建议统一为：

```json
{
  "tool_name": "web_search",
  "arguments": {
    "query": "进击的巨人 希加尼斯区 时间线",
    "top_k": 5
  }
}
```

### 4.2 参数建议

第一阶段建议支持：

- `query`
- `top_k`
- `search_scope`（可选，后续扩展）

### 4.3 返回结果结构

建议标准化为：

```json
{
  "tool_name": "web_search",
  "query": "进击的巨人 希加尼斯区 时间线",
  "items": [
    {
      "title": "...",
      "url": "...",
      "snippet": "...",
      "source": "site"
    }
  ],
  "summary": "搜索结果摘要...",
  "success": true
}
```

## 五、搜索执行子系统

### 5.1 搜索接入优先级

根据 PRD，优先级建议为：

1. Tavily Search API
2. SerpAPI
3. Bing Search API

### 5.2 执行流程

```text
接收 tool_call
  ↓
提取 query
  ↓
检查缓存
  ↓
若无缓存则请求搜索服务
  ↓
标准化结果
  ↓
生成 tool_result
```

### 5.3 缓存策略

建议按查询词缓存：

- TTL：24 小时
- 相同 query 直接复用

### 5.4 失败处理

如果搜索失败：

- 不阻塞主对话
- 将失败信息写入工具记录
- 允许模型在无搜索结果情况下继续回复

---

## 六、结果回注子系统

### 6.1 设计目标

搜索结果不应直接原样塞给最终用户，而应先回注入模型，作为继续生成的依据。

### 6.2 回注方式

建议将 `tool_result` 作为结构化注入项重新进入 Prompt 管线。

### 6.3 回注内容

建议包含：

- 查询词
- 搜索结果摘要
- 来源列表摘要

第一阶段不建议把整页长文本直接注入。

### 6.4 Prompt 层中的位置

搜索结果回注应在：

- 当前轮用户输入之后
- 最终模型生成之前

其最终位置由 Prompt 管线统一管理。

---

## 七、Prompt Inspector 与可追踪性

### 7.1 必须展示的内容

Prompt Inspector 中建议展示：

- 本轮是否触发工具调用
- 调用了什么工具
- 查询词是什么
- 是否命中缓存
- 原始结果摘要
- 是否成功

### 7.2 展示层次

普通模式下：

- 只展示“本轮调用了联网搜索”及摘要

开发者模式下：

- 展示完整查询词
- 展示标准化结果
- 展示回注内容

## 八、与三种模式的关系

### 8.1 创作模式

创作模式最适合工具调用，尤其是原著 IP 相关场景。

这里的目标不是实时问答，而是“为创作提供可信参考”。

### 8.2 游玩模式

游玩模式下工具调用应更克制。

原则上只有在剧情明确需要现实知识且世界书未覆盖时才触发，避免频繁打断沉浸感。

### 8.3 聊天模式

聊天模式下工具调用最自然，用户会直接问：

- 今天发生了什么
- 某个作品现在有什么新消息
- 某个知识点怎么解释

---

## 九、异常与边界处理

### 9.1 搜索结果为空

系统应：

- 返回空结果结构
- 允许模型继续回复
- 在 Prompt Inspector 中标记“无有效结果”

### 9.2 搜索结果冲突

如果不同来源内容冲突：

- 不由系统静默裁决真伪
- 由模型基于结果进行归纳
- Prompt Inspector 保留来源摘要

### 9.3 触发过于频繁

后续可考虑增加：

- 同一轮最多一次搜索
- 同一会话短时间内的搜索冷却

第一阶段可先不强制实现，只保留设计预留。

## 十、与其他模块的关系

### 10.1 对 Prompt 管线的支撑

工具调用模块为 Prompt 管线提供：

- 标准化 `tool_call`
- 标准化 `tool_result`
- 可记录的工具调用元数据

### 10.2 对聊天 / 游玩 / 创作模块的支撑

三种模式都通过该模块获得“按需联网搜索”的能力，而无需各自单独实现搜索流程。

### 10.3 对系统架构设计的反向要求

未来系统架构必须支持：

- 模型调用与工具调用串联
- 工具结果标准化
- 调用记录持久化
- 缓存层

---

## 十一、第一阶段开发建议切片

### 11.1 MVP-1 建议纳入

- 单一 `web_search` 工具
- 标准化 `tool_call` / `tool_result` 结构
- 查询缓存
- Prompt Inspector 中的调用记录展示

### 11.2 MVP-1 建议延后

- 多工具路由
- 高级搜索策略
- 多来源打分排序
- 原始网页深度抓取

---

## 十二、结论

联网搜索与工具调用模块虽然功能范围不大，但它对 Izumi Studio 很关键，因为它解决的是“模型不知道最新事实”这个基础问题。

它应该以轻量、标准化、可追踪的方式接入，而不是做成复杂的 Agent 系统。
