# 游玩模块详细设计

> 所属模块：游玩模式（Play Mode）
> 文档版本：v1.0
> 日期：2026-04-15
> 依赖文档：产品需求文档_V2.md、世界书生成系统详细设计.md、ReZero角色卡分析.md

---

## 一、功能定位

### 1.1 解决的核心问题

Izumi Studio 的游玩模式不是普通聊天页，而是一个面向角色扮演与剧情体验的沉浸式对话环境。

它要解决的核心问题有四个：

- 让用户可以围绕一个已发布角色卡稳定进入剧情，而不是每次重新拼 Prompt
- 让世界书、角色卡、状态系统、记忆系统在对话中自动协作
- 让用户可以以线性方式长期游玩，并在需要时复制当前对话继续尝试其他走向
- 让每一轮对话都可被检查、追踪、复盘，而不是只看到最终显示结果

### 1.2 模块目标

游玩模式的目标是：

- 选择一个角色卡进入沉浸式剧情对话
- 自动注入角色卡、世界书、Author's Note、记忆和状态上下文
- 支持线性会话、对话复制、重生成、消息编辑和长距离回溯
- 通过 Prompt Inspector 查看该轮实际发给模型的完整内容
- 通过状态栏 / 悬浮球实时展示剧情状态变化

### 1.3 模块边界

游玩模式负责：

- 加载并消费“已发布”的角色卡及其绑定世界书
- 创建和管理剧情会话
- 执行对话链路
- 展示消息、状态和 Prompt 详情

游玩模式不负责：

- 创建角色卡
- 编辑世界书
- 生成角色卡配置
- 世界书结构编辑

这些能力由创作模式负责，游玩模式只消费其产物。

### 1.4 典型用户故事

```
用户：
  选择“雷姆”角色卡，进入新剧情，选择第二个开场白开始对话。
  输入：“我推开宅邸的大门，外面在下雨。今天是不是出了什么事？”

系统执行：
  1. 读取角色卡、绑定世界书、默认参数、Author's Note、Regex、UI Schema
  2. 创建新的 Play Session
  3. 根据当前消息和最近上下文触发相关世界书条目
  4. 注入角色设定、世界条目、记忆、状态信息，组装完整 prompt
  5. 调用模型，流式返回回复
  6. 提取 <state_update> 块，更新状态栏和悬浮球
  7. 保存原始响应、清洗后文本、Prompt Trace、工具调用记录
  8. 用户可点击“详情”查看本轮 Prompt Inspector

系统交付：
  ✓ 一轮可追踪的角色剧情对话
  ✓ 可实时变化的状态面板
  ✓ 可长期使用的线性会话与对话复制机制
```

## 二、总体结构

### 2.1 游玩模块在系统中的位置

```text
角色卡选择界面
   ↓
开场白选择
   ↓
Play Session 创建
   ↓
对话主界面
   ↓
消息发送管线
   ├─ 角色卡注入
   ├─ 世界书触发
   ├─ 记忆召回
   ├─ Author's Note 注入
   ├─ 状态上下文注入
   ├─ 模型调用
   ├─ Regex / state_update 提取
   ├─ Prompt Trace 保存
   └─ UI 渲染更新
```

### 2.2 模块内部子系统

游玩模块建议拆为以下子系统：

- `角色卡选择子系统`
- `会话管理子系统`
- `消息发送与生成子系统`
- `世界书触发子系统`
- `消息操作子系统`
- `Prompt Inspector 子系统`
- `状态展示子系统`
- `导出与复盘子系统`

### 2.3 对外依赖

游玩模块依赖以下共享能力：

- 角色卡读取能力
- 世界书读取与触发能力
- 模型统一调度层
- Regex / 输出清洗管线
- 状态变量引擎
- 记忆读写能力
- Prompt Trace 存储能力
- 工具调用记录能力

### 2.4 对外暴露的数据与事件

游玩模块需要向系统其他部分暴露：

- 当前会话信息
- 消息列表与回溯位置
- 当前激活的状态变量
- 本轮 Prompt Trace
- 本轮工具调用记录
- 当前世界书触发明细
- 对话复制事件

## 三、核心用户流程

### 3.1 主流程概览

```text
进入游玩模式
  → 浏览角色卡
  → 选择角色卡
  → 选择开场白
  → 创建新会话或进入已有会话
  → 输入用户消息
  → 系统组装 Prompt
  → 模型回复
  → 清洗与状态更新
  → 显示消息与状态变化
  → 保存本轮 Trace
```

### 3.2 进入角色卡流程

1. 用户打开游玩模式首页。
2. 系统加载所有已发布角色卡的摘要数据。
3. 用户点击某个角色卡。
4. 系统检查该角色卡是否存在多个开场白。
5. 如存在多个开场白，弹出选择器。
6. 用户确认后，系统执行：
   - 创建新会话，或进入该角色卡的已有会话列表
   - 绑定角色卡当前发布版本
   - 绑定角色卡所引用的世界书版本
   - 初始化状态变量
   - 写入开场消息

### 3.3 发送消息流程

1. 用户输入文本，可附带场景描述或点击快捷回复。
2. 系统记录原始输入。
3. 系统收集本轮上下文素材：
   - 当前角色卡
   - 世界书触发条目
   - User Persona
   - Author's Note
   - 记忆内容
   - 状态变量摘要
   - 历史消息窗口
4. 系统组装 `messages` 数组并发送给模型。
5. 模型流式返回内容。
6. 系统记录原始响应。
7. 系统执行 Regex / `state_update` 提取与清洗。
8. 系统更新前端显示内容。
9. 系统保存 Prompt Trace 和相关记录。

### 3.4 对话复制与回溯流程

用户在任意消息节点可执行：

- 从当前会话创建一个全新的空对话
- 从任意历史节点复制当前对话
- 从历史消息编辑后继续
- 对最后一条回复重新生成并追加新 Swipe
- 将当前会话回溯到任意历史位置后继续

这些操作的核心原则是：

- 不破坏已保存的原始历史
- 新的剧情走向优先通过“复制对话”而不是“树状分支”来承载
- 用户始终只在一条线性剧情线上继续游玩

## 四、角色卡选择子系统

### 4.1 页面目标

游玩模式入口页应让用户快速从“已发布角色卡”中进入剧情，而不是先进入复杂配置面板。

### 4.2 页面内容

角色卡选择页需要展示：

- 封面图
- 角色名称
- 简介
- 标签
- 最近对话时间
- 当前发布版本号

### 4.3 支持操作

- 按名称搜索
- 按标签筛选
- 按最近使用排序
- 查看该角色卡已有剧情线列表
- 开始新剧情

### 4.4 边界约束

- 仅展示“已发布角色卡”
- 草稿角色卡不可直接进入游玩模式
- 游玩模式一次只允许选择一个角色卡

## 五、会话管理子系统

### 5.1 设计目标

会话管理要同时满足两个诉求：

- 用户把每段剧情当作“存档”管理
- 用户在需要时可以复制当前对话，形成新的独立剧情副本

### 5.2 会话层级建议

建议游玩模式采用两层结构：

- `角色卡`
- `剧情会话`

即：

- 一个角色卡下可以有多个独立剧情会话
- 每个剧情会话内部保持单线性消息历史
- 如果用户想尝试其他走向，应复制当前对话而不是创建分支树

### 5.3 会话状态

每个 Play Session 建议包含以下状态：

- `active`：当前可继续对话的主状态
- `archived`：已归档但可查看
- `deleted`：逻辑删除

### 5.4 会话操作

支持：

- 新建会话
- 复制当前会话
- 从任意历史节点复制会话
- 重命名会话
- 删除会话
- 切换会话
- 导出会话

### 5.5 与其他模块的边界

会话管理子系统负责剧情线组织，不负责：

- Prompt 组装
- 世界书触发
- 角色卡配置修改

## 六、消息发送与生成子系统

### 6.1 核心目标

该子系统负责把“用户输入”转成一轮完整的模型请求，并把模型回复转成：

- 对用户可见的内容
- 对开发者可追踪的 Prompt 详情
- 对系统可消费的结构化状态更新

### 6.2 输入来源

本轮可进入 Prompt 的输入包括：

- 用户原始输入
- 场景描述输入
- Quick Reply 内容
- Slash Command 展开结果
- 历史消息窗口
- 角色卡基础设定
- 角色卡 few-shot 示例
- 角色卡 System Prompt Override
- Post History Instructions
- 世界书触发条目
- User Persona
- 记忆注入
- Author's Note
- 当前状态变量摘要

### 6.3 消息组装原则

游玩模式下，Prompt 组装必须遵守：

- 角色一致性优先
- 世界规则优先于临时剧情自由发挥
- 用户当前输入优先于过远历史
- 被锁定消息优先保留
- 记忆是增强，不应覆盖明确剧情事实

### 6.4 建议的组装顺序

本轮发送给模型前，建议按以下顺序构建上下文：

1. 全局系统预设 / 模式预设
2. 角色卡系统设定
3. 世界书常驻设定
4. 本轮触发的世界书条目
5. User Persona
6. 长期记忆
7. 中期摘要
8. Author's Note
9. 状态变量摘要
10. 历史消息窗口
11. 本轮用户输入
12. Post History Instructions

说明：

- 这里是游玩模块视角下的建议顺序，最终会在后续《Prompt管线与检查器设计》中统一收敛。
- 本文先固定游玩模块对该顺序的需求。

### 6.5 模型返回处理

模型返回后，应拆成三层内容：

- `raw_response`：模型原始返回，未清洗
- `cleaned_response`：去除 `state_update`、Regex 清洗后的文本
- `display_response`：最终渲染展示给用户的文本

这三层都建议保留。

### 6.6 流式输出

游玩模式支持流式响应。

流式期间建议：

- 前端实时显示当前文本片段
- 完整 Prompt Trace 在该轮结束后落盘
- `state_update` 只在整轮结束后统一解析和更新

这样能避免中间状态抖动和多次重渲染。

## 七、世界书触发子系统

### 7.1 设计目标

游玩模式中的世界书不应要求用户手动挑选条目，而应按上下文自动触发，帮助模型维持剧情一致性。

### 7.2 触发来源

建议触发扫描范围包括：

- 用户当前输入
- 最近 N 条历史消息
- 当前场景描述
- 当前激活状态变量

### 7.3 触发结果

每一轮世界书触发后，系统应生成一份结构化结果：

- 触发条目 ID
- 条目标题
- 条目分类
- 触发关键词
- 触发来源位置
- 是否为强制注入

### 7.4 注入原则

- 常驻世界规则优先级高于普通剧情细节条目
- 角色类条目在被明确提及时优先注入
- 已明显冲突或重复的条目需要去重
- 同轮注入内容需要有 token 上限

### 7.5 对用户的可见性

游玩模式应至少提供两层可见性：

- 简版：上下文面板中显示“本轮注入了哪些世界书条目”
- 详版：Prompt Inspector 中显示条目全文、触发关键词和注入顺序

## 八、消息操作子系统

### 8.1 重新生成与 Swipe

#### 目标

允许用户对最后一条模型回复要求“再生成一个版本”，但不覆盖旧版本。

#### 设计原则

- 同一消息位置可以挂多个回复版本
- 默认只展示当前选中的一个版本
- 用户可以左右切换查看不同 Swipe

#### 建议行为

- 第一次生成时创建 `swipe_index = 0`
- 每次重新生成时新增一条 Swipe
- 当前选中的 Swipe 单独记录
- 删除 Swipe 时至少保留一条

### 8.2 编辑历史消息

编辑历史消息后继续对话，建议采用：

- 从该消息位置截断后续剧情线
- 保留被截断前的完整历史快照
- 如用户希望保留原线，优先提示“先复制当前对话再编辑”

这样既保留线性结构，又允许任意长度回溯。

### 8.3 撤回

撤回与回溯应分开定义。

撤回建议只作为快捷操作，默认只作用于“最后一轮”：

- 最后一条用户消息
- 对应的最后一条模型消息

如果存在锁定消息，则不可跨越锁定节点撤回。

### 8.4 任意长度回溯

游玩模式需要支持任意长度回溯。

这里的“回溯”指：

- 用户可以回到当前会话中的任意历史消息节点
- 从该节点继续后续对话
- 该节点之后的旧消息将不再属于当前活动时间线

建议行为：

- 回溯前弹出确认
- 默认展示即将被截断的消息数量
- 回溯后保留被截断历史的内部快照，用于审计和恢复
- 如果用户不想影响当前线，应使用“复制当前对话”功能

### 8.5 锁定消息

被锁定消息的作用：

- 不允许被撤回覆盖
- 不允许被回溯跨越
- 在上下文裁剪时提高保留优先级
- 在 Prompt Inspector 中显示“已锁定”标记

## 九、对话复制子系统

### 9.1 设计目标

用户应能在不引入复杂分支树的前提下，保留当前进度并尝试新的剧情走向。

### 9.2 为什么不采用 Branch

当前版本不将 Branch 作为核心能力，原因是：

- 用户的真实使用习惯以单线连续游玩为主
- 分支树会显著增加数据模型和交互复杂度
- 分支嵌套、切换、命名、状态继承都会抬高第一阶段成本
- “复制当前对话”已经可以覆盖大多数尝试不同路线的需求

因此，当前设计采取：

- 当前会话保持线性
- 不维护分支树
- 需要另一路线时，复制为新对话

### 9.3 对话复制的两种方式

#### 方式 A：创建全新对话

- 基于当前角色卡创建一个空白新对话
- 默认只保留开场配置和角色卡绑定
- 不继承当前剧情消息

#### 方式 B：复制当前对话

- 把当前对话复制成一个新的独立对话
- 可选择复制到“当前节点”为止，或复制完整对话
- 复制后新旧对话互不影响

### 9.4 从任意历史节点复制

为了替代 Branch，系统应支持从任意历史节点复制当前对话。

建议行为：

- 用户在任意消息节点点击“从这里复制对话”
- 新对话继承该节点之前的全部上下文
- 新对话从该节点开始继续线性发展
- 原对话保持不变

### 9.5 从 Swipe 复制对话

这是替代“从 Swipe 创建 Branch”的轻量方案。

原因：

- 同一条回复的不同 Swipe 可能导向完全不同的剧情
- 用户可能希望保留多个版本并分别继续

建议行为：

- 从指定 Swipe 复制对话时，新对话将该 Swipe 视为该节点的已选版本
- 复制后的新对话从该版本继续发展
- 原对话仍保留其当前选中的 Swipe 状态

### 9.6 命名规则

建议默认命名：

- 新建空对话：`角色名 - 新对话 #N`
- 复制对话：`原对话名 - 副本 #N`
- 从历史节点复制：`原对话名 - 节点副本 #N`

## 十、Prompt Inspector 子系统

### 10.1 设计目标

Prompt Inspector 是游玩模式的关键调试底座。

它必须回答：

- 这一轮用户到底发了什么
- 系统实际往模型里塞了什么
- 模型原始回了什么
- 中间发生了哪些自动注入、裁剪、清洗和工具调用

### 10.2 触发入口

每条“用户发送的消息”旁都应提供“详情”入口。

进入后展示该轮完整详情。

### 10.3 展示内容

建议至少包含：

- 用户原始输入
- 场景描述补充
- 本轮激活的模式预设
- 本轮注入的角色卡设定
- 本轮注入的世界书条目
- Author's Note
- User Persona
- 长期记忆与中期摘要
- 状态变量摘要
- 完整 `messages` 数组
- token 统计
- 模型原始响应
- Regex 清洗前后的对比
- `state_update` 原始块与解析结果
- 本轮工具调用记录

### 10.4 数据保存策略

游玩模式下建议把 Prompt Inspector 所需内容按轮保存为 `PromptTrace Snapshot`，而不是完全实时回算。

原因：

- 便于复盘历史轮次
- 避免世界书、角色卡更新后历史结果失真
- 便于导出与调试

### 10.5 开发者模式行为

当开发者模式开启时：

- 允许显示 `state_update` 原始块
- 允许查看 Regex 清洗前的原始模型输出
- 允许显示工具调用原始结果摘要

当开发者模式关闭时：

- 用户侧默认看不到原始控制块
- 但 Prompt Trace 仍保留在系统内部

## 十一、状态展示子系统

### 11.1 设计目标

游玩模式需要通过动态 UI 组件展示剧情状态，增强沉浸感，并让状态变化不只存在于模型文本里。

### 11.2 状态来源

状态主要来自：

- 角色卡默认状态配置
- 世界书绑定的 UI Schema
- 模型输出中的 `<state_update>` 块
- 系统初始化默认值

### 11.3 状态更新流程

建议流程如下：

1. 模型返回原始文本
2. Regex 管线提取 `<state_update>`
3. 状态引擎解析为键值对
4. 更新当前 Session 的状态变量
5. 持久化更新记录
6. 通过实时通道推送前端
7. 前端刷新状态栏 / 悬浮球 / 侧边面板

### 11.4 用户可见性

普通模式下：

- 状态变化体现在 UI 中
- 不显示控制指令文本

开发者模式下：

- 可在 Prompt Inspector 中查看原始指令
- 可查看本轮状态变化前后对比

### 11.5 回溯与复制时的状态行为

这是游玩模式必须明确的一点。

建议规则：

- 每个对话拥有自己的状态快照链
- 从历史节点复制对话时，新对话继承该节点对应的状态快照
- 当前对话执行回溯时，状态回退到目标节点对应的状态快照

## 十二、Quick Reply 子系统

### 12.1 设计目标

Quick Reply 用于减少重复输入，让用户更像在操控剧情而不是不断手打命令。

### 12.2 数据来源

Quick Reply 由两部分组成：

- 全局快捷回复
- 当前角色卡专属快捷回复

### 12.3 建议能力

- 点击即填入输入框
- 点击即发送
- 支持宏变量替换
- 支持 slash command
- 支持开局自动触发

### 12.4 与游玩模式的关系

Quick Reply 在游玩模式中更偏“剧情操作按钮”，因此其内容建议更偏：

- 场景推进
- 角色态度表达
- 行动选择
- 常用追问

## 十三、导出与复盘子系统

### 13.1 导出目标

用户需要把剧情结果导出为可阅读、可存档、可复盘的文本材料。

### 13.2 导出格式

第一阶段建议支持：

- TXT

### 13.3 导出内容层级

建议分两种导出：

- `用户阅读版`：只导出最终展示消息
- `开发调试版`：附带 Prompt Trace 摘要、状态变化、世界书触发记录

## 十四、异常与边界处理

### 14.1 角色卡缺少世界书

允许进入游玩模式，但系统应：

- 使用角色卡自身设定继续对话
- 关闭世界书触发相关能力
- 状态栏按角色卡默认配置或空状态运行

### 14.2 模型未输出合法 `state_update`

系统应：

- 正常展示文本回复
- 本轮状态保持不变
- 在开发者模式下记录解析失败信息

### 14.3 世界书触发过多

系统应：

- 按优先级裁剪
- 对重复条目去重
- 在 Prompt Inspector 中显示被裁剪信息

### 14.4 工具调用失败

系统应：

- 不阻塞本轮主对话输出
- 记录失败原因
- 在 Prompt Inspector 中标记“工具调用失败”

### 14.5 历史编辑导致结构冲突

系统应优先：

- 提示用户选择“直接回溯后编辑”或“先复制当前对话再编辑”
- 默认不自动创建复杂分支结构

### 15.1 依赖的共享能力

游玩模块依赖：

- 角色卡发布系统
- 世界书系统
- Prompt 管线
- 记忆系统
- 模型调度层
- Regex 管线
- 状态引擎
- 工具调用层

### 15.2 向其他模块输出的能力

游玩模块将为系统级设计提供以下需求约束：

- 需要统一 Session 抽象
- 需要统一 Message / Swipe / 回溯快照抽象
- 需要统一 PromptTrace 抽象
- 需要统一 StateStore 抽象
- 需要统一工具调用记录结构

### 15.3 对系统架构设计的反向要求

未来在系统架构设计中，必须满足游玩模块以下要求：

- 支持流式消息
- 支持按轮保存 Prompt Trace
- 支持对话复制
- 支持任意长度回溯
- 支持状态快照继承
- 支持后端执行 Regex 清洗

## 十八、结论

游玩模块是 Izumi Studio 第一阶段最核心的业务模块。

它不是单纯的聊天界面，而是一个由以下能力共同组成的剧情运行环境：

- 角色卡消费
- 世界书触发
- Prompt 可观察性
- 状态驱动 UI
- 线性会话、对话复制与剧情回溯

只要游玩模块设计清晰，后续聊天模式、Prompt 管线、状态系统和核心数据模型都会更容易收敛。
