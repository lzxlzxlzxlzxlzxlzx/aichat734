## 一、功能定位

### 1.1 解决的核心问题

在传统聊天产品里，剧情状态通常只存在于模型生成的文本中。

这会带来几个问题：

- 状态变化不稳定，模型容易遗忘
- 用户无法直观看到关键变量
- 很难在长剧情中维持连续性
- 回溯、复制、摘要后容易丢失状态

Izumi Studio 的状态系统要把“剧情状态”从纯文本中抽离出来，变成可持久化、可显示、可继承、可编辑的结构化数据。

### 1.2 模块目标

状态系统与面板模块的目标是：

- 提供统一的 `state_update` 协议
- 在会话维度保存状态变量
- 根据世界书 UI Schema 渲染状态栏、悬浮球、侧边面板
- 支持状态历史记录与状态快照
- 支持回溯与复制对话时的状态继承
- 让 Prompt Inspector 可查看原始状态更新与解析结果

### 1.3 模块边界

本模块负责：

- 状态变量结构
- `state_update` 提取与解析
- 状态持久化与历史记录
- UI 组件渲染绑定
- 状态快照生成与继承

本模块不负责：

- 世界书 UI Schema 的生成
- Prompt 组装逻辑
- 记忆生成逻辑
- 角色卡与世界书编辑界面

## 二、总体结构

### 2.1 状态生命周期

```text
角色卡 / 世界书载入
  ↓
状态 Schema 初始化
  ↓
Session StateStore 建立
  ↓
模型回复中输出 state_update
  ↓
后端解析状态更新
  ↓
变量写入 StateStore
  ↓
生成状态历史记录 / 快照
  ↓
推送前端刷新状态面板
  ↓
Prompt Inspector 可查看原始块与解析结果
```

### 2.2 模块内部子系统

建议拆为以下子系统：

- `State Schema 子系统`
- `StateStore 子系统`
- `state_update 解析子系统`
- `状态历史记录子系统`
- `状态快照子系统`
- `UI 组件绑定子系统`
- `手动编辑与校正子系统`

### 2.3 适用范围

当前状态系统主要服务于：

- 游玩模式

未来可选扩展到：

- 聊天模式中的轻量状态变量
- 创作模式中的预览状态模拟

## 三、状态 Schema 子系统

### 3.1 设计目标

状态系统不能只存“任意键值对”，还需要有一个约束层来定义：

- 有哪些变量
- 每个变量是什么类型
- 默认值是什么
- 是否有取值范围
- UI 如何展示

### 3.2 Schema 来源

状态 Schema 主要来自世界书中的 UI Components Schema。

如果当前角色卡未绑定世界书，则可使用：

- 角色卡默认状态配置
- 或空白状态 Schema

### 3.3 Schema 结构建议

```json
{
  "state_schema": {
    "hp": { "type": "number", "default": 100, "min": 0, "max": 100 },
    "location": { "type": "string", "default": "未知" },
    "affection": { "type": "number", "default": 0, "min": 0, "max": 5 },
    "mood": { "type": "string", "default": "平静" }
  }
}
```

### 3.4 支持的类型

建议第一阶段支持：

- `number`
- `string`
- `boolean`
- `enum`
- `object_map`

后续可扩展：

- `array`
- `progress_delta`
- `timeline_marker`

---

## 四、StateStore 子系统

### 4.1 设计目标

StateStore 是会话级状态变量容器。

它需要满足：

- 当前会话的状态可读取
- 每轮状态变化可记录
- 回溯时可恢复
- 复制对话时可继承

### 4.2 存储粒度

建议状态变量按 `session_id` 维度存储，而不是按角色卡全局共享。

原因：

- 同一个角色卡下可以有多个独立剧情会话
- 每个会话状态可能完全不同
- 对话复制后新会话也需要独立状态线

### 4.3 基本能力

StateStore 建议提供：

- `get(key)`
- `set(key, value)`
- `bulk_update(updates)`
- `snapshot()`
- `restore(snapshot)`
- `render_template(template)`

### 4.4 初始化流程

进入游玩模式并创建新会话时：

1. 读取角色卡绑定的世界书
2. 读取其 UI Schema 和状态默认值
3. 生成初始 StateStore
4. 将初始状态写入会话记录

## 五、state_update 解析子系统

### 5.1 设计目标

模型不能直接操作数据库状态，所以需要一个中间协议，让模型以结构化文本形式声明状态变化。

### 5.2 协议格式

建议统一采用：

```text
<state_update>
hp: 85
location: 蒙德城·西风骑士团
affection: +1
mood: 紧张
</state_update>
```

### 5.3 支持的值类型

第一阶段建议支持：

- 直接字符串
- 直接数字
- `+1` / `-2` 这类增量值
- 键值对对象列表，如 `艾伦=存活, 米卡莎=存活`

### 5.4 解析规则

建议解析流程：

1. 匹配 `<state_update>` 块
2. 按行切分
3. 解析 `key: value`
4. 如果是增量值，按旧值计算新值
5. 按 Schema 校验值类型和范围
6. 输出解析成功项与失败项

### 5.5 解析结果结构

```json
{
  "raw_block": "<state_update>...</state_update>",
  "parsed_updates": {
    "hp": 85,
    "location": "蒙德城·西风骑士团"
  },
  "delta_updates": {
    "affection": 1
  },
  "validation_errors": []
}
```

### 5.6 解析失败处理

如果部分字段解析失败：

- 有效字段仍正常写入
- 失败字段被记录但不阻塞整轮回复
- Prompt Inspector 中可查看错误明细

## 六、状态历史记录子系统

### 6.1 设计目标

用户不仅需要知道“现在是什么状态”，还需要知道“状态是怎么变成现在这样的”。

### 6.2 历史记录粒度

建议每轮模型回复结束后记录一条状态历史。

每条历史建议包含：

- 轮次对应的消息 ID
- 变化前值
- 变化后值
- 原始 `state_update` 块
- 变更时间

### 6.3 示例结构

```json
{
  "session_id": "uuid",
  "message_id": "uuid",
  "changes": [
    { "key": "hp", "old": 100, "new": 85 },
    { "key": "mood", "old": "平静", "new": "紧张" }
  ],
  "raw_block": "<state_update>...</state_update>",
  "created_at": "timestamp"
}
```

### 6.4 展示用途

状态历史可用于：

- Prompt Inspector 中查看本轮变化
- 后续状态时间线展示
- 回溯和恢复
- 记忆摘要中的状态快照引用

---

## 七、状态快照子系统

### 7.1 设计目标

只记录增量还不够，回溯和复制对话时需要快速恢复某一时刻的完整状态。

因此需要状态快照。

### 7.2 快照生成时机

建议在以下时机生成快照：

- 新会话创建时
- 每轮模型回复结束后
- 用户手动修改状态后
- 回溯确认后
- 从历史节点复制对话时

### 7.3 快照结构

```json
{
  "snapshot_id": "uuid",
  "session_id": "uuid",
  "message_id": "关联消息ID，可为空",
  "variables": {
    "hp": 85,
    "location": "蒙德城·西风骑士团",
    "affection": 3
  },
  "created_at": "timestamp"
}
```

### 7.4 与游玩模式回溯的关系

游玩模式支持任意长度回溯，因此：

- 回溯到某历史节点时，应恢复到该节点对应的状态快照
- 该节点之后的状态历史不再属于当前活动时间线
- 被截断历史仍保留内部快照，便于审计

### 7.5 与对话复制的关系

从当前节点复制对话时：

- 新对话应继承该节点对应的状态快照
- 新旧对话之后的状态变化彼此独立

## 八、UI 组件绑定子系统

### 8.1 设计目标

状态系统的价值不仅在于存数据，更在于把数据稳定映射为可视化 UI。

### 8.2 支持的组件

第一阶段建议支持：

- `状态栏（Status Bar）`
- `悬浮球（Floating Ball）`
- `侧边信息栏（Side Panel）`

### 8.3 绑定方式

每个 UI 组件字段通过 `key` 绑定到状态变量：

```json
{
  "fields": [
    { "key": "hp", "label": "HP", "type": "progress" },
    { "key": "location", "label": "当前位置", "type": "text" }
  ]
}
```

### 8.4 渲染策略

渲染时：

1. 读取当前会话的 StateStore
2. 读取 UI Schema
3. 将字段 key 映射到当前值
4. 输出前端组件所需的数据结构

### 8.5 缺失字段处理

如果某字段在 StateStore 中缺失：

- 优先使用 Schema 默认值
- 若无默认值则显示空态
- 不应导致整个组件渲染失败

---

## 九、手动编辑与校正子系统

### 9.1 设计目标

模型更新状态并不一定总是正确，用户需要拥有最终校正权。

### 9.2 支持能力

建议支持：

- 手动修改当前状态值
- 重置某个变量为默认值
- 查看某字段最近变更历史

### 9.3 写入原则

手动修改状态时：

- 生成一条状态历史记录
- 生成新的状态快照
- 标记本次修改来源为 `manual`

### 9.4 用户可见性

普通用户侧默认看到的是当前值和简单编辑入口。

开发者模式下可额外看到：

- 该值最近由哪条消息改变
- 最近一次自动解析来源

---

## 十、与 Prompt Inspector 的关系

### 10.1 Inspector 中应展示的内容

Prompt Inspector 需要展示：

- 本轮原始 `state_update` 块
- 解析成功的字段
- 解析失败的字段
- 本轮状态变化前后对比

### 10.2 开发者模式

开发者模式开启时可展示：

- 原始控制块全文
- Schema 校验结果
- 增量值换算结果

### 10.3 普通模式

普通模式下：

- 用户只看到 UI 变化
- 不直接看到控制指令文本

---

## 十一、异常与边界处理

### 11.1 模型未输出状态更新

系统应：

- 正常显示本轮文本回复
- 保持当前状态不变
- 记录“本轮无状态更新”

### 11.2 输出了未定义字段

系统应：

- 默认忽略
- 在开发者模式中记录为未识别字段
- 不自动写入 StateStore

### 11.3 数值越界

例如：

- `hp: 999`
- `affection: -10`

系统应：

- 按 Schema 范围裁剪或拒绝写入
- 将处理结果写入日志和 Prompt Inspector

### 11.4 UI Schema 与状态不匹配

系统应：

- 对单字段降级
- 组件局部显示空态
- 不影响其他字段渲染

## 十二、与其他模块的关系

### 12.1 对游玩模块的支撑

状态系统为游玩模块提供：

- 会话级状态存储
- 状态栏与悬浮球渲染数据
- 回溯与复制时的状态继承
- Prompt Inspector 中的状态更新详情

### 12.2 对世界书生成模块的支撑

状态系统为世界书生成模块提供：

- 可生成的状态 Schema 目标结构
- 面板模板能力
- `state_update` 协议约束

### 12.3 对系统架构设计的反向要求

未来在系统架构设计中，必须满足：

- 会话维度的状态持久化
- 快照与历史记录并存
- 状态更新与 UI 推送解耦
- Prompt 后处理可调用状态解析器

## 十五、结论

状态系统与面板模块是游玩模式沉浸感和连续性的关键支撑。

它的价值在于把“剧情中的状态”从一段容易丢失的文字，变成：

- 可结构化存储的变量
- 可追踪的历史记录
- 可回溯和可继承的快照
- 可直观展示的 UI 组件

只要状态系统先设计清楚，后续游玩模式的回溯、复制、记忆摘要和系统架构设计都会更容易稳定下来。
