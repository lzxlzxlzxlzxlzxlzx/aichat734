## 一、功能定位

### 1.1 解决的核心问题

当前 SillyTavern 的世界书是一个纯手工编辑系统：用户需要手写每一个 WI 条目的 JSON 字段、HTML 模板、变量语法，门槛极高。

**Izumi Studio 的目标**：用户用自然语言描述需求，系统自动生成完整、可直接使用的世界书结构，包括状态栏 UI、动态变量和章节/角色条目。

### 1.2 典型用户故事

```
用户：
  "帮我给《进击的巨人》创建一个世界书。需要一个状态栏，
   显示当前年代（850年）、所在墙区（玛利亚之墙/罗塞之墙/希纳之墙）、
   角色存活状态和巨人警戒等级（无/低/高/危急）。
   主要角色有艾伦、米卡莎、阿尔敏。"

系统交付：
  ✓ 完整 WI JSON（可直接导入）
  ✓ 状态栏条目（带 HTML 模板和变量引用）
  ✓ 变量更新指令条目（state_update 规则）
  ✓ 3 个角色条目（关键词触发）
  ✓ 世界基础设定条目（constant）
  ✓ 变量初始化条目（chapter/year/wall_zone/alert_level）
```

## 三、State Engine（变量引擎）

### 3.1 设计目标

替代 SillyTavern 中 MVU-BETA（外部 JS CDN 依赖）的功能，内置到 Izumi Studio 后端，无需外部依赖。

### 3.2 变量 Store

```python
# backend/state_engine.py

from dataclasses import dataclass, field
from typing import Any

@dataclass
class StateStore:
    """Session 级别的变量存储，随对话持久化"""
    session_id: str
    variables: dict[str, Any] = field(default_factory=dict)
    history: list[dict] = field(default_factory=list)  # 每轮变更记录

    def get(self, key: str, default=None) -> Any:
        return self.variables.get(key, default)

    def set(self, key: str, value: Any) -> None:
        old = self.variables.get(key)
        self.variables[key] = value
        self.history.append({"key": key, "old": old, "new": value})

    def bulk_update(self, updates: dict[str, Any]) -> None:
        for k, v in updates.items():
            self.set(k, v)

    def render_template(self, template: str) -> str:
        """展开模板中的 {variable_name} 占位符"""
        result = template
        for key, value in self.variables.items():
            result = result.replace(f"{{{key}}}", str(value) if value is not None else "")
        return result
```

### 3.3 state_update 解析协议

模型在每条回复末尾输出标准化更新块（由 Regex 脚本从消息中提取，不显示给用户）：

```
<state_update>
year: 850
wall_zone: 玛利亚之墙·希加尼斯区
alert_level: 危急
alive: 艾伦=存活, 米卡莎=存活, 阿尔敏=存活
time: 巨人突破后第3小时
</state_update>
```

**解析规则**：
- 每行 `key: value` 格式
- 支持简单值（字符串/数字）和键值对列表（`key=value, key=value`）
- 解析失败的行静默跳过，不报错
- 解析结果通过 WebSocket 推送到前端，面板实时刷新

### 3.4 Regex 提取管道（后端执行，非 ST 的前端 JS）

```python
# backend/regex_pipeline.py
import re

STATE_UPDATE_PATTERN = re.compile(
    r'<state_update>\s*(.*?)\s*</state_update>',
    re.DOTALL | re.IGNORECASE
)

def extract_state_update(message: str) -> tuple[str, dict]:
    """
    从模型回复中提取 state_update 块
    返回：(清理后的消息文本, 解析的变量字典)
    """
    updates = {}
    cleaned = message

    match = STATE_UPDATE_PATTERN.search(message)
    if match:
        block = match.group(1)
        for line in block.strip().splitlines():
            line = line.strip()
            if ':' not in line:
                continue
            key, _, raw_value = line.partition(':')
            key = key.strip()
            value = _parse_value(raw_value.strip())
            updates[key] = value
        # 从消息中移除该块（不展示给用户）
        cleaned = STATE_UPDATE_PATTERN.sub('', message).strip()

    return cleaned, updates

def _parse_value(raw: str) -> Any:
    """解析值：数字/布尔/键值对列表/普通字符串"""
    if raw.isdigit():
        return int(raw)
    if ',' in raw and '=' in raw:
        result = {}
        for item in raw.split(','):
            if '=' in item:
                k, _, v = item.partition('=')
                result[k.strip()] = v.strip()
        return result
    return raw
```

## 五、LLM 生成层（WorldBuilder Agent）

### 5.1 Agent 的任务

接收用户自然语言描述，输出标准 WI JSON 草稿，包含：
1. 世界基础设定条目（constant，`before_char`）
2. 变量初始化条目（constant，initvar）
3. 状态栏指令条目（constant，`after_char`，order=1）
4. state_update 格式规则条目（constant，`after_char`，order=2）
5. 角色条目（关键词触发，`before_char`）
6. 章节/事件条目（关键词触发，`after_char`，按需）

### 5.2 System Prompt 设计

```
你是 Izumi Studio 的世界书生成专家。根据用户描述，生成标准格式的 WI 条目 JSON 数组。

## 输出格式

必须输出合法 JSON，结构如下：
{
  "worldbook_name": "世界书名称",
  "panel_template_id": "选用的面板模板ID",
  "variables": { "变量名": 初始值, ... },
  "entries": [ ...WI条目数组... ]
}

## WI 条目字段规范

每个条目包含：
- comment: 条目备注（显示在编辑器中，不注入）
- content: 注入内容（支持 {变量名} 占位符）
- constant: true=永久激活，false=关键词触发
- keys: 触发关键词数组（constant=false 时必填）
- position: "before_char"（角色定义前）或 "after_char"（历史消息后）
- insertion_order: 注入排序（数字越小越优先）
- enabled: true

## 变量占位符语法

在 content 中使用 {变量名} 引用 State Engine 变量，例如：
"当前章节：{chapter}，年代：{year}"

## 状态栏条目规范

状态栏条目是告诉 LLM"必须输出 state_update 块"的指令，不是展示给前端的 HTML。
前端 Panel 由系统根据 State Engine 变量自动渲染，无需在 WI 中写 HTML。

状态栏条目 content 格式示例：
---
<状态栏指令>
在每条正文回复的最末尾，必须输出以下格式的变量更新块：

<state_update>
year: [当前年代，数字]
wall_zone: [所在墙区名称]
alert_level: [无/低/高/危急]
alive: 艾伦=[存活/阵亡], 米卡莎=[存活/阵亡], 阿尔敏=[存活/阵亡]
time: [当前时间描述]
</state_update>

规则：
- 必须在正文之后、任何思考链之前输出
- 所有字段都要填写，未知填"未知"
- 不要输出 <state_update> 以外的其他特殊标签
</状态栏指令>
```

```typescript
// stores/stateEngine.ts (Pinia)
export const useStateEngineStore = defineStore('stateEngine', {
  state: () => ({
    variables: {} as Record<string, any>,
    lastUpdated: null as Date | null,
    templateId: 'minimal_status' as string,
  }),

  getters: {
    get: (state) => (key: string, fallback = '') =>
      state.variables[key] ?? fallback,

    nearby: (state) =>
      state.variables['nearby'] as Record<string, string> | null,
  },

  actions: {
    applyUpdate(updates: Record<string, any>) {
      this.variables = { ...this.variables, ...updates }
      this.lastUpdated = new Date()
    },

    setTemplate(templateId: string) {
      this.templateId = templateId
    }
  }
})
```

### 7.2 变量持久化策略

- 每轮对话结束后，将当前 `StateStore` 快照写入 `session_state_snapshots`
- 恢复对话时，加载最新快照到内存
- 支持 Checkpoint/Branch 时，快照跟随分支独立存储

## 八、安全设计

### 8.1 LLM 生成内容的安全约束

| 风险 | 处理 |
|------|------|
| 生成包含 `<script>` 的 HTML | 后端 bleach/defusedxml 白名单过滤 |
| 生成恶意 CSS（url() 外链） | CSS 解析器过滤外部 URL |
| WI 条目内容注入 prompt | 内容经 token 统计，超限截断 |
| 变量名冲突（用户定义覆盖系统变量） | 系统变量加 `_iz_` 前缀保护 |
