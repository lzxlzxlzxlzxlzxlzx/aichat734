## 一、功能定位

### 1.1 解决的核心问题

大模型对长对话的主要问题不是“完全没有记忆”，而是：

- 上下文窗口有限
- 旧消息必须被裁剪
- 关键设定容易在多轮后漂移
- 用户无法知道系统究竟记住了什么

Izumi Studio 的记忆系统要解决的不是单纯“存档”，而是：

- 在上下文受限时保持角色、世界观、事件和用户偏好的连续性
- 让记忆成为可检查、可编辑、可追溯的结构化系统
- 让游玩、聊天、创作三种模式共享同一套记忆底座

### 1.2 模块目标

记忆系统的目标是：

- 提供短期 / 中期 / 长期三层记忆结构
- 保证原始消息永久保存，记忆只是压缩与提炼副本
- 支持自动摘要与手动记忆标记
- 为 Prompt 管线提供稳定的记忆注入结果
- 为用户提供记忆中心，查看和修正系统记忆

### 1.3 模块边界

本模块负责：

- 三层记忆结构
- 摘要生成策略
- 长期记忆提取与管理
- 记忆注入结果生成
- 记忆管理界面数据基础

本模块不负责：

- Prompt 组装顺序最终定义
- 世界书触发算法
- 状态变量解析
- 角色卡和世界书编辑

## 二、总体结构

### 2.1 三层记忆架构

```text
原始消息永久保存
   ↓
短期记忆：最近 N 轮原始消息
   ↓
中期记忆：阶段摘要
   ↓
长期记忆：关键事实锚点
```

### 2.2 核心原则

三层记忆必须同时满足以下原则：

- 原始消息永不删除
- 摘要不能替代原始记录
- 长期记忆必须可编辑
- 记忆注入必须可追踪
- 三种模式共享同一套机制

### 2.3 模块内部子系统

建议拆为以下子系统：

- `短期记忆子系统`
- `中期记忆摘要子系统`
- `长期记忆子系统`
- `记忆注入子系统`
- `记忆中心子系统`
- `记忆触发与调度子系统`

## 三、作用域设计

### 3.1 适用范围

记忆系统适用于：

- 创作模式
- 游玩模式
- 聊天模式

这是当前 PRD 已确认的统一设计方向。

### 3.2 记忆作用域层级

建议记忆系统采用多层作用域，而不是只按单一维度绑定。

第一阶段可先按以下思路设计：

- `session_scope`：会话级记忆
- `card_scope`：角色卡级长期记忆
- `global_scope`：全局用户偏好与长期事实

### 3.3 第一阶段建议实现范围

为了控制复杂度，第一阶段建议优先稳定支持：

- 会话级短期记忆
- 会话级中期摘要
- 角色卡级长期记忆
- 聊天模式会话级长期记忆

全局用户长期偏好可先保留结构预留，后续再强化。

### 3.4 与模式的关系

#### 游玩模式

- 更关注剧情、状态、角色关系、世界事件
- 中期摘要通常带有状态快照
- 长期记忆偏向不可逆事件、设定锚点

#### 聊天模式

- 更关注用户偏好、讨论上下文、长期关系感
- 长期记忆可包括用户习惯、项目偏好、常提主题

#### 创作模式

- 更关注项目目标、设定选择、反复确认过的创作方向

## 四、短期记忆子系统

### 4.1 定义

短期记忆就是最近 N 轮原始消息窗口。

它不是额外生成的内容，而是直接从原始消息历史中取出的一段连续消息。

### 4.2 设计目标

短期记忆负责维持当前对话的即时连续性。

### 4.3 默认策略

建议默认保留：

- 最近 20 轮原始消息

后续允许用户配置，范围与 PRD 保持一致。

### 4.4 裁剪原则

短期记忆裁剪时应优先考虑：

- 锁定消息优先保留
- 当前讨论主题相关消息优先保留
- 已被中期摘要覆盖的远历史消息可移出活跃窗口

### 4.5 与其他层的关系

短期记忆是当前上下文中的“原始现场感”，不能完全被摘要替代。

---

## 五、中期记忆摘要子系统

### 5.1 定义

中期记忆是对一段历史消息的结构化摘要，用来在消息被移出活跃窗口后，仍保留关键上下文。

### 5.2 设计目标

中期摘要要解决的问题是：

- 历史太长，不能一直保留原文
- 但又不能让剧情、讨论、创作决策直接消失

### 5.3 触发机制

摘要触发支持以下方式：

- 按轮次触发
- 按 token 预算触发
- 用户手动触发

建议第一阶段保留三种机制，但默认：

- 按轮次触发
- 后台异步生成

### 5.4 触发参数

建议参数包括：

- `summary_round_threshold`
- `summary_token_threshold`
- `summary_max_messages`
- `summary_model`
- `summary_template`

### 5.5 摘要生成流程

```text
检测触发条件
  ↓
选择待摘要消息段
  ↓
调用摘要模型
  ↓
生成结构化摘要
  ↓
保存摘要记录
  ↓
将原始消息移出活跃上下文窗口
  ↓
下轮对话按规则注入摘要
```

### 5.6 摘要结构建议

```json
{
  "id": "uuid",
  "session_id": "uuid",
  "segment_start": 41,
  "segment_end": 60,
  "summary": "本阶段摘要正文",
  "key_events": [],
  "state_snapshot": {},
  "frozen": false,
  "created_at": "timestamp"
}
```

### 5.7 摘要内容要求

摘要建议至少包含：

- 当前阶段发生了什么
- 关键事件
- 重要关系变化
- 关键设定变化
- 当前状态快照（游玩模式）

### 5.8 摘要冻结

用户应能冻结某条摘要。

冻结后：

- 后续自动摘要不覆盖它
- 仍可手动编辑

### 5.9 摘要模板

应支持可配置模板，例如：

```text
[剧情摘要 - 第41至第60轮]
{summary}

关键事件：{key_events}
当前状态快照：{state_snapshot}
[摘要结束]
```

### 5.10 多条摘要注入策略

中期摘要注入时建议：

- 按时间顺序注入
- 最多注入最近 3 段
- 优先保留最近摘要

## 六、长期记忆子系统

### 6.1 定义

长期记忆是那些不应随着消息裁剪而丢失的“关键事实锚点”。

### 6.2 设计目标

长期记忆负责保证：

- 重要设定不漂移
- 重大剧情结果不遗失
- 用户偏好不会在长对话后消失

### 6.3 来源

长期记忆可以来自：

- 用户手动标记
- AI 自动识别
- 用户后续手动编辑补充

### 6.4 自动识别标准

建议优先识别：

- 不可逆剧情事件
- 角色重大决定
- 世界观关键设定变化
- 长期稳定的用户偏好
- 创作中明确确认的方向性决定

### 6.5 手动标记

用户应支持：

- 在消息上右键标记为长期记忆
- 添加备注
- 指定重要等级

### 6.6 数据结构建议

```json
{
  "id": "uuid",
  "scope_type": "session | card | global",
  "scope_id": "uuid",
  "content": "记忆内容",
  "type": "auto | manual",
  "importance": "high | medium | low",
  "source_message_id": "uuid",
  "created_at": "timestamp"
}
```

### 6.7 长期记忆注入策略

长期记忆建议：

- 固定注入到上下文头部
- 优先于中期摘要
- 尽量保留高重要性条目

### 6.8 去重与冲突

长期记忆系统需要处理：

- 相同事实重复记录
- 新旧事实冲突

建议规则：

- 新记忆进入前做相似度或关键词去重
- 冲突时保留旧记录并标记候选冲突，不自动覆盖
- 用户可在记忆中心手动确认

## 七、记忆注入子系统

### 7.1 设计目标

记忆系统的最终价值体现在 Prompt 注入结果上，而不是只存在数据库里。

### 7.2 注入顺序

结合 Prompt 管线文档，当前建议顺序为：

- 长期记忆先于中期摘要
- 中期摘要先于短期消息窗口

### 7.3 注入结构

记忆注入建议转成结构化注入项，而不是直接拼字符串。

例如：

```json
{
  "source_type": "memory",
  "memory_level": "long_term | mid_term | short_term",
  "label": "长期记忆 / 中期摘要",
  "content": "...",
  "priority": 90,
  "token_estimate": 220
}
```

### 7.4 模式差异

#### 游玩模式

- 长期记忆偏剧情与设定锚点
- 摘要中应尽量保留状态信息

#### 聊天模式

- 长期记忆偏用户偏好、讨论历史、关系感
- 摘要更偏主题演进

#### 创作模式

- 长期记忆偏项目决策、设定确认结果
- 摘要偏草稿修改脉络

### 7.5 裁剪优先级

当 token 紧张时：

- 优先裁剪较旧中期摘要
- 尽量保留高重要性长期记忆
- 短期消息窗口仍要保留最基本连续性

---

## 八、记忆中心子系统

### 8.1 设计目标

用户必须能够看到“系统记住了什么”，否则记忆系统会变成不可控黑箱。

### 8.2 展示结构

记忆中心建议分层展示：

- 短期记忆
- 中期摘要
- 长期记忆

### 8.3 建议支持操作

- 查看
- 编辑
- 删除
- 冻结 / 解冻摘要
- 手动触发摘要
- 手动新增长期记忆

### 8.4 与模式的关系

记忆中心至少应在：

- 游玩模式
- 聊天模式

中提供入口。

创作模式后续也可提供简化版入口。

---

## 九、调度与后台任务子系统

### 9.1 设计目标

摘要生成不应阻塞用户主对话。

### 9.2 建议策略

摘要生成建议：

- 后台异步执行
- 使用单独的摘要模型
- 与主对话模型解耦

### 9.3 摘要模型建议

PRD 已建议：

- `qwen-turbo`
- `deepseek-chat`

作为快速廉价的摘要模型候选。

### 9.4 失败处理

摘要生成失败时：

- 不影响当前对话继续
- 保留原始消息
- 记录失败状态
- 支持用户稍后重试

---

## 十、与 Prompt Inspector 的关系

### 10.1 Inspector 中应展示的内容

Prompt Inspector 应展示：

- 本轮注入了哪些长期记忆
- 本轮注入了哪些中期摘要
- 短期消息窗口范围
- 是否有记忆被裁剪

### 10.2 可解释性要求

用户应能看到：

- 记忆来源
- 所属层级
- 注入顺序
- 是否被冻结

---

## 十一、异常与边界处理

### 11.1 摘要失真

如果摘要与原文明显不符：

- 用户可编辑摘要
- 原始消息始终可回看
- 不自动删除旧摘要记录

### 11.2 长期记忆误写入

系统应支持：

- 删除该条长期记忆
- 标记为错误样本
- 避免自动重复写入同类误判

### 11.3 记忆冲突

当两条长期记忆冲突时：

- 不自动静默覆盖
- 在记忆中心提示冲突
- 允许用户手动选择保留或合并

### 11.4 Token 超限

当记忆注入过多导致超限时：

- 优先裁剪中期摘要
- 低重要性长期记忆降级
- 在 Prompt Inspector 中标记裁剪情况

---

## 十二、与其他模块的关系

### 12.1 对 Prompt 管线的支撑

记忆系统为 Prompt 管线提供：

- 长期记忆注入项
- 中期摘要注入项
- 短期消息窗口范围

### 12.2 对游玩模块的支撑

记忆系统为游玩模块提供：

- 剧情连续性
- 状态快照摘要
- 重大事件锚点

### 12.3 对聊天模块的支撑

记忆系统为聊天模块提供：

- 长期陪伴感
- 用户偏好连续性
- 项目讨论延续性

### 12.4 对状态系统的关系

状态系统与记忆系统需要协作：

- 中期摘要可引用状态快照
- 长期记忆可记录关键状态变化结果

### 12.5 对系统架构设计的反向要求

未来在系统架构设计中，必须满足：

- 原始消息永久保存
- 摘要与原始消息分离存储
- 长期记忆可按作用域区分
- 记忆注入结果可结构化表示

## 十五、结论

记忆系统是 Izumi Studio 长期可用性的核心支撑之一。

它的价值不只是“帮模型记住一点东西”，而是建立一套完整的、可控的上下文压缩与事实保留机制：

- 短期记忆保留现场感
- 中期摘要保留阶段连续性
- 长期记忆保留关键锚点

只要这套机制先设计清楚，后续游玩模式、聊天模式、Prompt 管线和系统架构设计都会更容易稳定下来。
