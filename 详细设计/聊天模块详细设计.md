## 一、功能定位

### 1.1 解决的核心问题

Izumi Studio 的聊天模式不是普通通用助手页，而是一个带有稳定人格、长期记忆和跨模式理解能力的个人 AI 对话空间。

它要解决的核心问题有四个：

- 让用户无需选择角色卡，也能随时与常驻的泉此方设定进行自由对话
- 让聊天模式不仅能聊日常，还能讨论角色卡、世界书、剧情记录和创作想法
- 让聊天模式能够引用其他模式中的已有内容，而不是成为一个孤立会话
- 让聊天过程同样具备 Prompt 可观察性，便于调试设定、记忆和工具调用

### 1.2 模块目标

聊天模式的目标是：

- 提供一个以泉此方为核心人设驱动的长期对话入口
- 支持日常闲聊、创作讨论、设定分析、剧情复盘等多类场景
- 支持跨模式引用角色卡、世界书、游玩记录片段
- 支持同一会话中切换模型但保留历史上下文
- 复用 Prompt Inspector、记忆系统、工具调用和 Quick Reply 底座

### 1.3 模块边界

聊天模式负责：

- 创建和管理聊天会话
- 注入常驻人设与聊天模式预设
- 接收用户输入并组织聊天上下文
- 处理跨模式引用内容
- 展示消息流与 Prompt 详情

聊天模式不负责：

- 角色卡游玩
- 世界书生成与编辑
- 状态栏 / 悬浮球驱动的剧情状态展示
- 剧情式线性回溯与对话复制

这些能力属于游玩模式或创作模式，聊天模式只在需要时引用其数据。

### 1.4 典型用户故事

```text
用户：
  打开聊天模式，直接问：
  “聊一下我那个蒙德世界书里的温迪角色，你觉得他说话最容易写崩的地方是什么？”

系统执行：
  1. 加载聊天模式会话和泉此方常驻设定
  2. 识别用户提到了已有世界书与角色
  3. 读取被引用的世界书 / 角色卡片段
  4. 结合聊天会话历史、记忆和引用内容组装 prompt
  5. 调用模型流式回复
  6. 保存原始响应、清洗后文本和 Prompt Trace
  7. 用户可在 Prompt Inspector 中查看本轮引用了哪些外部内容

系统交付：
  ✓ 一个具有稳定人格的长期聊天助手
  ✓ 可讨论系统内已有创作资料
  ✓ 可追踪的上下文引用和 Prompt 详情
```

## 二、总体结构

### 2.1 聊天模块在系统中的位置

```text
聊天模式首页
   ↓
选择历史会话 / 新建会话
   ↓
聊天主界面
   ↓
消息发送管线
   ├─ 聊天模式预设注入
   ├─ 泉此方常驻设定注入
   ├─ 跨模式引用解析
   ├─ 记忆召回
   ├─ 工具调用判断
   ├─ 模型调用
   ├─ 输出清洗
   ├─ Prompt Trace 保存
   └─ UI 渲染更新
```

### 2.2 模块内部子系统

聊天模块建议拆为以下子系统：

- `聊天会话管理子系统`
- `聊天消息发送与生成子系统`
- `跨模式引用子系统`
- `模型切换子系统`
- `Prompt Inspector 子系统`
- `Quick Reply 子系统`
- `导出与复盘子系统`

### 2.3 对外依赖

聊天模块依赖以下共享能力：

- 模型统一调度层
- 记忆系统
- Prompt Trace 存储能力
- 工具调用层
- Quick Reply 配置能力
- 角色卡 / 世界书 / 对话记录读取能力
- Regex / 输出清洗管线

### 2.4 对外暴露的数据与事件

聊天模块需要向系统其他部分暴露：

- 当前聊天会话信息
- 消息列表
- 本轮 Prompt Trace
- 本轮工具调用记录
- 本轮跨模式引用明细
- 当前模型切换记录

## 三、核心用户流程

### 3.1 主流程概览

```text
进入聊天模式
  → 选择历史会话或新建会话
  → 输入用户消息
  → 系统判断是否需要跨模式引用
  → 组装聊天上下文
  → 调用模型
  → 返回并展示回复
  → 保存 Prompt Trace
```

### 3.2 会话进入流程

1. 用户打开聊天模式首页。
2. 系统展示历史聊天会话列表，按时间分组或最近使用排序。
3. 用户选择：
   - 进入已有会话
   - 新建会话
4. 系统加载：
   - 聊天模式默认预设
   - 泉此方常驻设定
   - 当前会话消息历史
   - 会话相关记忆

### 3.3 发送消息流程

1. 用户输入问题、想法或闲聊内容。
2. 系统记录原始输入。
3. 系统识别本轮是否涉及：
   - 跨模式引用
   - 需要联网搜索的实时问题
   - 图片或文件理解
4. 系统收集本轮上下文素材：
   - 聊天模式预设
   - 泉此方常驻设定
   - User Persona
   - 被引用的外部片段
   - 记忆内容
   - 历史消息窗口
5. 系统组装 `messages` 数组并发送给模型。
6. 模型流式返回内容。
7. 系统记录原始响应并清洗。
8. 系统更新前端显示。
9. 系统保存 Prompt Trace 和相关记录。

### 3.4 跨模式引用流程

1. 用户在聊天中提及某个角色卡、世界书、剧情记录或上次游玩内容。
2. 系统识别引用目标。
3. 系统尝试通过快捷入口、显式选择器或名称匹配确认引用对象。
4. 系统提取合适的摘要片段或原文片段。
5. 系统把该片段作为本轮上下文的一部分注入。
6. Prompt Inspector 记录本轮引用来源和注入内容。

## 四、聊天会话管理子系统

### 4.1 设计目标

聊天会话管理的目标是让用户能把不同主题、不同阶段的长期聊天分开保存，同时保持进入门槛极低。

### 4.2 会话结构

建议聊天模式采用两层结构：

- `聊天模式`
- `聊天会话`

每个聊天会话内部为单线性消息历史。

### 4.3 会话操作

支持：

- 新建会话
- 重命名会话
- 删除会话
- 归档会话
- 切换历史会话
- 导出会话

### 4.4 会话列表展示

建议列表展示：

- 会话标题
- 最近一条消息摘要
- 最近活跃时间
- 所用模型
- 是否有未完成流式生成

### 4.5 自动命名

聊天会话默认可根据首轮消息自动命名，也允许用户后续手动修改。

## 五、聊天消息发送与生成子系统

### 5.1 核心目标

该子系统负责把聊天中的用户输入转为一轮完整模型请求，并把模型回复转成：

- 对用户可见的自然回答
- 对开发者可追踪的 Prompt 详情
- 对系统可记录的外部引用和工具调用明细

### 5.2 输入来源

本轮可进入 Prompt 的输入包括：

- 用户原始输入
- Quick Reply 内容
- Slash Command 展开结果
- 历史消息窗口
- 聊天模式主提示
- 泉此方人设预设
- User Persona
- 长期记忆
- 中期摘要
- 外部引用片段
- 工具调用结果

### 5.3 消息组装原则

聊天模式下，Prompt 组装必须遵守：

- 人格稳定性优先
- 用户当前问题优先于历史闲聊
- 引用内容是补充信息，不应淹没主对话
- 记忆用于维持连续性，但不应误替代用户最新意图

### 5.4 建议的组装顺序

本轮发送给模型前，建议按以下顺序构建上下文：

1. 全局系统预设 / 模式预设
2. 泉此方常驻设定
3. User Persona
4. 长期记忆
5. 中期摘要
6. 被引用的跨模式片段
7. 工具调用结果
8. 历史消息窗口
9. 当前用户输入

说明：

- 这里是聊天模块视角下的建议顺序，最终会在后续《Prompt管线与检查器设计》中统一收敛。
- 本文先固定聊天模块对该顺序的需求。

### 5.5 模型返回处理

模型返回后，应拆成三层内容：

- `raw_response`：模型原始返回
- `cleaned_response`：清洗后的文本
- `display_response`：最终展示给用户的文本

聊天模式通常不会像游玩模式那样依赖 `state_update`，但仍建议保留统一的三层输出结构。

### 5.6 流式输出

聊天模式支持流式响应。

流式期间建议：

- 实时显示回复文本
- 完整 Prompt Trace 在本轮结束后保存
- 如存在工具调用，则在工具调用结束后再继续主回复

## 六、泉此方常驻设定子系统

### 6.1 设计目标

聊天模式的最大特征是“泉此方常驻”，因此系统不能把她仅仅当作一段静态 prompt，而应把她视为聊天模式的核心人格层。

### 6.2 设定内容建议

常驻设定建议至少覆盖：

- 说话语气
- 性格倾向
- 兴趣偏好
- 对用户的称呼方式
- 对创作讨论的偏好表达
- 与普通角色卡游玩时的语气差异

### 6.3 行为目标

系统应确保聊天模式中的泉此方：

- 风格稳定
- 长期对话中不轻易漂移
- 既能闲聊，也能认真讨论项目和剧情
- 不和游玩模式中的具体角色卡混淆

### 6.4 与角色卡的边界

泉此方在聊天模式中是“常驻助手人格”，不是一个由用户手动选择的角色卡。

因此：

- 聊天模式启动时默认激活
- 不需要角色卡选择过程
- 不参与游玩模式的角色卡发布 / 切换机制

---

## 七、跨模式引用子系统

### 7.1 设计目标

聊天模式应能成为整个产品的信息枢纽，让用户自然地引用其他模式中的内容进行讨论。

### 7.2 可引用对象

建议支持引用：

- 角色卡
- 世界书
- 世界书条目
- 游玩模式对话片段
- 创作模式草稿片段
- 聊天模式历史片段

### 7.3 引用方式

建议支持三种方式：

- 自然语言提及
- 输入框快捷插入
- 侧边引用选择器

### 7.4 注入形式

被引用内容进入上下文时，建议包含：

- 来源类型
- 对象名称
- 摘要文本或截取片段
- 是否为用户主动插入

### 7.5 防止引用污染

聊天模式要避免一个问题：用户只是“讨论某个角色”，并不代表当前聊天助手要“扮演那个角色”。

因此系统应区分：

- `讨论对象`
- `当前说话人格`

也就是说：

- 被引用角色卡只作为分析材料
- 当前输出人格仍然是泉此方

### 7.6 Prompt Inspector 可见性

Prompt Inspector 应明确展示：

- 本轮引用了哪些对象
- 每个对象引用了哪些字段
- 注入的是摘要还是原文片段

---

## 八、模型切换子系统

### 8.1 设计目标

聊天模式中的模型切换要尽量轻，不应像切换角色一样打断会话连续性。

### 8.2 基本行为

支持：

- 在同一会话中切换模型
- 切换后保留全部历史消息
- 后续消息继续沿用新模型

### 8.3 建议记录

建议每条消息记录生成时所用模型，以便：

- 后续复盘
- Prompt Inspector 查看
- 对比不同模型风格

### 8.4 切换时的用户提示

切换模型后建议轻量提示：

- 当前已切换到的模型
- 切换只影响后续消息，不重写历史内容

## 九、Prompt Inspector 子系统

### 9.1 设计目标

聊天模式也必须具备 Prompt Inspector。

它需要回答：

- 本轮聊天究竟注入了哪些人设、记忆和外部引用
- 是否发生了工具调用
- 模型原始响应和最终展示结果有什么区别

### 9.2 触发入口

每条用户消息旁都应提供“详情”入口。

### 9.3 展示内容

建议至少包含：

- 用户原始输入
- 聊天模式预设
- 泉此方常驻设定摘要
- User Persona
- 长期记忆与中期摘要
- 被引用的角色卡 / 世界书 / 对话片段
- 完整 `messages` 数组
- token 统计
- 工具调用记录
- 模型原始响应
- 清洗前后对比

### 9.4 数据保存策略

聊天模式下同样建议按轮保存 `PromptTrace Snapshot`。

原因：

- 跨模式引用内容在未来可能变化
- 不保存快照会导致历史回看失真
- 便于分析人格漂移和记忆注入效果

## 十、Quick Reply 子系统

### 10.1 设计目标

Quick Reply 在聊天模式中主要承担“轻量表达”和“常用动作入口”的作用。

### 10.2 数据来源

建议来源为：

- 全局 Quick Reply 集
- 聊天模式专属 Quick Reply 集

### 10.3 典型内容

聊天模式下更适合的 Quick Reply 类型包括：

- 日常问候
- 继续追问
- 请求总结
- 请求吐槽 / 分析
- 插入跨模式引用入口

### 10.4 触发方式

支持：

- 点击即填入
- 点击即发送
- 宏变量替换
- slash command

---

## 十一、导出与复盘子系统

### 11.1 导出目标

聊天模式需要支持将长期聊天记录导出为可阅读文本，用于备份、复盘和引用。

### 11.2 导出格式

第一阶段建议支持：

- TXT
- Markdown

### 11.3 导出层级

建议分两种导出：

- `阅读版`：只导出最终展示消息
- `调试版`：附带 Prompt Trace 摘要、引用对象、工具调用摘要

---

## 十二、异常与边界处理

### 12.1 引用对象不存在

系统应：

- 提示引用失败
- 允许用户手动重新选择对象
- 不阻塞本轮主对话

### 12.2 实时信息缺失

当用户询问实时信息但工具调用失败时，系统应：

- 明示信息可能不准确
- 记录失败原因
- 在 Prompt Inspector 中标记工具调用失败

### 12.3 模型切换后风格漂移

系统应：

- 继续注入泉此方常驻设定
- 保留人格约束
- 在 Prompt Inspector 中保留本轮模型信息

### 12.4 跨模式引用过多

系统应：

- 对引用内容进行摘要裁剪
- 优先保留用户主动插入的引用
- 在 Prompt Inspector 中展示裁剪记录

## 十三、与其他模块的关系

### 13.1 与游玩模块的共享能力

聊天模块与游玩模块共享：

- 会话持久化框架
- 消息持久化框架
- Prompt Trace 机制
- 记忆系统
- Quick Reply 系统
- 工具调用层
- 输出清洗管线

### 13.2 与游玩模块的差异

聊天模块不同于游玩模块的关键点：

- 不依赖角色卡选择
- 不依赖世界书自动触发机制
- 不依赖状态栏和状态更新
- 不强调线性剧情回溯与对话复制
- 当前说话人格固定为泉此方

### 13.3 对系统架构设计的反向要求

未来在系统架构设计中，必须满足聊天模块以下要求：

- 支持跨模式内容读取
- 支持同会话模型切换
- 支持按轮保存 Prompt Trace
- 支持工具调用与引用记录并存

## 十六、结论

聊天模块是 Izumi Studio 中“个人 AI 助手”定位的核心承载者。

它的价值不只是一个常驻聊天页，而是把以下能力汇聚在一起：

- 稳定人格
- 长期记忆
- 跨模式讨论能力
- 实时信息获取能力
- Prompt 可观察性

只要聊天模块设计清晰，后续系统架构和数据模型就能更自然地统一“个人助手”与“剧情游玩”这两条产品线。
