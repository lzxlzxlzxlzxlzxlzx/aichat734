## 一、文档目标

本文件用于将 Izumi Studio 的前端方案从“结构草案”提升为“可直接指导设计和开发的正式版详细设计”。

本文件重点解决以下问题：

- 三个模式的导航、布局、页面层级如何统一又区分
- 聊天、游玩、创作三种模式各自的首页、详情页、会话页如何设计
- 页面视觉如何做到美观、实用、不过度花哨
- 前端如何参考 SillyTavern 的能力组织，但不照搬其复杂界面
- 哪些样式、布局、动效、密度和组件规范应当在项目中统一
- 后续前端实现应按什么顺序推进

本文件不负责输出高保真视觉稿，但会尽量把页面结构、样式方向和交互细节说清楚，保证后续：

- UI 设计有边界
- 前端开发有依据
- 不会因为审美方向不清晰而返工

## 二、外部设计调研结论

为了让“前端做得美观实用”不沦为空话，本次正式版设计参考了几类成熟设计原则，并结合本项目的产品特征做了取舍。

### 2.1 导航设计结论

调研结论：

- 顶级导航应稳定、可见、用于“切换区域”，而不是用于执行动作
- 底部导航适合承载少量、同级、需要随时切换的主入口
- 桌面端的大屏场景更适合把局部导航放在左侧，而不是把所有导航都塞到底部

结合本项目后的落地结论：

- `聊天 / 游玩 / 创作` 三个模式固定放在底部主导航
- 底部导航只承载模式切换，不承载设置、搜索、创建、调试等动作
- 模式内部的局部导航由左侧栏、页头和二级标签承担

### 2.2 视觉层级结论

调研结论：

- 页面必须通过大小、层级、留白、对比和分组来表达主次
- 一个页面里字体尺寸层级不宜过多，否则会显得乱
- 重要内容应该靠体量、位置和节奏突出，而不应该主要依赖鲜艳颜色

结合本项目后的落地结论：

- 每个页面只允许 3 级主文本层级 + 1 级辅助说明层级
- 主任务区必须在首屏视觉上成为焦点
- 次级信息靠密度降低、颜色减弱、位置后移来处理

### 2.3 菜单与侧栏结论

调研结论：

- 大屏下应尽量让主要导航可见，避免全部折叠进汉堡菜单
- 菜单文案必须短、明确、稳定、可扫描
- 当前所在位置必须明确可感知

结合本项目后的落地结论：

- 桌面端聊天模式、游玩模式、创作模式均允许可见侧栏
- 所有导航文案应避免内部术语化
- 当前模式、当前页面、当前对象都必须有明确高亮或标题反馈

### 2.4 设计令牌结论

调研结论：

- 现代可维护前端应尽量以 design tokens 管理颜色、字号、间距、阴影、主题
- CSS 变量是足够稳定且可扩展的落地方式

结合本项目后的落地结论：

- 前端需要从第一阶段就建立设计变量体系
- 不允许在业务组件里大量散落魔法数和临时颜色

### 2.5 美观与实用的取舍结论

本项目不是展示型官网，也不是传统后台。
因此我们最终采用的原则不是“极简到什么都没有”，而是：

- 功能要完整
- 结构要克制
- 重点要强
- 页面要耐看

换句话说：

- 不做 ST 那种“功能齐全但密度过高”的界面
- 也不做“看着高级但难以高效使用”的纯展示界面

## 三、前端总体设计原则

### 3.1 产品级原则

- 三模式共用统一品牌语言
- 三模式共用统一底层组件和样式变量
- 三模式在首页结构、会话布局、视觉气质上必须可明确区分
- 普通用户优先使用主流程，不应被 Prompt、状态、记忆等技术概念压迫
- 高级能力通过抽屉、面板、二级入口逐步暴露

### 3.2 交互级原则

- 页面首屏只突出一个主任务
- 高频操作必须一眼可见
- 低频配置统一二级化
- 所有会话输入区都要稳定、可靠、低干扰
- 模式切换要稳定，不打断用户心智

### 3.3 视觉级原则

- 优先使用留白、层级、边界和材质变化建立秩序
- 少用强饱和背景大面积铺满
- 不使用浮夸拟物和夸张发光效果
- 不让界面陷入默认组件库的“企业后台感”
- 动效用于强化结构与反馈，不用于堆砌存在感

## 四、信息架构与导航

### 4.1 三模式定义

- `聊天模式`
  - 面向长期陪伴型对话
  - 参考 DeepSeek、Kimi、ChatGPT 的熟悉结构
- `游玩模式`
  - 面向角色卡驱动的沉浸体验
  - 以角色卡为入口，以会话体验为核心
- `创作模式`
  - 面向世界书、角色卡、项目资产的生产和编辑
  - 以工作台、编辑器、预览区为核心

### 4.2 顶层路由

建议路由：

```text
/chat
/chat/session/:sessionId

/play
/play/:cardId
/play/session/:sessionId

/creation
/creation/new
/creation/project/:projectId
/creation/card/:cardId

/settings
/settings/models
/settings/presets
/settings/quick-replies
/settings/personas
```

### 4.3 底部模式导航

底部导航是本项目前端的一级主导航。

组件建议：

- `BottomModeNav`

规则：

- 始终显示 3 个一级入口
- 顺序固定：`聊天`、`游玩`、`创作`
- 图标 + 中文标签共同出现
- 当前模式高亮
- 不支持横向滚动
- 不承载动作型按钮

文案建议：

- `聊天`
- `游玩`
- `创作`

不建议使用更抽象或更长的名称，例如：

- `陪伴聊天`
- `剧情游玩`
- `内容创作中心`

因为底部导航最需要的是短、稳、易扫读。

### 4.4 模式内部导航

为了避免把所有导航压力都放到底部，模式内部需要二级导航：

- 聊天模式：
  - 左侧栏承载会话列表与最近角色卡
- 游玩模式：
  - 首页承载角色卡网格
  - 角色卡详情页承载进入会话的中转
- 创作模式：
  - 首页承载项目与角色卡资产入口
  - 工作台内由左树、页签和右侧预览区承载局部导航

## 五、布局系统

### 5.1 布局目标

前端整体布局需要满足以下目标：

- 桌面端有足够空间展示层级，但不显得空散
- 移动端不损失关键能力
- 游玩模式的沉浸感、聊天模式的熟悉感、创作模式的工作台感都能成立

### 5.2 响应式断点建议

建议断点：

- `sm`: 640px
- `md`: 768px
- `lg`: 1024px
- `xl`: 1280px
- `2xl`: 1440px

### 5.3 页面宽度策略

- 聊天模式主消息列：
  - 理想阅读宽度控制在 `720px - 860px`
- 游玩模式主消息列：
  - 理想阅读宽度控制在 `760px - 940px`
- 创作模式编辑区：
  - 可更宽，但单块表单正文宽度不应过长

原因：

- 太窄会显得拥挤
- 太宽会降低文本可读性与操作效率

### 5.4 壳层布局

建议采用：

- `AppShell`
- `ModeShell`

#### AppShell 负责

- 全局背景层
- 全局主题
- Toast / Dialog / 全局抽屉层
- 全局上传任务浮层
- 底部模式导航

#### ModeShell 负责

- 当前模式头部
- 当前模式是否显示左栏
- 当前模式是否显示右栏
- 当前模式内容区滚动策略

### 5.5 安全区域与固定区

由于底部固定导航常驻，需要预留：

- `页面底部安全区`
- `输入框额外底边距`
- `移动端键盘弹出时的可用空间修正`

建议：

- 底部主导航高度：`64px`
- 桌面端主导航与内容间距：`12px`
- 移动端为输入框额外保留：`72px - 88px` 可视安全区

## 六、设计系统

### 6.1 设计令牌总则

前端必须建立统一的 design tokens，至少包括：

- 颜色
- 字号
- 行高
- 间距
- 圆角
- 阴影
- 边框
- 动效时长
- 层级 z-index

建议实现方式：

- 以 CSS Variables 为核心
- 每个模式基于共享 token 做少量主题覆盖

### 6.2 颜色策略

#### 共享基础色

- `--bg-app`
- `--bg-surface-1`
- `--bg-surface-2`
- `--bg-elevated`
- `--fg-primary`
- `--fg-secondary`
- `--fg-muted`
- `--border-subtle`
- `--border-strong`
- `--accent-primary`
- `--accent-soft`
- `--success`
- `--warning`
- `--danger`

#### 三模式气质色建议

- 聊天模式：
  - 中性浅灰/暖灰为主
  - 少量清爽强调色
- 游玩模式：
  - 更深的背景基调
  - 更丰富的封面色彩，但 UI 容器保持克制
- 创作模式：
  - 更冷静、更专业的中性色
  - 重点操作用清晰的功能色

#### 颜色使用规则

- 强调色只用于关键状态和关键按钮
- 文本层级主要靠明度和粗细，而不是靠五颜六色
- 背景层次最多 3 层同时竞争，避免面板层数过多

### 6.3 字体与排版

建议中文界面字体优先使用：

- `PingFang SC`
- `Noto Sans SC`
- `Microsoft YaHei UI`

英文与数字可自然回退到系统无衬线。

建议字号层级：

- `Display`: 30/38
- `Title-1`: 24/32
- `Title-2`: 20/28
- `Title-3`: 18/26
- `Body`: 15/24
- `Body-small`: 14/22
- `Meta`: 12/18

规则：

- 一级页面最多使用两种标题大小
- 页面正文默认使用 `15px` 左右，不建议过大
- 辅助说明统一弱化，不可与正文竞争

### 6.4 间距系统

建议基础单位采用 `4px` 网格。

常用间距：

- `4`
- `8`
- `12`
- `16`
- `20`
- `24`
- `32`
- `40`

规则：

- 卡片内部优先 `16 / 20 / 24`
- 页面分区优先 `24 / 32`
- 超过 `40` 的留白只在大标题或首页头图区使用

### 6.5 圆角与边界

建议：

- 小控件：`10px`
- 输入框：`14px`
- 卡片：`18px`
- 大面板：`22px`
- 浮层：`20px`

边框规则：

- 默认使用低对比描边
- 大面板可依赖阴影和底色差异，不必全部描边
- 不要出现过多硬直矩形边界

### 6.6 阴影与材质

建议阴影分三级：

- 低层卡片阴影
- 抽屉/弹窗阴影
- 高层悬浮控件阴影

规则：

- 阴影用于层次，不用于“炫”
- 浮层与底部导航可以使用轻微毛玻璃或半透明材质，但必须保证文字对比度

### 6.7 动效策略

建议时长：

- 快速反馈：`120ms - 160ms`
- 面板进入：`180ms - 240ms`
- 页面切换：`220ms - 280ms`

动效类型：

- 淡入淡出
- 轻微位移
- 高度展开收起
- 列表错位渐显

禁止：

- 大幅缩放弹跳
- 高频闪烁
- 无意义粒子和发光

## 八、聊天模式详细设计

### 8.1 产品定位

聊天模式应让用户产生“这是一个真正能长期使用的现代 AI 对话产品”的第一感受。

参考方向：

- DeepSeek
- Kimi
- ChatGPT

但不能只是它们的翻版，因为本项目需要支持角色卡引用、多模态和 Prompt Inspector。

### 8.2 页面目标

- 像主流 AI 平台一样自然好上手
- 支持快速新建、切换和继续聊天
- 通过“最近使用角色卡”把本项目独有能力自然接入
- 不让 RPG 感压倒通用聊天体验

### 8.3 页面结构

桌面端建议双栏：

- 左栏：`280px - 320px`
- 主区：剩余空间

左栏包括：

- 新建聊天按钮
- 会话搜索
- 最近聊天会话列表
- 最近使用角色卡区域

主区包括：

- 顶部会话标题区
- 中部消息区
- 底部输入区
- 右侧 Prompt Inspector 抽屉入口

### 8.4 最近角色卡区

这是聊天模式区别于普通 AI 平台的关键。

区域命名建议：

- `最近角色卡`

不建议用：

- `角色快捷注入`
- `上下文角色对象`

原因是用户不需要理解内部机制，只需要理解“这些是我最近常用的角色卡”。

展示内容：

- 头像或封面缩略图
- 角色名
- 简短标签
- 最近使用时间

支持操作：

- 引用到当前聊天
- 查看角色卡简介
- 跳转游玩
- 跳转创作

### 8.5 页面布局细节

#### 顶部

- 左：当前会话标题
- 中：可选的模型/模式提示
- 右：更多操作、Prompt Inspector、会话菜单

#### 中部消息区

- 消息列居中
- 保持稳定阅读宽度
- 新消息自底部生长
- 空会话时显示欢迎态

#### 底部输入区

- 固定在页面底部上方
- 上方可挂一层 `Quick Reply`
- 输入框支持多行
- 附件预览在输入框上方轻量展开

### 8.6 视觉策略

- 背景要干净，不做重纹理
- 左侧栏颜色略深于主区，以形成区域分隔
- 主消息区要留足白，让阅读轻松
- 空状态可以更有品牌感，但正式对话中需收敛

### 8.7 页面状态

- 空状态：
  - 显示一句欢迎语和最近角色卡入口
- 加载状态：
  - 用骨架屏替代大转圈
- 错误状态：
  - 在输入区附近给出可重试反馈

## 九、游玩模式详细设计

### 9.1 产品定位

游玩模式是最有产品辨识度的模式。

它不是普通聊天页面，而是：

- 通过角色卡进入故事
- 以对话推进剧情
- 结合状态与角色氛围形成沉浸体验

### 9.2 页面层级

游玩模式包含三个核心页面：

- `/play`
  - 角色卡总览页
- `/play/:cardId`
  - 角色卡详情页
- `/play/session/:sessionId`
  - 正式游玩会话页

### 9.3 角色卡总览页

页面目标：

- 让用户像浏览游戏角色或视觉小说对象一样浏览角色卡
- 短时间内找到想玩的角色

页面结构：

- 顶部：标题、搜索、筛选
- 中部：角色卡网格
- 底部：底部模式导航

卡片建议尺寸：

- 桌面端：`260px - 300px` 宽
- 封面纵向比例优先，形成更强角色感

卡片内容：

- 封面图
- 角色名称
- 一句话简介
- 标签
- 最近游玩时间

卡片交互：

- hover 时显示轻微抬升
- 点击进入角色卡详情
- 不建议首页直接在卡片上塞过多次级按钮

### 9.4 角色卡详情页

页面目标：

- 让用户在进入游玩前完成选择
- 展示角色信息与会话入口
- 成为游玩与创作之间的跳转枢纽

页面结构建议：

- 顶部大头图区
- 中部左右双栏

左侧：

- 大封面
- 基本资料
- 标签

右侧：

- 默认开场白
- 备用开场白
- 历史会话列表
- 操作按钮组

操作按钮组：

- `开始新会话`
- `继续最近会话`
- `复制当前会话`
- `进入创作模式`

### 9.5 正式游玩会话页

这是项目最重要的前端页面之一。

设计目标：

- 中区对话体验足够沉浸
- 状态与工具存在但不过度抢主视觉
- 能体现“参考 ST 但比 ST 清晰很多”的差异

### 9.6 桌面端布局

建议布局：

- 左栏：`280px`
- 中区：`minmax(760px, 1fr)`
- 右栏：`320px`

左栏内容：

- 当前角色卡摘要
- 当前会话列表
- 历史会话入口
- 快速模式跳转

中区内容：

- 会话标题栏
- 消息时间线
- Quick Reply
- 输入区

右栏内容：

- 状态栏卡片
- 世界状态面板
- 辅助信息
- Inspector 入口

### 9.7 移动端布局

- 默认只显示中区
- 左栏收纳为侧滑抽屉
- 右栏收纳为状态抽屉
- Prompt Inspector 改为全屏抽屉或底部抽屉

### 9.8 输入区设计

游玩模式输入区要比聊天模式更有“舞台感”，但仍然高效。

建议：

- 外层采用更有层次的容器
- 支持附件上传
- 支持多行输入
- 支持显示开场、回溯、复制、导出等轻量工具按钮

不建议：

- 在输入区堆叠一整排复杂小图标
- 让每种能力都常驻暴露

### 9.9 状态栏与辅助信息

状态栏设计原则：

- 作为陪衬，不是主页面焦点
- 默认显示高价值状态
- 次级状态收纳进悬浮球或右侧面板

建议分层：

- 第一层：顶端横向简明状态条
- 第二层：右侧面板详细信息
- 第三层：开发者模式下原始 state_update 细节

### 9.10 对 ST 的取舍策略

需要继承的部分：

- 消息主链路强完整性
- 角色卡和世界书深度参与
- Prompt Inspector 的强调试性
- 会话操作能力完整

需要主动规避的部分：

- 高密度控件堆叠
- 多个侧栏与多个工具层同时常驻
- 技术概念直接裸露给普通用户
- 旧式网页工具站气质

### 9.11 视觉方向

游玩模式应该像“沉浸式故事工作台”，而不是“RPG 管理台”。

建议风格：

- 深色或中深色背景
- 角色封面主导气氛
- 面板采用半透明或低对比容器
- 消息区层次清晰，突出角色台词和叙事段落

## 十、创作模式详细设计

### 10.1 产品定位

创作模式是资料生产与编辑中心，但不应是一上来就密密麻麻的表单后台。

它需要兼顾：

- 资产浏览
- 结构化编辑
- 快速预览
- 发布与跨模式跳转

### 10.2 页面层级

建议页面：

- `/creation`
  - 创作首页
- `/creation/project/:projectId`
  - 项目工作台
- `/creation/card/:cardId`
  - 角色卡编辑页

### 10.3 创作首页

页面目标：

- 像“资产与项目入口页”
- 与游玩首页有结构呼应
- 让用户先选项目或角色卡，再进入具体编辑

页面结构：

- 顶部：页面标题、搜索、新建项目
- 中部：项目卡片网格
- 下方或并列区域：角色卡资产入口
- 侧边或顶部：最近编辑记录

### 10.4 项目工作台

建议采用三栏结构：

- 左栏：项目树 / 世界书结构
- 中区：编辑器主体
- 右栏：预览 / AI 助手 / 属性面板

左栏：

- 项目结构树
- 世界书条目列表
- 角色卡列表

中区：

- 当前条目编辑器
- 文本编辑、表单编辑、Schema 配置

右栏：

- 预览结果
- 角色卡展示效果
- 与 AI 的创作会话

### 10.5 角色卡编辑页

目标：

- 既要高效，又不能让新用户被配置淹没

页面结构建议：

- 顶部：角色卡名称、状态、发布信息、操作按钮
- 中部：主编辑区
- 右侧：预览区与模式跳转

主编辑区建议分块：

- 基础信息
- 开场白
- 设定内容
- 预设与绑定
- 图片与封面
- 发布与导入导出

右侧预览区：

- 角色卡封面预览
- 基础展示卡
- 快速进入游玩模式按钮

### 10.6 视觉方向

- 专业
- 清晰
- 高信息密度但不过载
- 更强调结构秩序，而非情绪氛围

## 十一、跨模式跳转设计

### 11.1 设计目标

三模式不是孤立系统，而是围绕“角色卡”和“项目”流动。

必须明确区分：

- `模式切换`
- `对象跳转`

### 11.2 角色卡跨模式跳转

建议统一组件：

- `ModeJumpActions`

出现位置：

- 聊天模式最近角色卡区
- 游玩模式角色卡详情页
- 游玩模式会话页头部
- 创作模式角色卡编辑页

支持：

- 跳转到游玩
- 跳转到创作

### 11.3 跳转表现

- 文案明确
- 目标清晰
- 点击后聚焦到目标对象
- 保留必要上下文

文案建议：

- `进入游玩`
- `编辑角色卡`
- `返回创作`

## 十二、多模态与附件前端设计

### 12.1 统一入口

聊天模式与游玩模式输入区统一支持：

- 点击上传
- 拖拽上传
- 粘贴图片

创作模式额外支持：

- 封面上传
- 参考图上传
- 文档导入

### 12.2 展示策略

图片：

- 缩略图展示
- 点击放大
- 支持下载

文件：

- 使用文件卡片
- 显示文件名、大小、解析状态

生成图：

- 显示来源轮次
- 显示生成时间
- 支持再次查看与下载

### 12.3 页面整合原则

- 附件区必须是消息流程的一部分
- 不允许附件交互变成完全独立的侧系统
- Inspector 中必须能看到哪些附件进入当前轮

## 十三、Prompt Inspector 前端设计

### 13.1 定位

Prompt Inspector 是开发者与高级用户的调试工具，不是普通用户的主操作界面。

### 13.2 展示方式

- 聊天模式：
  - 右侧抽屉
- 游玩模式：
  - 右侧抽屉或消息详情抽屉
- 创作模式：
  - 右侧调试面板

### 13.3 分区

- Overview
- Input
- Presets
- Injections
- Final Messages
- Response
- State Update
- Tool Calls
- Token Stats

### 13.4 样式策略

- 默认弱化，不与主界面争夺注意力
- 信息用分组卡片而不是大块 JSON
- 原始内容可折叠
- 开发者模式下保留更直接的数据视图

---

## 十四、可访问性与可用性要求

### 14.1 可读性

- 正文与背景保持足够对比度
- 不用颜色单独表达关键信息
- 文本行长控制合理

### 14.2 键盘与焦点

- 底部导航可键盘聚焦
- 抽屉打开后焦点正确进入
- Dialog 关闭后焦点回到触发点

### 14.3 点击与触控

- 关键按钮触控面积足够
- 底部导航项不可过窄
- 图标按钮需要 tooltip 或文字补充

### 14.4 状态反馈

- 所有异步操作必须可感知
- 上传、生成、保存、失败都要有明确反馈
- 错误提示贴近动作发生位置

---

## 十五、页面状态设计

每个关键页面必须定义以下状态：

- 加载中
- 空状态
- 正常态
- 错误态
- 无权限或不可用态

### 15.1 空状态要求

- 不只写“暂无数据”
- 提供下一步动作
- 保持品牌感

### 15.2 加载状态要求

- 优先使用骨架屏
- 长任务使用进度反馈
- 避免整个页面空白等待

### 15.3 错误状态要求

- 说明问题
- 给出重试
- 不泄漏后端实现细节给普通用户

## 十八、最终结论

当前前端正式设计的核心结论如下：

1. 三模式固定通过底部主导航切换，这是整个应用最稳定的一级导航结构。
2. 聊天模式要像现代 AI 平台一样熟悉、轻量、清晰，并通过“最近角色卡”接入项目特色能力。
3. 游玩模式必须以角色卡浏览和会话沉浸体验为核心，参考 ST 的能力组织，但显著降低复杂度。
4. 创作模式必须像专业工作台，但首页仍应保持卡片化和资产入口感，不做纯后台表单首页。
5. 游玩与创作之间必须支持围绕角色卡的双向跳转。
6. 前端必须从第一阶段建立统一的设计令牌体系，否则后续无法真正做出“好看而一致”的产品。
7. 审美要求不是附加项，而是正式设计约束，应直接进入实现方案。
