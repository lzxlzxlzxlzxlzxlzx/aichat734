> 文档状态：v1.0
> 日期：2026-04-15
> 数据库目标：SQLite 第一阶段落地方案
> 参考文档：产品需求文档_V2.md、系统架构设计.md、核心数据模型设计.md、详细设计目录下全部模块设计

---

## 1. 文档目标

本文用于把当前已经确定的核心数据模型，进一步收敛为真实可落地的数据库表结构设计。

这份文档重点回答四个问题：

- 第一阶段数据库里到底有哪些表
- 每张表有哪些字段、类型、默认值、主外键与约束
- 哪些复杂结构继续放 JSON
- 哪些内容不放数据库，而放项目目录文件系统

本文默认数据库文件位于：

```text
ST-1/data/db.sqlite
```

---

## 2. 总体设计原则

## 2.1 SQLite 优先

第一阶段以 SQLite 为唯一正式数据库。

原因：

- 单用户本地优先
- 部署成本最低
- 足够支撑当前对象规模
- 后续可以迁移 PostgreSQL

## 2.2 结构化字段关系化，复杂块 JSON 化

建议关系化存储：

- 主键、外键
- 名称、状态、模式、类型
- 时间戳
- 列表页需要排序过滤的字段
- 需要唯一约束的字段

建议 JSON 化存储：

- Prompt 分层结构
- ST 导入原始结构
- 多模态结构化内容块
- token 统计
- 状态变量全集
- 正则脚本详细配置

## 2.3 媒体文件不直接入库

图片、文档原文件不存进 SQLite。

数据库只保存：

- 文件路径
- MIME 类型
- 大小
- 元数据
- 关联关系

文件系统建议目录：

```text
data/
  db.sqlite
  images/
    uploads/
    generated/
    references/
    covers/
    personas/
  files/
    uploads/
  exports/
```

## 2.4 会话强约束为线性消息流

第一阶段不做 branch 树。

因此数据库层不建立：

- 消息树主结构
- branch 表
- branch_message 映射表

而是采用：

- `sessions`
- `messages`
- `message_swipes`
- `conversation_snapshots`

来实现线性会话、复制、回溯与编辑前保护点。

## 3. SQLite 类型约定

SQLite 没有严格类型系统，因此这里采用工程约定：

| 语义类型 | SQLite 类型 | 说明 |
|---|---|---|
| UUID / ID | `TEXT` | 应统一使用字符串 UUID |
| 枚举 | `TEXT` | 通过 `CHECK` 约束限制 |
| 布尔 | `INTEGER` | `0/1` |
| 时间戳 | `TEXT` | ISO 8601 字符串 |
| 长文本 | `TEXT` | 普通文本、Markdown、JSON 字符串 |
| 数值 | `INTEGER` / `REAL` | 按需求区分 |
| JSON | `TEXT` | 存 JSON 字符串，应用层保证序列化 |

通用约定：

- 所有主键统一为 `id TEXT PRIMARY KEY`
- 所有时间字段统一为 ISO 8601 UTC 字符串
- 所有 JSON 字段默认存合法 JSON 字符串
- 所有布尔字段统一使用 `0/1`

## 16. 建表顺序建议

建议按下面顺序建立数据库：

1. 基础配置表：`presets`、`preset_versions`、`user_personas`、`quick_reply_sets`、`regex_scripts`
2. 创作资产表：`creation_projects`、`worldbooks`、`worldbook_entries`
3. 媒体表：`media_assets`
4. 角色卡表：`character_cards`、`character_card_versions`、关联表
5. 会话主表：`sessions`
6. 消息与 swipe：`messages`、`message_swipes`
7. 运行快照：`conversation_snapshots`、`prompt_traces`
8. 记忆与状态：`memory_summaries`、`long_term_memories`、`state_snapshots`、`state_change_logs`
9. 附件与工具：`message_attachments`、`tool_call_records`

这样做的好处是：

- 外键依赖清晰
- 迁移脚本更好写
- 后续验证也更容易分阶段进行

---

## 17. 下一步建议

现在这份文档已经把完整的数据库表结构范围确定下来了。最自然的下一步是二选一：

1. 继续写成 `SQLite 建表 SQL 草案`，把 `CREATE TABLE`、`CREATE INDEX` 全部生成出来。
2. 继续写成 `后端 Schema 设计`，把每张表对应的 API request/response 和 Pydantic/ORM 模型定下来。

如果按开发顺序来说，我更建议下一步先做第 1 个，也就是直接产出可执行的 SQLite 建表 SQL。这样你的数据库设计就从“文档方案”进入“可以真正开始编码”的阶段了。
