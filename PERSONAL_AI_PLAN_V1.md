# SillyTavern 个人专属 AI 对话工具改造方案 V1

## 1. 文档目标

本方案用于把 `SillyTavern` 从通用型 AI 前端，逐步改造成一个更适合个人长期使用的 AI 对话工具。

本阶段目标不是直接实现全部功能，而是先明确以下内容：

- 核心使用目标
- 功能范围
- 第一阶段优先级
- 与 `graphify-3`、`OpenHarness` 的协同方式

---

## 2. 总体产品目标

目标产品应具备以下特征：

- 对话链路透明，用户能清楚看到自己输入了什么、系统实际发送了什么、模型返回了什么
- 拥有稳定的短期记忆、中期摘要和长期记忆，不因上下文过长而丢失关键设定
- 支持文本、图片、文件等多模态输入，并支持按要求生成图片
- 支持世界观、角色、人际关系和剧情设定，适合长期角色扮演、共创和个人陪伴式使用
- 允许持续个性化定制，而不是只能依赖一次性 Prompt

一句话目标：

> 将 `SillyTavern` 改造成一个高透明度、强记忆、多模态、可自定义世界观与角色系统的个人 AI 对话平台。

---

## 3. 用户核心需求

### 3.1 消息透明化

用户需要清楚看到：

- 自己输入的原始内容
- 实际发送给模型的完整消息
- 模型返回的原始响应
- 渲染后的最终回复
- 中间发生的处理过程，例如：
  - 历史裁剪
  - 摘要压缩
  - 记忆注入
  - 世界观注入
  - 角色卡注入
  - 系统提示词拼接

建议定义一个统一模块：`消息检查器 / Prompt Inspector`

建议至少支持以下视图：

- `用户原始输入`
- `实际发送 payload`
- `上下文拼接明细`
- `模型原始输出`
- `最终展示结果`

该模块是后续调试记忆、角色、工具调用的基础。

### 3.2 长期记忆

用户希望对话不会因为上下文过长而丢失关键内容。

需求拆分如下：

- 保留用户偏好和长期事实
- 保留世界观规则、角色关系和重要事件
- 对长对话自动生成阶段摘要
- 在后续轮次中按需召回最相关记忆
- 允许用户查看和修正记忆内容

建议采用三层结构：

- `短期记忆`：最近消息窗口，直接进入上下文
- `中期记忆`：阶段摘要，用来替代被裁剪的旧对话
- `长期记忆`：持久化事实库，包括人物、设定、事件、关系、偏好

需要避免的错误：

- 只依赖上下文窗口，导致旧信息被直接丢弃
- 自动总结后失真，且用户无法纠正
- 世界观与角色记忆混在一起，后续难以控制

### 3.3 多模态输入输出

用户不仅要发送文本，还要支持图片、文件，以及根据需求生成图片。

输入侧需求：

- 上传图片并发送给支持视觉的模型
- 上传文件供模型读取或总结
- 支持图片作为参考素材

输出侧需求：

- 文本回复正常显示
- 生成图片后可直接嵌入对话流
- 支持对生成图的参数控制

建议拆分为三类能力：

- `看图理解`
- `读文件理解`
- `按要求生成图片`

如果后续做深一点，还可以支持：

- 用世界观和角色设定辅助出图
- 用用户上传的参考图做风格或角色一致性约束
- 为图片生成自动记录对应的提示词和参数

### 3.4 世界观与角色系统

用户希望能在工具中长期维护自己的世界观和个性鲜明的角色，并让模型理解其他作品中的角色特点。

这部分建议拆成四层：

#### 世界观层

- 世界规则
- 历史背景
- 阵营与组织
- 地点设定
- 时间线
- 禁忌和限制

#### 角色层

- 角色基础信息
- 性格标签
- 说话风格
- 行为偏好
- 价值观
- 人际关系
- 角色秘密
- 成长弧线

#### 参考作品层

- 收集其他作品中的角色资料
- 提炼这些角色的核心特点
- 区分“灵感参考”和“直接扮演”
- 避免多个角色特征相互污染

#### 演绎控制层

- 当前主要扮演角色
- 是否允许多角色同场
- 是否允许旁白
- 是否允许角色成长偏离初始设定
- 是否优先遵守世界观设定还是参考原作风格

## 4. 第一阶段建议优先级

按收益和依赖关系，建议分三期推进。

### 第一期：先解决可观察和可控制

目标：

- 让用户知道模型到底收到了什么
- 让后续调试有依据

优先功能：

- 消息检查器
- 上下文拼接明细
- 原始响应查看
- 记忆注入来源标记

### 第二期：建立稳定记忆

目标：

- 让角色、人设、世界观不会随上下文增长而丢失

优先功能：

- 阶段摘要
- 长期记忆存储
- 记忆召回注入
- 记忆查看与编辑

### 第三期：增强多模态和角色生态

目标：

- 把产品从聊天工具升级为创作与陪伴平台

优先功能：

- 图片与文件输入
- 图片生成工作流
- 世界观管理界面
- 角色管理界面
- 参考作品角色资料库

## 5. 产品模块草案

### 5.1 对话面板

保留现有聊天体验，但增加：

- 当前角色
- 当前世界观
- 当前记忆状态
- 当前模型
- 当前模式

建议引入“模式切换”概念，例如：

- `日常对话`
- `角色扮演`
- `剧情共创`
- `世界观设定`
- `图像生成`
- `资料分析`

### 5.2 消息检查器

建议为每轮对话提供一个展开面板，显示：

- 用户原始输入
- 发送前的上下文结构
- 实际发给模型的消息数组
- 模型返回原始内容
- 工具调用记录（未来需要时）

### 5.3 记忆中心

建议单独做一个“记忆中心”，支持：

- 查看长期记忆
- 查看自动摘要
- 手动锁定重要事实
- 删除错误记忆
- 调整不同记忆的优先级

### 5.4 世界观与角色中心

建议引入结构化编辑能力，而不是只靠大段文本：

- 世界观卡片
- 阵营卡片
- 地点卡片
- 角色卡片
- 关系图
- 时间线

### 5.5 多模态工作台

建议统一管理以下流程：

- 上传图片
- 上传文件
- 选择视觉模型
- 触发图像生成
- 复用上一次图片生成参数

## 6. `graphify-3` 能提供什么帮助

`graphify-3` 最适合在前期分析和长期维护阶段使用。

### 6.1 代码理解

它可以帮助快速理解 `SillyTavern` 代码结构：

- 找出消息发送链路
- 找出 Prompt 拼接位置
- 找出角色卡、世界书、扩展系统的关系
- 找出适合插入“消息检查器”和“记忆中心”的模块

在改造开始前，可优先让它回答以下问题：

- 用户消息从输入框到模型请求的完整链路在哪里
- 历史消息是在哪里裁剪和组装的
- 角色卡和世界书的注入点在哪里
- 图片生成相关能力在哪些模块
- 插件系统和扩展系统如何接入

### 6.2 架构梳理

`graphify-3` 的优势不是直接改代码，而是：

- 帮助降低理解成本
- 帮助持续追踪结构变化
- 帮助发现跨模块耦合关系

这对于 `SillyTavern` 这种功能较多、前后端逻辑交织的项目非常有价值。

### 6.3 需求验证

在做功能设计时，可以用它分析：

- 哪些功能已经有近似实现
- 哪些地方适合扩展，哪些地方应该重构
- 记忆、角色、世界观、多模态相关逻辑是否已经存在基础设施

### 6.4 结论

`graphify-3` 适合作为：

- `代码理解工具`
- `架构导航工具`
- `持续维护辅助工具`

它不直接提供最终产品能力，但能显著降低定制开发难度。

## 7. `OpenHarness` 能提供什么帮助

`OpenHarness` 不适合作为直接替换 `SillyTavern` 的主体，但它的架构思路值得借鉴。

### 7.1 可借鉴的核心能力

可重点参考的方向：

- Agent loop
- tools / skills / plugins
- memory / session resume
- permissions / hooks
- multi-agent coordination

这些能力中，最相关的是：

- 记忆体系
- 工具调用体系
- 可观察的执行过程
- 会话恢复

### 7.2 对本项目的潜在价值

如果未来你的 `SillyTavern` 不只是聊天，而是要发展成“个人 AI 助手”，那么 `OpenHarness` 的思路会非常有用。

适合借鉴的具体能力：

- 为对话加入“任务模式”
- 为复杂操作加入工具调用
- 为高风险操作加入权限确认
- 为长期使用加入持久记忆和会话恢复
- 为后续扩展加入插件和技能化设计

### 7.3 不建议直接照搬的地方

需要注意：

- `OpenHarness` 偏 Agent 基础设施，不是以角色扮演和聊天体验为核心
- 如果过早引入完整 Agent 架构，会显著提高复杂度
- 第一阶段不应尝试把 `SillyTavern` 直接改成完整的 Agent 平台

更合理的方式是：

- 先借鉴设计思想
- 再按需求挑选最有价值的部分逐步引入

### 7.4 结论

`OpenHarness` 适合作为：

- `高级能力设计参考`
- `记忆与工具调用架构参考`
- `未来 Agent 化演进参考`

它更适合中后期，而不是一开始就深度整合。

---

## 8. 三个项目的组合建议

建议采用如下分工：

- `SillyTavern`：主产品和主界面
- `graphify-3`：代码理解和架构分析工具
- `OpenHarness`：高级能力设计参考来源

推荐顺序：

1. 用 `graphify-3` 理清 `SillyTavern` 的消息链路、角色系统和扩展点
2. 在 `SillyTavern` 中优先落地消息透明化和记忆能力
3. 后续再选择性吸收 `OpenHarness` 的记忆、工具调用和会话恢复思路

---

## 9. 当前建议结论

当前最应该优先做的不是大量加功能，而是先建立“可观察性”。

原因：

- 如果看不清实际发送给模型的内容，就无法判断角色设定、记忆和世界观为什么失效
- 如果没有稳定记忆机制，角色和世界观再丰富也会很快崩掉
- 如果没有清晰模块划分，后续加入多模态和图片生成会越来越乱

所以当前建议的第一阶段目标是：

- 做消息检查器
- 梳理上下文拼接流程
- 设计短期/中期/长期记忆结构
