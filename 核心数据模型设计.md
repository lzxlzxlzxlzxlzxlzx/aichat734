# Izumi Studio 核心数据模型设计

> 文档状态：v1.1 细化版
> 日期：2026-04-15
> 参考文档：系统架构设计.md、产品需求文档_V2.md、详细设计目录下全部模块设计、SillyTavern代码分析_V1.md

---

## 1. 文档目标

本文用于把当前已经完成的系统架构和模块设计，收敛为一套可落地的数据模型。

这份文档的目标不是抽象罗列对象名称，而是明确：

- 核心实体有哪些
- 实体之间如何关联
- 哪些对象应该版本化
- 哪些对象应该快照化
- 哪些字段适合关系化存储
- 哪些字段第一阶段应保留为 JSON
- 角色卡与对话模型如何兼容 SillyTavern，同时保持 Izumi Studio 自己的产品方向

---

## 2. 设计原则

## 2.1 单用户本地优先

第一阶段围绕单用户本地使用设计，不引入多租户、团队协作、复杂权限等字段。

## 2.2 原始数据与派生数据分离

必须明确区分以下对象：

- 原始消息
- 摘要
- PromptTrace
- 状态快照
- 工具调用记录
- 媒体元数据

派生数据不能覆盖原始数据。

## 2.3 会话统一抽象

创作、游玩、聊天三种模式共享 `Session` 抽象，但通过 `mode` 区分业务语义。

## 2.4 兼容 SillyTavern，但不被其绑定

对角色卡、世界书、预设系统：

- 导入层尽量兼容 ST
- 内部存储按 Izumi Studio 的需求设计
- 导出层保留 ST 兼容能力

## 2.5 关系字段与复杂结构混合存储

第一阶段主数据库为 SQLite，因此建议采用：

- 可检索、可排序、可关联的字段关系化
- 结构复杂、变化频繁的字段 JSON 化

## 5. 预设系统数据模型

## 5.1 `Preset`

表示一个逻辑预设对象。

建议字段：

```json
{
  "id": "uuid",
  "name": "Izumi_0407",
  "source_type": "builtin | imported_st | user_created",
  "status": "active | archived",
  "created_at": "timestamp",
  "updated_at": "timestamp"
}
```

## 5.2 `PresetVersion`

用于保存预设版本快照。

```json
{
  "id": "uuid",
  "preset_id": "uuid",
  "version": 1,
  "raw_source": {},
  "mapped_layers": {
    "global_core": [],
    "mode_specific": [],
    "persona_style": [],
    "compat_legacy": []
  },
  "variables": {},
  "created_at": "timestamp"
}
```

设计说明：

- `raw_source` 用于保留 ST 原始结构
- `mapped_layers` 用于保存 Izumi 内部分层结果
- 无法结构化映射的内容进入 `compat_legacy`

## 7. 角色卡数据模型

## 7.1 设计原则

角色卡模型应尽量参考 SillyTavern Character Card 的结构思路，但不能直接照搬。

Izumi Studio 的角色卡必须满足：

- 草稿与已发布版本分离
- 会话创建时冻结到具体版本
- 支持 ST PNG / JSON 导入
- 支持后续导出回 ST 兼容格式
- 支持头像、封面、参考图等附属资源

## 7.2 `CharacterCard`

`CharacterCard` 是卡片头对象，面向列表页、搜索、筛选、关系绑定。

```json
{
  "id": "uuid",
  "project_id": "uuid | null",
  "name": "角色卡名称",
  "name_normalized": "用于搜索与去重",
  "description": "短简介",
  "tags": [],
  "cover_asset_id": "uuid | null",
  "avatar_asset_id": "uuid | null",
  "worldbook_id": "uuid | null",
  "default_preset_id": "uuid | null",
  "status": "draft | published | archived",
  "source_type": "created | imported_st_png | imported_st_json",
  "current_draft_version_id": "uuid | null",
  "current_published_version_id": "uuid | null",
  "latest_session_id": "uuid | null",
  "created_at": "timestamp",
  "updated_at": "timestamp",
  "published_at": "timestamp | null"
}
```

设计说明：

- `current_draft_version_id` 表示编辑器当前草稿
- `current_published_version_id` 表示游玩模式默认使用的版本
- `latest_session_id` 用于角色卡列表快速跳转最近对话
- `name_normalized` 用于做搜索与半唯一约束

## 7.3 `CharacterCardVersion`

`CharacterCardVersion` 是完整运行快照，面向发布、回滚、导出、创建会话。

```json
{
  "id": "uuid",
  "card_id": "uuid",
  "version": 3,
  "version_label": "v0.3-playtest",
  "is_published": true,
  "spec": "izumi_v1 | st_v3_imported",
  "source_type": "created | imported_st_png | imported_st_json",
  "base_info": {
    "name": "角色名",
    "description": "...",
    "tags": []
  },
  "prompt_blocks": {
    "system_prompt": "...",
    "post_history_instructions": "...",
    "first_mes": "...",
    "alternate_greetings": [],
    "mes_example": "...",
    "scenario": "...",
    "personality": "...",
    "speaking_style": "...",
    "background": "...",
    "creator_notes": "..."
  },
  "play_config": {
    "worldbook_id": "uuid | null",
    "default_model": "claude-sonnet-4-6",
    "default_temperature": 0.9,
    "default_max_tokens": 1200,
    "allow_multimodal_input": true
  },
  "extension_blocks": {
    "depth_prompt": {},
    "authors_note": {},
    "preset_config": {},
    "image_config": {},
    "regex_script_ids": [],
    "quick_reply_set_ids": [],
    "npcs": []
  },
  "import_meta": {
    "raw_source": {},
    "mapping_report": {},
    "unsupported_fields": []
  },
  "created_at": "timestamp"
}
```

设计说明：

- `CharacterCard` 保存索引层字段
- `CharacterCardVersion` 保存真正用于运行的完整快照
- ST 导入对象天然更接近版本快照，而不是卡片头
- `import_meta.raw_source` 用于保留 ST 原始结构，降低信息损失

## 7.4 为什么版本和主对象分开

因为你的产品已经明确要求：

- 草稿与发布分离
- 发布后保留历史版本
- 游玩模式默认绑定已发布版本
- 会话创建时角色设定应被冻结

因此角色卡必须版本化，而不是在单一对象上原地覆盖。

## 9. 会话数据模型

## 9.1 `Session`

三种模式共用统一 `Session` 抽象。

```json
{
  "id": "uuid",
  "mode": "play | chat | creation",
  "name": "会话名",
  "status": "active | archived | deleted",
  "card_id": "uuid | null",
  "card_version_id": "uuid | null",
  "worldbook_id": "uuid | null",
  "project_id": "uuid | null",
  "persona_id": "uuid | null",
  "preset_version_id": "uuid | null",
  "origin_session_id": "uuid | null",
  "origin_snapshot_id": "uuid | null",
  "message_count": 128,
  "last_message_id": "uuid | null",
  "last_message_at": "timestamp | null",
  "current_state_snapshot_id": "uuid | null",
  "model_name": "claude-sonnet-4-6",
  "created_at": "timestamp",
  "updated_at": "timestamp"
}
```

说明：

- `play` 会话通常绑定 `card_id + card_version_id`
- `chat` 会话通常不强制绑定角色卡
- `creation` 会话绑定 `project_id`
- `origin_session_id` / `origin_snapshot_id` 用于记录复制来源
- 这不是 branch 树，只是复制追踪

## 9.2 不做消息树，做线性会话

按当前方案，第一阶段对话模型必须围绕“线性消息流”构建，而不是消息树。

因此：

- 不引入 `parent_message_id` 作为主结构
- 不建立 branch 树
- 通过 `ConversationSnapshot` 支持复制、回溯、编辑前保护点

## 10. 消息数据模型

## 10.1 `Message`

`Message` 表示时间线上的一个消息位置。

```json
{
  "id": "uuid",
  "session_id": "uuid",
  "role": "system | user | assistant | tool",
  "sequence": 120,
  "reply_to_message_id": "uuid | null",
  "content": "最终展示文本",
  "raw_content": "原始文本",
  "structured_content": [],
  "active_swipe_id": "uuid | null",
  "token_count": 320,
  "is_hidden": false,
  "is_locked": false,
  "is_edited": false,
  "source_type": "normal | opening | regenerated | tool_result | imported",
  "created_at": "timestamp",
  "updated_at": "timestamp | null"
}
```

设计说明：

- `content` 是最终展示文本
- `raw_content` 用于保留清洗前文本
- `structured_content` 预留给多模态与富内容块
- `reply_to_message_id` 只作为辅助关联，不作为树结构核心
- `active_swipe_id` 仅对 assistant 消息有意义

## 10.2 `MessageSwipe`

游玩模式和聊天模式都保留 swipe，因此不应把每个备选回复都存成普通消息。

```json
{
  "id": "uuid",
  "message_id": "uuid",
  "swipe_index": 0,
  "generation_status": "completed | failed | aborted",
  "raw_response": "...",
  "cleaned_response": "...",
  "display_response": "...",
  "provider_name": "anthropic",
  "model_name": "claude-sonnet-4-6",
  "finish_reason": "stop | length | tool_calls",
  "token_usage": {},
  "trace_id": "uuid | null",
  "created_at": "timestamp"
}
```

设计说明：

- `Message` 代表该位置的逻辑消息
- `MessageSwipe` 代表该位置的多个候选 assistant 回复
- `trace_id` 可以精确关联一次 PromptTrace

## 11. 回溯与复制数据模型

## 11.1 `ConversationSnapshot`

该对象用于承载：

- 任意长度回溯
- 从历史节点复制对话
- 编辑历史消息前的自动保护点

```json
{
  "id": "uuid",
  "session_id": "uuid",
  "snapshot_type": "rollback_point | copy_source | edit_cutoff",
  "message_id": "uuid | null",
  "message_sequence": 58,
  "inclusive": true,
  "state_snapshot_id": "uuid | null",
  "memory_summary_ids": [],
  "label": "编辑前自动快照",
  "summary": "可选摘要",
  "created_by": "system | user",
  "created_at": "timestamp"
}
```

设计说明：

- 不做 branch，不代表不需要强回溯能力
- 回溯本质是把当前会话裁切回某个 `message_sequence`
- 同时恢复对应的 `StateSnapshot`
- 必要时可恢复关联的中期摘要

## 18. 与 SillyTavern 的兼容策略

## 18.1 角色卡兼容

兼容原则：

- 导入时尽量保留 ST V3 核心字段语义
- 导出时根据 `CharacterCardVersion` 重新生成 ST 兼容结构

重点兼容字段：

- `name`
- `first_mes`
- `alternate_greetings`
- `mes_example`
- `system_prompt`
- `post_history_instructions`
- `depth_prompt`
- `extensions.world`
- `regex_scripts`

## 18.2 预设兼容

兼容原则：

- 保留 ST 原始预设结构
- 映射到 Izumi 分层结构
- 无法映射部分进入 `compat_legacy`

## 18.3 世界书兼容

兼容原则：

- `WorldBookEntry` 尽量保持与 ST 世界书相近
- 方便导入现有条目
- 方便未来导出

- `sessions(mode, updated_at)` 建索引

- `messages(session_id, sequence)` 建联合唯一索引
- `message_swipes(message_id, swipe_index)` 建联合唯一索引
- `conversation_snapshots(session_id, message_sequence)` 建索引

## 21. 结论

当前这套核心数据模型有三个最重要的结论：

1. 角色卡采用“卡片头 + 版本快照”的双层模型，兼容 SillyTavern，同时满足草稿与发布分离。
2. 对话采用“会话头 + 线性消息流 + swipe + 快照回溯”的模型，不引入 branch 树复杂度。
3. 进入真实数据库设计时，优先把检索字段关系化，把复杂且不稳定的块继续保留在 JSON 中。

按这份模型继续推进，下一步最自然的工作是：

- 明确 SQLite 表结构草案
- 定义后端 API request / response schema
