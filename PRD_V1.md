## 1. 项目概述

### 1.1 项目背景

用户希望构建一个个人专属 AI 对话与创作平台，以泉此方（Izumi Konata）预设为核心文风驱动，支持自定义世界书和角色卡系统。区别于 SillyTavern 等通用工具，本项目定位为**高度个人化、模式清晰、创作与游玩分离**的独立应用。

### 1.2 项目目标

- 提供三种清晰分离的使用模式：创作、游玩、聊天
- 支持与大模型协作生成和迭代世界书、角色卡
- 涉及原著 IP 时，通过联网搜索保证内容符合原著设定
- 支持多个主流大模型的自由切换
- 支持图片理解与图片生成的多模态交互
- 具备自动记忆机制，保证长期对话不丢失关键内容
- 本地部署，浏览器访问，数据私有

### 1.3 项目范围

**包含：**
- 三种模式的完整对话界面
- 世界书创建、编辑、管理系统
- 角色卡发布与管理系统
- 多模型统一调度层
- 联网搜索辅助（用于 IP 查询）
- 泉此方预设集成与变量管理
- 自动记忆系统（分层摘要，防止上下文丢失）
- 世界书状态栏与悬浮球 UI 组件系统
- 多模态输入输出（图片理解 + 图片生成）

**不包含（当前版本）：**
- 多用户/账号系统
- 移动端适配
- 云端同步

## 3. 功能需求

### 3.1 模式概览

系统提供三种独立模式，通过顶部导航或侧边栏切换：

```
┌─────────────────────────────────────────┐
│  Izumi Studio    [创作] [游玩] [聊天]    │
└─────────────────────────────────────────┘
```

---

### 3.2 创作模式（Creation Mode）

#### 3.2.1 目标

通过与大模型的多轮对话，协作生成完整的世界书和角色卡，最终发布供游玩模式使用。

#### 3.2.2 核心流程

```
新建项目 → 确定主题/IP方向 → 对话迭代生成条目 → 预览完整世界书 → 发布为角色卡
```

#### 3.2.3 功能清单

**F-C-01 新建创作项目**
- 用户输入项目名称、简要描述、类型标签（原创 / 原著改编）
- 选择目标模型
- 系统初始化空白世界书和角色卡模板

**F-C-02 对话式世界书生成**
- 用户通过自然语言描述需求（如"帮我生成一个赛博朋克风格的都市世界观"）
- 模型生成结构化世界书条目，以卡片列表形式展示
- 用户可继续对话修改（如"把科技水平降低，加入魔法元素"）
- 每次修改追加为新版本，支持回退

**F-C-03 原著 IP 支持**
- 当用户声明涉及某一原著 IP 时（如"我要基于《原神》蒙德地区"）
- 系统触发联网搜索，获取该 IP 的核心设定数据
- 模型基于搜索结果生成符合原著的世界书条目
- 生成时标注哪些内容来源于原著，哪些为用户自定义扩展
- 支持用户覆盖原著内容（明确标记为"自定义改编"）

**F-C-04 世界书结构编辑器**
- 生成结果可在结构化编辑器中手动修改
- 支持按分类管理条目（世界观 / 角色 / 地点 / 事件 / 规则 / 关系）
- 支持条目的增删改、排序、启用/禁用
- 支持条目关键词绑定（用于游玩时触发注入）

**F-C-05 角色卡配置**
- 配置角色基础信息：名称、简介、性格、说话风格、人际关系
- 关联世界书（一个角色卡可绑定一个世界书）
- 设置对话模式默认参数：文风选择、思维链开关、字数范围
- 预览角色卡效果（发布前可以进行试对话）

**F-C-06 发布角色卡**
- 将配置完成的角色卡发布为可用状态
- 发布后在游玩模式中可见
- 支持更新已发布的角色卡（保留历史版本）
- 支持导出为标准 JSON 格式

**F-C-07 草稿管理**
- 未发布的项目保存为草稿
- 支持继续编辑草稿

### 3.3 游玩模式（Play Mode）

#### 3.3.1 目标

选择已发布的角色卡，沉浸式进行剧情对话与角色扮演。

**定位说明：** 游玩模式每次选定且仅选定**一个角色卡**，该角色卡定义故事的主角色及其设定。故事中可出现大量 NPC，这些 NPC 通过角色卡的 `character.npcs` 字段和世界书的 `character` 类型条目进行定义，由模型在叙事中动态扮演，无需额外角色卡。

#### 3.3.2 核心流程

```
选择角色卡 → 进入场景/新建对话 → 流式对话 → 保存剧情记录
```

#### 3.3.3 功能清单

**F-P-01 角色卡选择界面**
- 展示所有已发布的角色卡（卡片网格形式）
- 显示角色名、简介、标签、最近对话时间
- 支持按标签/名称过滤

**F-P-02 对话界面**
- 流式输出，实时显示模型生成内容
- 展示当前使用的角色卡、文风、模型信息
- 支持场景描述输入（当前地点、时间、氛围等）
- 消息气泡区分用户输入和模型输出

**F-P-03 世界书触发机制**
- 对话时根据关键词自动触发相关世界书条目注入
- 用户可查看当前轮次注入了哪些条目（可展开的「上下文面板」）

**F-P-04 对话管理**
- 同一角色卡下支持多个独立剧情线（对话存档）
- 支持新建、切换、重命名、删除对话
- 支持从某条消息处分支出新的剧情线

**F-P-05 消息操作**
- 重新生成：对最后一条模型回复重新生成
- 编辑：直接编辑历史消息后继续
- 撤回：撤回最后一轮对话

**F-P-08 Prompt Inspector（消息详情）**
- 每条用户发送的消息旁提供「详情」入口（点击展开）
- 展示该轮实际发送给模型的完整内容，包括：
  - 用户原始输入
  - 本轮激活的 Izumi 预设条目列表
  - 本轮注入的世界书条目（含触发关键词）
  - 注入的记忆内容（长期记忆 + 中期摘要）
  - 完整 messages 数组结构（可折叠）
  - Token 统计（各部分占比）
- 模型返回的原始 response 也可查看（含被过滤的 STATE_UPDATE 指令块）
- 此功能在三种模式中均可用

**F-P-06 对话导出**
- 支持将对话记录导出为 TXT / Markdown 格式

**F-P-07 状态栏与悬浮球显示**
- 游玩时根据角色卡配置渲染世界书绑定的状态组件
- 状态数值随剧情自动更新（详见 §3.5）

### 3.4 聊天模式（Chat Mode）

#### 3.4.1 目标

以泉此方人设进行自由对话，可以聊日常、讨论角色卡和剧情内容，相当于有记忆的个人 AI 助手。

#### 3.4.2 功能清单

**F-H-01 自由对话**
- 直接对话，无需选择角色卡
- 泉此方人设常驻，语气风格保持一致
- 支持流式输出

**F-H-02 跨模式上下文引用**
- 用户可以在聊天中引用已有内容，如：
  - "聊一下我那个蒙德世界书里的温迪角色"
  - "上次游玩时发生的那个剧情，你怎么看"
- 系统提供快捷入口，可将角色卡/对话记录片段插入到当前聊天上下文中

**F-H-03 会话管理**
- 聊天历史按时间分组保存
- 支持新建会话、切换历史会话
- 会话内的上下文自动管理（超长时触发摘要）

**F-H-04 模型切换**
- 聊天过程中可随时切换模型
- 切换后的对话继续在同一会话中，历史消息保留

### 3.5 状态栏与悬浮球系统（UI Components）

#### 3.5.1 目标

为每个世界书配置一套动态 UI 组件，在游玩模式中实时展示剧情状态（角色数值、关系、时间线等），增强沉浸感和信息可见性。

#### 3.5.2 组件类型

| 组件类型 | 说明 | 示例 |
|---------|------|------|
| **状态栏（Status Bar）** | 固定在对话界面顶部或底部的横条，展示若干关键数值 | HP 100/100 · 好感度 ♥♥♥ · 当前地点：蒙德城 |
| **悬浮球（Floating Ball）** | 可拖动的圆形浮层，点击展开详情面板 | 展示角色状态卡、当前任务、剧情阶段 |
| **侧边信息栏（Side Panel）** | 对话区右侧可折叠栏，展示多项属性 | 角色关系图、世界状态、当前持有道具 |

#### 3.5.4 状态更新机制

模型在每轮回复末尾输出结构化状态更新指令，系统解析后更新 UI：

```
[STATE_UPDATE]
hp: 85
location: 蒙德城·西风骑士团
affection: +1
mood: 紧张
[/STATE_UPDATE]
```

- 状态更新指令对用户不可见（前端过滤）
- 状态历史随对话记录一起持久化
- 用户可手动修改当前状态值

#### 3.5.5 创作模式中的配置

**F-C-08 UI 组件配置**
- 用户描述需求（如"我要显示血量、好感度和当前任务"）
- AI 自动生成完整 Schema，包含字段类型、颜色、图标建议
- 用户可在可视化编辑器中调整
- 提供若干预设模板（RPG 数值型 / 视觉小说好感型 / 纯文字信息型）

## 4. 世界书系统

### 4.1 世界书结构定义

世界书由元数据、条目列表和 UI 组件配置三部分组成：

**世界书元数据：**
```json
{
  "id": "uuid",
  "name": "世界书名称",
  "description": "简介",
  "ip_name": "原著名称（如有）",
  "ui_components": { "...见 §3.5.3" },
  "created_at": "timestamp",
  "updated_at": "timestamp"
}
```

**条目结构（Entry）：**
```json
{
  "id": "uuid",
  "category": "worldview | character | location | event | rule | relation",
  "title": "条目标题",
  "keywords": ["关键词1", "关键词2"],
  "content": "条目正文内容",
  "source": "original | ip_canon | ip_extended",
  "enabled": true,
  "priority": 100,
  "created_at": "timestamp",
  "updated_at": "timestamp"
}
```

### 4.2 条目分类说明

| 分类 | 说明 |
|------|------|
| worldview | 世界规则、历史背景、科技水平、魔法体系等 |
| character | 角色基础信息、性格、说话风格、背景故事 |
| location | 地点描述、特征、关联事件 |
| event | 重要历史事件、剧情节点 |
| rule | 世界限制、禁忌、系统规则 |
| relation | 角色/势力之间的关系网络 |

### 4.3 触发机制

游玩模式下，系统根据对话内容扫描关键词，将匹配条目注入上下文。触发逻辑：

- 精确匹配：条目关键词完全出现在最近 N 轮消息中
- 优先级排序：priority 值越高越优先注入
- Token 预算：根据当前模型上下文窗口动态计算可注入条目数量

- `temperature`
- `max_tokens`
- `top_p`
- 自定义 `base_url`（支持代理/中转）

## 8. 自动记忆系统

### 8.1 设计目标

解决长期对话因上下文超限而丢失关键剧情、角色设定和重要事件的问题。采用三层记忆结构，确保无论对话多长，核心内容始终在模型的感知范围内。

**适用范围：三种模式（创作 / 游玩 / 聊天）均使用同一套三层记忆机制。**

**原始消息永久保存：** 无论是否生成摘要，所有对话的原始消息记录均完整持久化，不会被删除。摘要是压缩后注入上下文用的副本，不替代原始记录。用户随时可以回溯历史消息全文。

### 8.2 三层记忆架构

```
┌─────────────────────────────────────────────────┐
│                 上下文窗口                        │
│  ┌──────────────┐  ┌──────────────┐  ┌────────┐ │
│  │  短期记忆     │  │  中期记忆    │  │ 长期   │ │
│  │ 最近 N 轮消息 │  │ 阶段摘要     │  │ 记忆   │ │
│  │ (直接保留)   │  │ (压缩替代)   │  │ (锚点) │ │
│  └──────────────┘  └──────────────┘  └────────┘ │
└─────────────────────────────────────────────────┘
```

| 层级 | 内容 | 触发时机 | 注入方式 |
|------|------|---------|---------|
| **短期记忆** | 最近 20 轮原始消息 | 常驻 | 直接放入 messages 数组 |
| **中期记忆** | 每 20 轮自动生成的阶段摘要 | 每 20 轮触发一次 | 以 system message 注入，替代被裁剪的旧消息 |
| **长期记忆** | 手动或自动标记的关键事实 | 用户标记 / AI 识别 | 固定注入上下文头部 |

### 8.3 中期记忆：阶段摘要

**触发条件：**
- 对话轮次达到设定阈值（默认每 20 轮，用户可在设置中配置，范围 10-100 轮）
- 用户手动触发

**摘要生成流程：**
```
检测到触发条件 → 后台调用模型 → 对最近 N 轮消息生成结构化摘要
→ 摘要存储到数据库 → 旧消息从活跃上下文移出（保留原始记录）
→ 下一轮对话时摘要注入
```

**摘要结构：**
```json
{
  "id": "uuid",
  "session_id": "会话ID",
  "segment_start": 41,
  "segment_end": 60,
  "summary": "本阶段剧情摘要正文...",
  "key_events": ["事件1", "事件2"],
  "state_snapshot": { "hp": 85, "location": "蒙德城", "affection": 3 },
  "created_at": "timestamp"
}
```

**摘要注入格式（注入到 system message）：**
```
[剧情摘要 - 第3至第60轮]
{summary}

关键事件：{key_events}
当前状态快照：{state_snapshot}
[摘要结束]
```

### 8.4 长期记忆：关键事实库

长期记忆用于存储跨会话、不应被遗忘的核心信息：

**自动识别（AI 判断）：**
- 角色重大决定（如"主角选择了加入骑士团"）
- 世界观关键设定变更
- 不可逆的剧情转折点

**手动标记（用户操作）：**
- 右键任意消息 → "标记为长期记忆"
- 可添加备注说明

**长期记忆数据结构：**
```json
{
  "id": "uuid",
  "card_id": "角色卡ID",
  "content": "记忆内容",
  "type": "auto | manual",
  "importance": "high | medium | low",
  "created_at": "timestamp"
}
```

### 8.5 记忆管理界面

游玩界面提供"记忆中心"侧边栏：

- **查看**：分层展示短期/中期/长期记忆内容
- **编辑**：直接修改摘要文本或长期记忆条目
- **删除**：移除错误的记忆条目
- **锁定**：锁定重要摘要，禁止被后续自动覆盖
- **手动触发摘要**：不等待阈值，立即生成当前摘要

### 8.6 上下文组装策略

每次发送请求前，系统按以下顺序组装 messages：

```
1. [System] 主提示 + Izumi预设激活条目
2. [System] 世界书触发条目（关键词匹配）
3. [System] 长期记忆（全部注入）
4. [System] 中期记忆摘要（按时间顺序，最多注入 3 段）
5. [User/Assistant] 短期记忆（最近 20 轮原始消息）
6. [User] 当前用户输入
```

Token 超出时优先裁剪：中期摘要（保留最新）→ 世界书条目（按 priority 裁剪）

## 9. 联网搜索（工具调用）

### 9.1 设计定位

联网搜索作为一个**系统工具（Tool / Skill）**，由模型在对话中自主判断是否需要调用，无需用户手动触发。**三种模式（创作 / 游玩 / 聊天）均可使用。**

典型触发场景：
- 创作模式：用户提及某一原著 IP，模型调用搜索获取原著设定
- 游玩模式：剧情中涉及某个模型不确定的历史/地理/专业知识
- 聊天模式：用户询问需要实时信息的问题

### 9.2 工具调用流程

```
模型判断需要搜索
→ 输出 tool_call: web_search(query="...")
→ 后端执行搜索 API 调用
→ 返回搜索结果摘要作为 tool_result
→ 模型基于结果继续生成回复
→ 用户消息详情中可查看本轮是否调用了搜索及使用的查询词
```

### 9.3 搜索接入方案（优先级）

1. Tavily Search API（推荐，AI 友好，原生支持 tool_call 格式）
2. SerpAPI
3. Bing Search API

### 9.4 搜索结果处理

- 搜索结果由模型摘要提炼后融入回复，不直接展示原始内容
- Prompt Inspector 中可查看本轮搜索查询词和原始返回摘要
- 搜索结果按查询词缓存（TTL 24h），避免重复请求相同内容

## 10. 多模态输入输出

### 10.1 能力概览

| 能力 | 适用模式 | 说明 |
|------|---------|------|
| 图片理解 | 游玩 / 聊天 / 创作 | 上传图片，模型分析内容并融入对话 |
| 图片生成 | 游玩 / 聊天 | 根据剧情描述生成对应插图 |

### 10.2 图片理解

#### 10.2.1 支持的视觉模型

| 模型 | 视觉能力 |
|------|---------|
| Claude 3.5/3.7 | ✓ |
| GPT-4o / GPT-5 | ✓ |
| Qwen-VL 系列 | ✓ |
| Kimi Vision | ✓ |
| DeepSeek-V3 | ✗（纯文本） |

#### 10.2.2 功能说明

**F-M-01 图片上传**
- 输入框支持粘贴图片、点击上传、拖拽
- 支持格式：PNG / JPG / WEBP / GIF（静态帧）
- 单张最大 20MB

**F-M-02 图片理解融入对话**
- 图片随消息一起发送给模型
- 游玩模式：可将图片作为场景参考素材（"这是现在的地点，请描述角色进入后的反应"）
- 创作模式：可上传参考图辅助生成世界书条目（上传角色官图，辅助生成角色描述）

**F-M-03 图片存储**
- 上传图片保存至本地 `data/images/uploads/`
- 历史消息中以文件路径引用，不重复存储

### 10.3 图片生成

#### 10.3.1 支持服务

| 服务 | 接入方式 | 说明 |
|------|---------|------|
| DALL-E 3 | OpenAI API | 质量高，需 OpenAI Key |
| Stable Diffusion（本地） | WebUI / ComfyUI API | 完全本地，需单独部署 |
| Flux | API / 本地 | 高质量开源模型 |

用户在设置中配置启用哪个服务，生成时可选择。

#### 10.3.2 触发方式

**自动触发（游玩模式可选开启）：**
- 每次模型回复后，系统提取场景描述，自动调用图片生成
- 生成插图嵌入对话流，显示在回复下方

**手动触发：**
- 用户点击"生成插图"按钮，可自定义提示词
- 右键某条回复 → "为此场景生成图片"

#### 10.3.3 提示词生成流程

```
当前场景描述 + 角色外貌设定（来自世界书 character 条目）
→ 模型生成英文图片提示词
→ 拼接角色卡中配置的固定风格词
→ 发送给图片生成服务
```

用户可在生成前预览并编辑提示词。

#### 10.3.4 风格参考图（角色一致性）

支持在角色卡中上传**参考图**，用于约束生成图片的角色外貌和风格一致性：

- 参考图存储于 `data/images/references/{card_id}/`
- 支持上传多张（最多 5 张），系统自动选择最相关的一张
- 接入支持 IP Adapter / Reference 功能的本地 SD / Flux 服务时，参考图作为风格约束输入
- DALL-E 3 不支持参考图约束，此功能仅对本地服务生效
- 用户可标记某张参考图为"主角外貌参考"或"场景风格参考"

#### 10.3.5 生成结果管理

- 生成图片保存至 `data/images/generated/{session_id}/`
- 每张图片记录：生成时间、提示词、关联消息 ID、使用的参考图 ID
- 支持重新生成（保留原图，新图追加）
- 支持导出当前会话的全部生成图片

### 11.1 存储方案

本地部署，使用文件系统 + SQLite：

| 数据类型 | 存储方式 |
|---------|---------|

## 14. 非功能需求

### 14.1 性能

- 流式输出延迟 < 500ms（首字符）
- 角色卡列表加载 < 200ms
- 世界书条目触发匹配 < 50ms
- 摘要生成在后台异步执行，不阻塞对话
- 图片生成异步执行，通过 SSE 推送进度

### 14.2 可靠性

- 对话记录实时持久化，不因刷新丢失
- 模型调用失败时提供明确错误提示和重试入口
- 流式输出中断时支持从断点继续
- 摘要生成失败时保留原始消息，不影响对话继续

### 14.3 可扩展性

- 新增模型只需在配置中添加，不修改业务逻辑
- 新增文风只需添加预设条目，不修改前端
- 新增图片生成服务只需实现统一 ImageService 接口
- 世界书条目结构预留扩展字段
- UI 组件类型可通过 Schema 扩展，无需修改前端渲染逻辑

### 14.4 安全

- API Key 存储在 `.env`，不暴露给前端
- 所有外部请求（LLM / 搜索 / 图片生成）通过后端代理
